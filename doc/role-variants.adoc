= Role Variants and Per-Role Models
:toc:
:toclevels: 3

Overview and entry-point map for the per-role model variant-emission system. The canonical specifications live next to the code they govern; this document is a navigational summary that points there.

Authors configure `models.roles.<name>` in `.plan/marshal.json`. The Claude target emits `{base}-{level}.md` agent variants for every canonical agent that opts in via `implements: ext-point-dynamic-level-executor`. Dispatch sites resolve the role's level via `manage-config models read --role <role>` and call the matching variant by name. End users get the right model per role at dispatch time without any user-side build step.

== Quick reference

* **Configure a role** — set `plan.models.roles.<role>` in `.plan/marshal.json` to one of `low | medium | high | xhigh | xxhigh | inherit`. See `role-variants.md`.
* **Resolve a role at dispatch** — `python3 .plan/execute-script.py plan-marshall:manage-config:manage-config models read --role <role>` returns `level: <value>`. See dispatch-site patches in `phase-3-outline`, `phase-4-plan`, `phase-6-finalize`, `plan-marshall/workflow/planning.md` and `recipe.md`.
* **Make a new agent role-eligible** — add `implements: plan-marshall:extension-api/standards/ext-point-dynamic-level-executor` to its frontmatter; remove any hardcoded `model:` / `effort:`. Plugin-doctor enforces both rules. See `ext-point-dynamic-level-executor.md`.
* **Inspect emitted variants** — generate the Claude target output (`python3 marketplace/targets/generate.py --target claude --output target/claude`) and look under `target/claude/{bundle}/agents/{base}-{level}.md`. The canonical no-suffix file remains the `inherit` target.

== Rationale for keeping the canonical specs in `marketplace/bundles/`

Each standard is co-located with the skill or extension that owns it so it stays under the same review lifecycle as the code it governs. This `doc/` page is intentionally a thin summary — duplicating the substance here would create two places to update on every change and the `marketplace/bundles/` copy is the one that downstream consumers (plugin-doctor, the build target, dispatch sites) read.
