= Execution-Context Overview
:toc:

Why plan-marshall ships exactly one role-eligible agent (`plan-marshall:execution-context`), and how every plan-marshall `Task:` invocation dispatches through it.

== The Proliferation Problem

Before the consolidation, plan-marshall shipped eleven canonical agents (`phase-agent`, `q-gate-validation-agent`, `automated-review-agent`, `create-pr-agent`, `sonar-roundtrip-agent`, `lessons-capture-agent`, `research-best-practices-agent`, `detect-change-type-agent`, plus three pm-plugin-development agents). Each canonical declared `implements: ext-point-dynamic-level-executor`, so the Claude target emitted six variants per canonical ‚Äî sixty-six emitted agent files in total. Every new dispatch target (a new LLM-judgement workflow worth its own model-pinning role) meant a new canonical, six new emitted files, and a new role-key in the registry.

Each new agent file also duplicated a body of dispatcher boilerplate: load `dev-general-practices`, load skill, locate workflow doc, execute, emit return TOON. That boilerplate diverged across canonicals (subtle differences in worktree handling, log-line formats, return-shape constraints) and was a maintenance hazard.

== The One-Dispatcher Answer

`plan-marshall:execution-context` is the sole role-eligible canonical agent. It owns the dispatcher boilerplate exactly once and parameterises the workflow body at dispatch time. The caller passes five prompt-body fields:

* `name` ‚Äî human label for logging and `mark-step-done`
* `plan_id` ‚Äî plan identifier (or sentinel `none` for free-standing dispatches)
* `skills[]` ‚Äî caller-loaded skills, in load order
* `workflow` ‚Äî bundle-prefixed notation for the workflow doc to run (e.g. `plan-marshall:phase-1-init/SKILL.md`, `plan-marshall:plan-marshall/workflow/triage.md`)
* `WORKTREE` ‚Äî repo-relative working directory (`.` for main checkout)

The dispatcher loads `dev-general-practices` implicitly, then the caller-specified `skills[]`, then reads the resolved workflow path and follows it. Workflow-specific runtime inputs (`finding_type`, `track`, `scope`, etc.) flow through additional prompt-body fields the workflow doc declares.

== Level Variants

The Claude target emits six variants per ordinal level ‚Äî `execution-context.md` (canonical, inherit), `execution-context-low.md`, `-medium.md`, `-high.md`, `-xhigh.md`, `-xxhigh.md`. Each variant has `model:` and `effort:` baked into its frontmatter. Dispatch sites resolve which variant to call via `manage-config models resolve-target --role <key>`, which returns the canonical name when level is `inherit`/empty and the matching variant otherwise. The resolver reads `.plan/marshal.json` fresh per dispatch, so a `models` block edit takes effect on the next call ‚Äî no Claude Code restart required.
