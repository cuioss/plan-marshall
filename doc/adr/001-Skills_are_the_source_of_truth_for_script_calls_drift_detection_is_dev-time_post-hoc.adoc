= ADR-001: Skills are the source of truth for script calls; drift detection is dev-time + post-hoc
:toc: left
:toclevels: 2
:sectnums:

== Status

Accepted

== Context

The marketplace executor (`.plan/execute-script.py`) dispatches every workflow's script invocations of the form `python3 .plan/execute-script.py {notation} {verb_chain...} {flags...}`. Each `{notation}` resolves to a Python script that uses `argparse` (or an equivalent declarative CLI library) to publish its subcommand and flag surface. Workflow markdown — `SKILL.md`, `standards/*.md`, `workflow/*.md` — documents these invocations as code blocks. The contract has two sides that can drift:

* the **declarative argparse schema** in each script (the live truth)
* the **prose-documented invocations** in workflow markdown (what callers actually type)

Two failure modes follow from the gap:

1. **Invented-subcommand drift** — workflow prose references a verb / flag that does not exist (renamed, never implemented, or paraphrased). The runtime call lands an `argparse` `exit_code=2` rejection.
2. **Stale-prose drift** — argparse renames a verb but the markdown lags. Same surface failure.

The architectural question this ADR settles: **where does the check for these drift classes belong?**

A natural-feeling first answer is "validate at runtime before dispatch": pre-flight the call against an embedded allowlist generated from each script's argparse declarations. The allowlist must be produced by static analysis of the source (typically AST walking).

That answer has two structural problems:

* The AST walker is only correct for declarations that are **always module-top-level literals**. The moment a script registers subcommands inside a loop, conditional, or runtime-resolved iterable (e.g. a list returned by a function that reads disk), the static-analysis layer must either silently miss the names or implement a small Python interpreter to evaluate module-level expressions.
* Even when the walker produces a correct list, the runtime pre-flight reproduces the rejection logic argparse itself runs at execution. The pre-flight layer adds a second specification of the same contract — one whose correctness must be tracked against every change argparse accepts.

The principle the marketplace adopts:

[quote]
Skills are the source of truth for script calls. Drift detection lives in plugin-doctor (dev-time) and plan-retrospective (post-hoc). The runtime executor stays dumb.

== Decision

The executor does **not** carry an embedded subcommand allowlist or any pre-flight validator. Two complementary mechanisms enforce the script-call contract at the layers where they are correct on their merits:

1. **Dev-time static analyzer** — `pm-plugin-development:plugin-doctor` carries a `script-call-drift` rule that walks marketplace markdown, extracts every `python3 .plan/execute-script.py {notation} {verb_chain...} {flags...}` invocation including nested subparser chains, probes each `{notation}` for `--help` to obtain argparse's published surface, and reports drift when a documented verb / flag does not appear. Drift is caught at edit time, in the same review where the prose drifts, when the maintainer can fix it.

2. **Post-hoc retrospective sweep** — `plan-marshall:plan-retrospective/scripts/script-failure-analysis.py` reads `.plan/script-execution.log`, buckets `argparse_rejection` failures by `notation × verb`, and surfaces clusters in the plan retrospective. Drift that slips past the dev-time analyzer surfaces as a structured finding within the same finalize cycle.

The runtime executor's role narrows to: resolve notation → path, dispatch the subprocess, forward stdin/stdout/stderr/exit-code. argparse owns the schema contract.

== Consequences

=== Positive

* The executor contains no static-analysis layer. Whatever subcommand-registration pattern a script uses (literal, loop, runtime-resolved, conditional) just works.
* Drift is caught in the same review that introduces it, by reading argparse's own published help surface — there is no second specification to maintain.
* The exit code from argparse is the contract. Workflows that don't check it fail loud at the post-hoc sweep instead of silently rejecting.

=== Negative

* Drift survives in prose until either the dev-time analyzer runs or a caller hits the rejection. The retrospective sweep bounds the survival to one plan cycle.
* `script-call-drift` parses markdown shell invocations heuristically; exotic line continuations or paraphrased forms can escape detection. Patterns that escape become follow-up rule extensions, not workarounds in the executor.

=== Risks

* If a maintainer adds an invocation that is documented wrong AND never runs the dev-time analyzer AND no plan exercises the call path, the drift can persist. The risk is bounded by the fact that any plan that does exercise the path surfaces it on its next finalize via the retrospective sweep.

== Alternatives Considered

Three alternatives fall under the "extend the runtime safety-net" family. All three were rejected.

=== Option A — Source-side annotation contract

Each script declares a module-level literal alongside its argparse declarations:

[source,python]
----
_PREFLIGHT_SUBCOMMANDS = ['work', 'decision', 'script', 'read', 'separator']
----

The pre-flight layer reads the constant when present; otherwise falls back to AST extraction.

*Rejected because:* duplicates the argparse declarations the constant is supposed to describe. The duplication drifts the moment a maintainer adds a subcommand without updating the constant — re-introducing the failure mode the layer was added to prevent. Adds a per-script touch-up cost for a class of error argparse already handles.

=== Option B — AST evaluation for module-level loop constants

The extractor finds module-top-level literal-list assignments and resolves `for X in VAR:` loops by evaluating the list inline.

*Rejected because:* solves only the subset of cases where the iterable is a module-level literal. Scripts that register subcommands from `for X in get_X():` (runtime-resolved iterables, including disk-reading ones) remain unhandled. The static-analysis layer cannot evaluate runtime calls without becoming a Python interpreter — open-ended complexity in service of a guard argparse already provides.

=== Option C — Hybrid (B with A as escape hatch)

Option B for resolvable cases plus Option A as opt-in for runtime-resolved iterables.

*Rejected because:* compounds the cost of both. The runtime layer still exists, with two code paths whose correctness must be maintained against future scripts, and the escape hatch reintroduces Option A's per-script duplication problem.

=== Why all three fall in the same class

Each option extends the safety-net validator with more cleverness. The safety net itself is the over-engineering: the runtime contract argparse already enforces is the contract; the genuine value-add of a pre-flight layer would be edit-time enforcement, which moves to a static analyzer (the dev-time `script-call-drift` rule), not the runtime executor.

== Generalisation

The pattern, stated independently of this codebase:

When a proposed safety-net layer's stated job is to enforce a contract that an **underlying tool already enforces at runtime**, ask:

> **What value does this validation layer provide that the underlying tool doesn't already provide?**

Two legitimate answers:

[cols="1,1", options="header"]
|===
| Answer | Where the check belongs

| Edit-time enforcement so prose / docs / call-sites stay aligned with the live schema
| Dev-time static analyzer (lint pass, CI gate, plugin-doctor rule)

| Runtime safety the underlying tool doesn't provide
| Keep the runtime layer — but first verify the underlying tool genuinely doesn't provide it
|===

When neither answer holds — the underlying tool's runtime behaviour is the contract — delete the safety-net layer. The underlying tool's exit code, surfaced cleanly, is the contract. If post-hoc cleanup is desired, a retrospective sweep produces structured findings without blocking workflows.

This pattern applies wherever runtime validation is built on top of static analysis of declarative schemas: `argparse` / `click` / `typer` for CLI surfaces, `pydantic` / `dataclasses` for value objects, type-checker output for type contracts. AST extraction is appropriate only when declarations are **always** module-top-level literals; the moment any declaration shifts to a loop / conditional / runtime-resolved form, the static-analysis layer becomes load-bearing fragile code enforcing a wrong specification.
