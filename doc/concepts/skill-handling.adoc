= Skill Handling
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] ┬╗ xref:README.adoc[Concepts]

In an unconstrained Claude Code session, "which skill applies here" is a runtime guess the model makes from whatever the prompt happened to surface. Two tasks with similar wording can pull in different skill sets and produce divergent outputs. The skill load isn't even logged in a way that lets you check after the fact. Plan Marshall declines to play that game. Every skill loaded by a subagent is **named by the dispatch site**, resolved against `marshal.json`, materialised into the task record at plan time, and loaded in a deterministic order at dispatch time. The LLM running the subagent has zero discretion about its own context ŌĆö it loads what it is told.

Two consequences fall out of that:

* **Reproducibility.** The same `(module, profile)` pair produces the same `task.skills` list, which produces the same loaded context across runs.
* **Auditability.** Every `Skill:` load lands in `work.log` with the plan id, the dispatch name, and the skill notation. The log is the authoritative answer to "what was in this subagent's context."

== Three tiers

Every dispatch's context is built from up to three tiers, stacked in order:

[cols="1,1,3", options="header"]
|===
| Tier | Source | Role

| **Tier 1 ŌĆö System** | `skill_domains.system.defaults` in `marshal.json`. Loaded unconditionally by the dispatcher. | The foundation that *every* subagent gets ŌĆö `dev-agent-behavior-rules` is the canonical Tier-1 skill, carrying the hard rules.
| **Tier 2 ŌĆö Domain** | `skill_domains.{domain}` in `marshal.json`, materialised into `task.skills` via per-module `skills_by_profile`. | The domain knowledge appropriate to *this* task: Java patterns, JUnit conventions, frontend ESLint rules, OCI container standards.
| **Workflow + Extensions** | Profile-routed (`execute-task` for phase-5) or finding-routed (`ext-triage-{domain}` during triage). Resolved via `manage-config` lookups and passed through `skills[]`. | The dispatch-site-specific logic: the workflow body that knows what to do, plus per-domain triage knowledge for finding-handling.
|===

Tier 1 always loads first. Tier 2 and the workflow / extension entries load in the order the dispatch site arranged them ŌĆö typically domain knowledge before workflow logic so the workflow can apply the loaded standards.

== From architecture to dispatch

Skills travel through the planning pipeline as a deterministic data flow. Nothing about "which skills load" is decided by the model at runtime.

image::../resources/diagrams/skill-loading-flow.svg[Skill loading data flow ŌĆö architecture's skills_by_profile becomes task.skills at plan time, travels through the dispatch verbatim, and is loaded by execution-context as Tier 2 on top of the implicit Tier-1 dev-agent-behavior-rules, align=center]

Each transition is a `manage-*` script call or a workflow-driven assignment, not an LLM inference. The architecture-enrichment step records the per-module `skills_by_profile` map; phase-4-plan materialises it into per-task `task.skills` arrays; the dispatcher reads those arrays and issues a `Skill:` load per entry. A skill that didn't enter `skills_by_profile` at enrichment time will not be in `task.skills` at plan time and will not load at dispatch time. The trail is checkable end-to-end.

== Profiles

The profile is the *category of work* a task represents ŌĆö production code vs test code vs integration test vs quality / docs. It drives two things: which skills enter the dispatched context (the Tier-2 selection from `skills_by_profile`), and which workflow body executes the task (the system default is `execute-task`, but a domain can override it per profile).

Profile selection happens at architecture-discovery time, signal-based per module: a Java module with sources gets `implementation` and `quality`; with a Failsafe plugin or testcontainers dependency it adds `integration_testing`; a documentation module with `doc/*.adoc` files gets `documentation`. The detection rules per profile live in `extension-api/standards/profiles.md`. A sixth slot, `core`, is special: it carries foundation patterns that always merge into every other profile's load.

== Auditing what loaded

Grep the work log to answer "what did task N actually load." If a skill is missing from the log, it never loaded ŌĆö regardless of what the workflow body claims to have applied. The work-log line format and the `[SKILL]` category tag live in `manage-logging`'s standards; the dispatch-site instrumentation is in the execution-context agent body. Plan Marshall does not assume the LLM remembered to load anything; the log is the record of what actually happened.
