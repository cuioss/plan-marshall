= ADR-002: Plan-scoped operations move into a cwd-pinned hermetic worktree

== Status

Accepted

== Context

A plan-scoped operation MUST resolve against, and write into, ONLY the plan's own
worktree-resident state while that plan is executing. It MUST NOT touch shared
main-checkout state, repo-root state, or host-global state during execution. This is the
boundary invariant the architecture enforces.

Plan-marshall runs a plan through six phases. Phases 1-4 (init, refine, outline, plan)
operate on the main checkout. Phase 5 (execute) and phase 6 (finalize) operate on an
isolated git worktree dedicated to the plan. The plan's non-git-controlled runtime state —
the plan directory (`.plan/local/plans/{plan_id}`) and any other runtime state required for
execution — must be reachable from whichever phase is running, and only one authoritative copy
of that state may exist at any moment so that no reconciliation or merge step is ever required
between a worktree copy and a main copy. The executor (`.plan/execute-script.py`) is the one
exception: it is per-tree DERIVED state (generated by `generate_executor.py`), not moved plan
state — main keeps its copy and the worktree generates its own (see _The executor is per-tree
derived state_ below).

Concurrent plans may reach finalize at the same time. Their write-back to main (moving the
plan directory back, merging the feature branch) must be serialized so that two finalizes
never interleave their main-checkout mutations.

== Decision

=== The move-based, cwd-pinned model

Phases 1-4 run on the main checkout: the working directory IS the main checkout, so every
`.plan/` lookup resolves to main's `.plan/local`.

At phase-5 start the worktree is materialized and the plan-scoped, non-git-controlled state
is MOVED (not copied) into it: the plan directory and any other required runtime state. The
executor is NOT moved — it is per-tree derived state, generated into the worktree at move-in
while main keeps its own copy (see below). After the move there is exactly one authoritative
copy of the plan directory — it lives in the worktree, and main no longer holds it during
execution. Because there is only one copy, there is no reconcile-or-merge problem between a
worktree copy and a main copy.

Phase-5+ pins the working directory to the worktree root. Resolution is then a SINGLE uniform
cwd/worktree-relative rule, evaluated in this precedence:

1. an explicit `set_base_dir()` override,
2. the `PLAN_BASE_DIR` environment override,
3. a walk up from the current working directory to the nearest ancestor containing `.plan/local`.

There is no per-phase branch in the resolver and no sideways indirection. Phases 1-4 resolve
to main because the working directory is main; phase-5+ resolve to the worktree because the
working directory is pinned there. `PLAN_BASE_DIR` is the sole environment override and exists
as a test-isolation hook.

The SOLE execution-time invariant is that the working directory is never changed away from the
worktree during phase-5+.

=== Implementation shape: two atomic lifecycle scripts

The move-in and the move-back are each performed by ONE dedicated atomic script in the
`workflow-integration-git` skill, alongside the other worktree-lifecycle machinery.

`prepare_execute.py` (notation `plan-marshall:workflow-integration-git:prepare_execute`)
materializes the worktree and feature branch, moves the plan-scoped non-git state into the
worktree, and RETURNS the canonical worktree path together with a status. It is
atomic-with-rollback (a failed move-in leaves the plan state wholly on main, never half-moved)
and idempotent (an already-moved plan is a no-op success returning the same path).

`integrate_into_main.py` (notation `plan-marshall:workflow-integration-git:integrate_into_main`)
acquires the merge lock, folds the plan's own global logs into the plan directory, moves the
plan directory back to main, and releases the merge lock. It does NOT regenerate the executor
(see _The executor is per-tree derived state_ below). It runs while the worktree is still
present; worktree removal happens after it returns.
It is atomic-with-rollback (the lock is released on every exit path) and idempotent (an
already-moved-back plan is a no-op success).

=== CWD is the caller's responsibility

A subprocess cannot change its parent's working directory. Neither lifecycle script mutates
the caller's working directory. Each script RETURNS a status; the caller pins ITS OWN working
directory to the returned path (mirroring `manage-status get-worktree-path`). A script MAY
assert/guard that it was invoked from the expected working directory — that cheap caller-side
guard is the script-side realization of the single cwd-unchanged invariant. The phase-5
orchestrator calls `prepare_execute.py`, reads the returned worktree path, and pins its own
working directory there for the remainder of phase-5+; the finalize orchestrator returns its
own working directory to main after `integrate_into_main.py` completes.

Cross-session re-entry from the main checkout re-anchors cwd to the worktree as a supported
mechanism. The single-session case above pins cwd once, at move-in, inside the orchestrator that
performed the move. A *fresh* session — a new shell entering `/plan-marshall action=execute|finalize
plan={plan_id}` against a plan already past phase-5 — starts from the main checkout, where the
plan directory no longer lives (it was moved into the worktree at move-in). That session resolves
the plan's checkout location via the read-only `locate-plan-checkout` verb
(`plan-marshall:workflow-integration-git:git-workflow locate-plan-checkout --plan-id {plan_id}`),
which reports `current` (plan dir on the current checkout), `worktree` (plan dir moved into a
registered worktree, with the resolved `worktree_path`), or `not_found`. On `worktree` the session
issues a standalone `cd {worktree_path}` and continues; after the `cd`, the single uniform
cwd-relative rule resolves all `.plan/` lookups to the worktree-resident copy, exactly as in the
single-session case. The re-anchor is idempotent — a session already cwd-pinned inside the worktree
resolves `current`, so there is no double-`cd`. The verb reuses the same shared resolution channel
(`_resolve_worktree_path_for_plan` over `manage-status get-worktree-path`) as the other worktree
verbs rather than re-parsing `git worktree list --porcelain`.

=== The executor is per-tree derived state

The executor (`.plan/execute-script.py`) is a DERIVED artifact (generated by
`generate_executor.py`), not plan state — so it is NOT a moved slot. Main keeps its executor
present and untouched throughout phase-5+, and the worktree generates its OWN executor at
move-in: `prepare_execute.py` invokes `generate_executor.py generate --marketplace-root {worktree}`
with the subprocess working directory pinned to the worktree, so the cwd-relative output path
lands inside the worktree (never on main). Keeping main's executor always present is what lets
main-anchored hooks — the `session render-title` `UserPromptSubmit` hook and its siblings, which
shell out to `.plan/execute-script.py` resolved against the main checkout — keep working while
any worktree-backed plan is mid-phase-5. Treating the executor as a moved slot was the original
boundary defect this ADR's amendment corrects.

Regenerating the on-main executor when a plan changes the marketplace script SET (newly added
notations must resolve after merge) remains necessary, but it is a PROJECT-LEVEL, meta-project-only
finalize step — `finalize-step-sync-plugin-cache`, run after the cache sync — NOT a responsibility
of `integrate_into_main.py`. Only the plan-marshall meta-project's own plans ever modify
`marketplace/bundles/*/skills/*/scripts/`, so only that project ever needs the on-main mapping
rebuilt; consumer projects never regenerate. The executor is therefore distinct from BOTH the
moved plan directory AND the three main-anchored shared corpora below: it is independently derived
in each tree, never a moved slot and never routed through `resolve_main_anchored_path`.

=== One sanctioned main-anchored resolver utility for three shared corpora

A small, bounded set of `.plan/local` state is genuinely shared across sessions and must
resolve to the MAIN checkout regardless of which worktree a caller's cwd is pinned to. There
are exactly THREE such corpora:

* `merge.lock` — the cooperative finalize lock (a single lock file whose contents record the
  holder `plan_id` / session). `merge_lock.py`
  (notation `plan-marshall:workflow-integration-git:merge_lock`) provides `acquire` (atomic
  `O_EXCL` create writing the holder, simple-backoff retry, minimal stale-holder reclamation)
  and `release`; `integrate_into_main.py` acquires it before move-back/merge and
  releases it on every exit path.
* `run-configuration.json` — the adaptive-timeout corpus written/read by every plan
  (`manage-run-config/run_config.py`).
* `lessons-learned` — the global lessons corpus, including the id-allocation `plans` scan
  (`manage-lessons/manage-lessons.py`).

These three are reached through ONE sanctioned main-anchored resolver UTILITY:
`resolve_main_anchored_path` in `script-shared/marketplace_paths.py`. It resolves a subpath
under the MAIN checkout's `.plan/local` cwd-independently — honouring the
`set_base_dir()` / `PLAN_BASE_DIR` test override first, then resolving via
`git rev-parse --git-common-dir` (which points at main's `.git` even from a linked worktree).
All three consumers CALL the utility rather than each carrying its own git-common-dir copy;
`merge_lock.py` is now one of three consumers, not the sole owner of the resolution.

This is the SINGLE, deliberate exception to cwd-relative resolution: every other resolution in
the codebase is uniform cwd-relative. Confining the main-anchored resolution to ONE small,
explicitly-enumerated utility — used by exactly three consumers — prevents a pervasive
main-anchored resolution layer from re-forming; new cross-session shared state MUST route
through the utility rather than re-implement git-common-dir resolution. Correspondingly, the
worktree's `.plan/local` carries NO symlinks: it is a fully real directory, and cross-session
visibility comes from the utility, not from any filesystem symlink back to main.

== Consequences

=== What the move model subsumes

The move-based cwd-pinned model collapses the shared-versus-worktree split, so several
instances of the boundary invariant fall out structurally and need no independent fix:

* *Two-dot diff versus merge-base* — with plan state and execution pinned to one authoritative worktree, the plan's diff surface is the worktree branch's own `base...HEAD`; there is no stale-worktree-tree-versus-advancing-shared-tip framing.
* *Recursive walk into a sibling worktrees tree* — a recursive walk rooted at the pinned worktree never reaches sibling worktrees, because plan state and source no longer share a `.plan/local` parent reached sideways; the cross-worktree recursion surface does not exist.
* *Finalize-step discovery against the main-checkout working directory* — cwd-pinning during phase-5+ makes discovery resolve against the tree that holds the feature-branch step, so a step that lives in the worktree is found.
* *`marshal.json` re-canonicalization drift* — a config save from the pinned worktree resolves to the worktree-resident copy via the uniform cwd rule, so it does not touch shared main-checkout config during execution.

=== What stays independent

These remain orthogonal to the shared-versus-worktree split and are not addressed by the move
model:

* The `.plan/temp` lifecycle reaper is a lifecycle gap, not a resolution gap.
* `.claude/skills/` routing relative to the module-scoped quality gate is a gate-routing gap.
* Fresh-worktree virtualenv versus a pinned CI toolchain is a toolchain-pinning gap.

=== Which state moves and which stays main-shared

The plan directory MOVES into the worktree during phase-5+ and moves back at finalize. The
executor is per-tree DERIVED state — main keeps its copy present throughout phase-5+, the
worktree generates its own at move-in, and the on-main copy is refreshed by the project-level
`finalize-step-sync-plugin-cache` step when a plan changed the marketplace script set; it is
neither a moved slot nor a main-anchored shared resource. The three genuinely-shared
cross-session corpora — `merge.lock`,
`run-configuration.json`, and `lessons-learned/` — stay main-anchored by design, each resolved
through the single `resolve_main_anchored_path` utility rather than via filesystem symlinks
(the worktree's `.plan/local` is fully real, no symlinks). `archived-plans/` is also
main-resident but is touched only at finalize after cwd has returned to main, so the uniform
cwd rule suffices and it needs no resolver.

=== Concurrent-session visibility

While a plan is executing, its plan directory lives in its worktree. A main-checkout listing
that resolves the plan directory via the uniform cwd rule from main will not show that plan
mid-flight; the plan reappears in the main-checkout listing after the finalize move-back. This
is expected behaviour.
