= Simplification
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:README.adoc[Concepts]

LLMs over-build. Asked for one retry, they reach for a `RetryStrategy` interface with a single implementation and a backoff knob no caller sets. Asked to move a symbol, they leave a re-export shim "for compatibility". Asked to handle a failure, they wrap an already-handled exception in a defensive catch-all. Every speculative parameter, shim, and abstraction layer is something a future reader must understand, a static analyzer must scan, and a maintainer must keep correct — and the surplus is cheap to add later against a real requirement but expensive to retrofit-remove once callers depend on its accidental presence. Telling the model "keep it minimal" once does not hold across a multi-task plan.

Plan Marshall responds with four cooperating mechanisms ordered from soft (skill-loaded posture) to hard (mechanical static rule). Each layer assumes the layer above it can be bypassed: the posture sets the intent, the per-action rule applies it at authoring time, the finalize pass reasons over everything the authoring rule missed, and the static rules mechanically catch the deterministically-recognisable patterns. The same anti-pattern catalogue — the seven "minimum viable code" anti-patterns — anchors all four, so the layers reinforce one definition rather than drifting apart.

== Layer 1 — The simplicity posture

Every plan records a `simplicity` posture at refine/outline time — one of `lean`, `pragmatic`, or `defensive` — and that value travels with the plan into every downstream step. The posture is not a rule the model follows directly; it is a dial that tunes how aggressively the deeper layers delete surplus structure. `lean` deletes everything not justified by a live caller; `pragmatic` keeps low-risk surplus; `defensive` only flags the clearest cases. By resolving the posture once and threading it through finalize-step-simplify (xref:planning-workflow.adoc[Concepts › Planning Workflow] persists it in the phase-2-refine config block), Plan Marshall makes "how minimal should this changeset be" an explicit, audited plan input rather than a per-task improvisation.

**Strength**: a skill-loaded *posture*. It sets intent and tunes the layers below it, but does not itself remove a single line. The layers below it turn the posture into concrete deletions.

== Layer 2 — The minimum-viable-code rule

`plan-marshall:dev-agent-behavior-rules` carries Principle 7 — "Implement the Minimum, Not the Maximum" — and every plan-marshall subagent loads that skill as Tier-1 before any work (see xref:skill-handling.adoc[Skill Handling]). The rule directs the model to write only what the present requirement asks for: no error handling for failures that cannot occur, no configurability for callers that do not exist, no abstraction for a second implementation that is not planned, no comment that restates well-named code. The full catalogue — the seven anti-patterns and their Trigger / Detection / Action treatment — lives in `dev-general-code-quality` `standards/code-organization.md` `#minimum-viable-code`, which both Principle 7 and the deeper layers cite as the single source of truth.

**Strength**: skill-loaded *guidance* applied at authoring time. The model might still over-build. Layer 2 reduces the probability one edit at a time; the layers below it catch what slips through.

== Layer 3 — finalize-step-simplify

The dynamic, judgement-driven layer. The `default:finalize-step-simplify` phase-6 step runs a cognitive simplification pass over the plan's own change surface — bounded to the `modified_files` recorded in `references.json` — BEFORE the commit is materialised. It dispatches a domain-agnostic review (loading only the three foundation standards: Principle 7, the minimum-viable-code anti-patterns, and the resolved simplicity posture string) that reasons about every anti-pattern the static rules cannot mechanically recognise: unused parameters, thin re-export shims, defensive catch-alls around already-handled failures, near-identical helpers collapsible into one, signature-restating docstrings, single-caller config keys, and speculative abstractions with no second implementation. Surplus is deleted directly in the worktree; a deletion that would change a public element or could plausibly serve an imminent requirement is left in place and recorded as a finding instead.

**Strength**: a *finalize-time cognitive pass*. It reasons rather than pattern-matches, so it catches the instances no regex can name — at the cost of being LLM judgement, gated by the posture and the modified-files bound rather than mechanically guaranteed.

== Layer 4 — plugin-doctor static rules

The mechanical layer. The `SIMPLICITY_*` rule cluster runs inside `plugin-doctor` (behind the `doctor-marketplace analyze` interface, alongside `argparse_safety`) and flags the deterministically-recognisable subset of the anti-patterns in marketplace bundle scripts at edit time: `SIMPLICITY_UNUSED_PARAMETER` (a `del <param>` or `# unused` marker), `SIMPLICITY_BACKWARD_COMPAT_REEXPORT` (an import tagged `# backward compat` / `# re-exported for`), `SIMPLICITY_DEFENSIVE_CATCHALL` (an `except` handler tagged `# defensive only`), `SIMPLICITY_THIN_WRAPPER` (a body that is a single forwarding `return <call>(...)`), and `SIMPLICITY_SIGNATURE_DOCSTRING` (a docstring that only restates `Args:`/`Returns:` headers). These are the static counterpart to Layer 3's reasoning: the doctor catches the patterns a rule can name with certainty, finalize-step-simplify reasons about the rest.

**Strength**: a *mechanical static rule*. It fires the same way every run with no judgement, so it is the most reliable of the four — but it only sees the patterns expressible as a deterministic check, which is why Layer 3 sits above it to reason about everything else.

== What the layers add up to

The four layers compose into a posture-then-mechanism funnel anchored on one anti-pattern catalogue. The simplicity posture (Layer 1) sets how aggressive the whole stack should be. Principle 7 (Layer 2) applies the catalogue at authoring time, one edit at a time. Whatever the model still over-builds is reasoned about by the finalize-time cognitive pass (Layer 3) against the plan's actual changeset. And the deterministically-recognisable patterns are mechanically refused at edit time by the static `SIMPLICITY_*` rules (Layer 4). Authoring guidance reduces the surplus that gets written; the finalize pass and the static rules remove the surplus that gets written anyway — the cognitive pass for the judgement calls, the static rules for the patterns a check can name.

[CAUTION]
====
**Does this guarantee minimal code? No.**

Layer 1 is a dial, not a deletion. Layer 2 is skill-loaded text the model can ignore. Layer 3 is LLM judgement bounded by the modified-files list and the posture — it reasons well but is not mechanical, and it deliberately leaves public elements and imminent-requirement structure in place rather than deleting them. Layer 4 is mechanical but narrow: it only fires on the patterns a static check can recognise with certainty, and only on marketplace bundle scripts.

The point is not "the code cannot carry surplus." The point is that surplus is *progressively less likely to survive* the deeper a plan runs: discouraged at authoring time, reasoned away at finalize time, and mechanically flagged at edit time for the patterns a rule can name. What remains is a deliberate residue — public surfaces and imminent-requirement structure the layers were instructed to leave — recorded as findings the developer can read rather than silently kept.
