= Recipes
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:README.adoc[Concepts]

The standard pipeline re-discovers the scope of every plan from first principles — refine iterates on confidence, outline walks the architecture, plan creates tasks. Useful for the request the user just typed; wasteful for the transformation you have already done forty times. Recipes are the workflow's escape hatch for that case: predefined, repeatable transformations whose plan shape is already known, so the lifecycle skips the refine confidence loop and the outline-discovery work and produces deliverables from a deterministic mapping.

== Overview

The standard pipeline (xref:planning-workflow.adoc[Planning Workflow]) assumes a request whose scope must be *discovered*: phase-2-refine iterates on confidence, phase-3-outline's Complex Track walks the architecture and proposes deliverables, phase-4-plan creates tasks. That is the right shape for "build the thing the user just described." It is the wrong shape for recurring transformations like "refactor every package to the configured profile standards" or "verify every architecture diagram is up to date" — there the plan shape is *already known*, and asking the LLM to re-discover it from scratch each time burns tokens for no gain.

Recipes are the extension point for that case. A recipe declares "I know what this plan looks like; ask me, not the LLM."

== How the lifecycle changes

A recipe-driven plan still passes through every phase, but the work inside the early phases shortcuts. Refine bypasses its confidence iteration and returns immediately. Outline routes to the recipe skill instead of the Simple- or Complex-Track LLM analysis; the recipe skill produces the link:../../marketplace/bundles/plan-marshall/skills/manage-solution-outline/SKILL.md[solution outline] via its own deterministic discovery + deliverable mapping. The Q-Gate at outline is bypassed when the recipe yields a surgical single-deliverable shape and runs normally otherwise. From phase-4-plan onward, the plan flows through the same pipeline as any other.

The cost saving is real: a Complex-Track outline on a multi-module codebase is a multi-thousand-token LLM exercise. A recipe-driven outline producing the same N deliverables is mostly script invocations (`architecture modules`, `architecture module --full`, `manage-files discover`) plus a templated write. The shortcut shows up directly in the per-phase metrics for recipe-driven runs.

User entry point:

[source,bash]
----
/plan-marshall action=recipe
----

The list pulls from `manage-config list-recipes`, the user picks one, and `phase-1-init` writes the recipe metadata into `status.json` (`recipe_key`, `recipe_skill`, `recipe_domain`, `recipe_profile`, `recipe_package_source`).

== The contract

A recipe is a skill whose SKILL.md declares `implements: plan-marshall:extension-api/standards/ext-point-recipe`. It is registered for discovery by one of two routes — both fully documented in link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-recipe.md[`ext-point-recipe.md`]:

* **Extension-provided** — a domain bundle implements `provides_recipes()` and returns a list of recipe dicts (`key`, `name`, `description`, `skill`, `default_change_type`, `scope`).
* **Project-local** — the project ships its own recipe under `.claude/skills/recipe-*` (mirrors the finalize-step pattern).

At runtime the recipe skill is invoked under phase-3-outline Step 3 with `plan_id`, optional `recipe_domain`, optional `recipe_profile`, and optional `recipe_package_source`. It is responsible for:

1. Resolving the applicable skills for the recipe's profile (e.g. via `manage-config resolve-domain-skills`).
2. Querying the architecture for the input set (modules, packages, diagrams, ADRs, …).
3. Mapping the input set to deliverables — one row per "unit of repetition" (one deliverable per package, per diagram, …).
4. Writing the link:../../marketplace/bundles/plan-marshall/skills/manage-solution-outline/SKILL.md[solution outline] via `manage-solution-outline` so the file is structure-validated like any other outline.

The recipe skill is the *single source of truth* for what its deliverables look like — phase-3-outline does not second-guess it.

== Recipe-match routing — reaching a recipe without naming it

The user entry point above (`/plan-marshall action=recipe`) is the *explicit* route: the user picks a recipe from the list. But a recurring transformation is often described informally — "clean up the logging across the module", "check the diagrams are current" — without the user knowing a matching recipe exists. *Recipe-match routing* (Tier 1 of the routing model) closes that gap: `phase-1-init` scores the request narrative against the live recipe registry and, when a recipe is a strong enough match, routes the informal request into that recipe's deterministic path instead of the standard discovery pipeline.

The scoring is deterministic and heuristic-first — `manage-config recipe-match` tokenizes the request text and scores keyword overlap against each registered recipe, returning a ranked `matches[]` list with per-match confidence. The init phase then gates on two configurable knobs:

* **Auto-route** — when `auto_route_recipe` is enabled and the top match's confidence clears `auto_route_recipe_threshold` (default `0.6`), the matched recipe is persisted (`status.metadata.recipe_key`) and the plan flows straight into the recipe path — no user prompt.
* **Propose** — when auto-route is disabled or the top match falls below the floor, the ranked matches are surfaced via a prompt so the user can pick a recipe or decline and proceed with the standard pipeline.

An explicit `recipe=` selection always wins — recipe-match never overrides a recipe the user named directly. The enforcement-critical process steps and the gate semantics live in xref:../../marketplace/bundles/plan-marshall/skills/phase-1-init/SKILL.md[`phase-1-init/SKILL.md`] Step 5c; the configurable knobs are documented in xref:../user/configuration.adoc#recipe-routing[Configuration › Recipe-match routing].

Recipe-match is paired with *request-aspect classification* (`manage-config aspect-classify`), which labels a request as analysis, planning, or implementation so the execution manifest can drop build / quality-gate / test verification steps for analysis- and planning-only requests that carry no production footprint. The aspect classifier uses its own independent confidence threshold (`0.7`) — not the recipe-match auto-route floor.

== The audit recipe family

Most recipes (above) are *plan-shape* recipes: they shortcut the outline-discovery work and emit a deterministic deliverable set into a plan. The *audit* recipe family is a second kind — standalone, single-envelope entry points that run a cognitive review over a footprint and emit *findings*, not deliverables:

* link:../../marketplace/bundles/plan-marshall/skills/recipe-code-review/SKILL.md[`recipe-code-review`] — a diff-aware structural/quality review; emits `lint-issue` findings.
* link:../../marketplace/bundles/plan-marshall/skills/recipe-security-audit/SKILL.md[`recipe-security-audit`] — an on-demand security audit using best-effort security context (`persona-security-expert` + `untrusted-ingestion` + `workflow-permission-web`); emits `bug` / `anti-pattern` findings.

Both take an optional path/file `scope` and an optional explicit `base_branch` (defaulting to the current-branch-vs-base diff), run inside their own plan-directory, and emit into `manage-findings` so the discovered issues flow through the same triage / suppression / loop-back / re-review pipeline every findings producer uses — the structural difference from external audit tools, which print a report and stop. This is the *universal-sink* principle: anything that discovers a problem emits into the findings pipeline rather than a one-off report.

=== The shared audit engine

`recipe-security-audit` does not inline its procedure. The five-stage security-audit procedure (compute footprint → detect affected domains → gather security context → run the cognitive audit → emit findings and dispatch to triage) is authored once in link:../../marketplace/bundles/plan-marshall/skills/recipe-security-audit/standards/audit-engine.md[`recipe-security-audit/standards/audit-engine.md`] and consumed by two callers running the *identical* stages: this recipe (the on-demand entry point) and — later — workstream 05's `finalize-step-security-audit` (the automatic pre-ship gate). Workstream 05 reuses the engine *additively* at stage 3 only — it supplies per-domain `skills_by_profile.security` skills as an extra context input layered on top of the engine's fixed action-general set, with stages 1, 2, 4, and 5 unchanged. The engine is never reshaped; the on-demand and automatic callers share it.

== Where this fits

Recipes are the workflow's *known-shape escape hatch*. The standard pipeline is what makes Plan Marshall safe for arbitrary requests; recipes are what make it efficient for recurring ones. The hooks the recipe pattern rides on (xref:execution-context.adoc[execution-context dispatch], xref:skill-handling.adoc[skill loading], xref:audit-trail.adoc[audit trail]) are the same — the recipe just supplies its own discovery step.

Recipes do not make the recurring shape any less recurring — the underlying drudgery is still there, just dispatched efficiently. The recipe pattern is for "this is going to happen again", not "we have solved the underlying problem."
