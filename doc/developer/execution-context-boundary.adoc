= Plan-Worktree Execution-Context Boundary — Design Discussion
:nofooter:
:toc: left
:toclevels: 3

xref:../../README.md[Plan Marshall] » xref:README.adoc[Developer Guide]

[NOTE]
====
This is a **design discussion**, not a finalized decision record. It enumerates a
recurring problem class, proposes an invariant, and lays out enforcement options
for review. No code change is implied by this document's existence. Once the
invariant and enforcement mechanism are agreed, the chosen approach should be
recorded as an ADR and implemented through one or more scoped plans.

Empirical motivation: lesson `2026-06-01-20-002` (the worktree-bound executor,
3 recurrences) and the cross-lesson analysis it triggered. See
<<corpus>> for the full lesson list.
====

== Problem statement

A plan runs in an isolated git worktree under `.plan/local/worktrees/{plan_id}/`.
The worktree exists so the plan can edit, build, and test without touching the
main checkout or any sibling worktree. The isolation is, however, **only
partial and unenforced**: several plan-scoped operations silently *resolve
against*, *write into*, or *compute over* shared, main-checkout, repo-root state
that other concurrent sessions depend on.

The defect signature is always the same:

* It is **silent at the originating task** — the operation appears to succeed.
* It **surfaces later** — at a downstream task, in a concurrent session, after
  merge, or in CI.
* The fix that ships is **per-symptom** (restore-after, reorder steps,
  regenerate, narrow a scan) rather than an invariant that prevents the whole
  class.

The canonical instance is the shared executor. `.plan/execute-script.py` is a
**single file at the main-checkout repo root** (resolved via
`git rev-parse --git-common-dir`, shared by every worktree). When a
worktree-scoped step regenerates it with worktree-bound script paths and a later
`branch-cleanup` removes that worktree, *every* `python3 .plan/execute-script.py`
call — in this session and every concurrent session — breaks with
`ModuleNotFoundError` until `regenerate-executor` re-binds to main. This was
observed **live** during the design investigation: the shared executor's
PYTHONPATH lines pointed at a deleted
`.plan/local/worktrees/reconcile-files-leaves-upstream-absorbed-files-in/...`
tree.

== The shared-vs-isolated state map

The split is resolved in
`marketplace/bundles/plan-marshall/skills/script-shared/scripts/marketplace_paths.py`
(`git_main_checkout_root()` → `git rev-parse --path-format=absolute
--git-common-dir`) and built upon by `tools-file-ops/scripts/file_ops.py`.

[cols="3,1,4", options="header"]
|===
| State | Scope | Resolver / note

| `<main>/.plan/execute-script.py` (the executor)
| SHARED
| `file_ops.get_executor_path()`; `generate_executor.executor_path()` →
  `get_tracked_config_dir()` (git-common-dir). Single host-critical file; no
  per-worktree variant. Its embedded `sys.path.insert` / `_ALL_SCRIPT_DIRS`
  PYTHONPATH lines *can be made worktree-specific* by the `--marketplace-root`
  argument while the write target stays the shared path.

| `<main>/.plan/marshal.json`
| SHARED
| `file_ops.get_marshal_path()` / `get_tracked_config_dir()`. Writes go through
  `manage-config`, which always resolves to the main checkout — so config
  *writes* are not worktree-bound, but *discovery* that feeds them can be (see
  finalize-step discovery below). Note also that `save_config()` rewrites the
  file with canonical key order and load-time default back-fill, so *any*
  `manage-config` call that reaches the save path — including nominally
  read-only or effort-resolution calls during the worktree-deferred phases
  (states 1–2, which run against the main checkout) — leaves the SHARED
  `marshal.json` dirty in the main checkout even with no semantic change.
  Observed live (this design pass): dirtied during `phase-2-refine`
  (`lightweight_track_override` default back-fill + key reorder) and
  `phase-3-outline` (`phase-6-finalize.steps` reordered to frontmatter `order`).
  This drift is invisible to the `main_dirty` handshake invariant — see
  <<marshal-drift-gap>>.

| `<main>/.plan/local/` (plans, tasks, logs, findings, handshakes,
  `marshall-state.toon`)
| SHARED
| `get_base_dir()`. Intentionally shared so every worktree sees the same plan
  metadata (Bucket A). Honors `PLAN_BASE_DIR` for tests.

| `~/.claude/plugins/cache/plan-marshall/`
| HOST-GLOBAL
| Singleton per machine. Regenerating from a worktree swaps cache contents for
  every project on the host.

| `<main>/.plan/temp/`
| PER-WORKTREE
| `get_temp_dir()` keeps temp project-local per worktree by design. But it has
  **no lifecycle reaper** — `archive-plan` never sweeps a plan's own scratch, so
  it accumulates as a shared graveyard (lesson `2026-05-31-20-003`).

| `.plan/local/worktrees/{plan_id}/` source trees
| PER-WORKTREE
| `get_worktree_root()`. Isolated by design — but a recursive walk rooted above
  it (e.g. `rglob` from `.plan/local/`) recurses into *every* worktree checkout
  (lesson `2026-06-02-10-002`).

| `origin/main` tip
| SHARED + MOVING
| Two-dot diffs against the advancing tip (rather than the plan branch's own
  merge-base) bloat the plan footprint when the worktree is stale relative to
  main (lesson family `2026-05-28-20-002`).
|===

The architecture is sound **except** that there is no enforced rule keeping
plan-scoped operations on the isolated side of this line.

== The invariant already exists — as prose

The principle this whole class needs is **already written down**, in
`marketplace/bundles/plan-marshall/skills/tools-script-executor/standards/cwd-policy.md`.
It defines three buckets:

* **Bucket A — plan metadata** (cwd-agnostic): resolve `.plan/` via
  `get_base_dir()` → git-common-dir. Shared by design.
* **Bucket B — worktree-scoped operations** (explicit path): build/test/lint/CI
  accept `--plan-id` or `--project-dir`; never infer the tree from cwd.
* **Bucket C — meta-tools** (always main checkout): tools that read/write
  `marketplace/bundles/`, the plugin cache, or the executor "**must refuse to
  operate against a worktree path.**" `generate_executor` is explicitly listed
  as Bucket C (`cwd-policy.md` lines 63, 73).

So the executor bug is, precisely, a **Bucket-C violation**. The gap is not a
missing idea — it is that **Bucket C exists only as prose, is unenforced, and is
actively contradicted by the repo's own tooling**:

[cols="3,4", options="header"]
|===
| Site | Violation

| `.claude/skills/finalize-step-plugin-doctor/SKILL.md` Step 4 (order 65)
| Runs `generate_executor generate --marketplace-root {worktree_path}` with no
  isolation and no restore — writes worktree-bound mappings into the shared
  executor. The in-flight plan `worktree-bound-executor-regeneration-is-never-rest`
  fixes *this* caller via a scratch `--output` path.

| `marshall-steward` (`references/menu-maintenance.md:87`,
  `references/wizard-flow.md:121`)
| Explicitly instructs `generate_executor generate --marketplace-root "$REPO_ROOT"`
  when inside a worktree — the steward's *own prose* directs the Bucket-C
  violation. Untouched by the in-flight plan; the next `/marshall-steward` from a
  worktree re-creates the identical corruption.

| `generate_executor.py` itself
| **Zero** worktree guards. `executor_path()` always returns the shared path;
  nothing refuses a worktree `--marketplace-root` when the write target is the
  shared file.
|===

No automated gate catches a worktree-bound executor: the phase-handshake
`config_hash` invariant hashes only `manage-config` output, not the executor's
embedded paths; `generate_executor verify`/`paths`/`drift` exist only in
marshall-steward's **manual** health-check menu. Worse, the handshake itself
subprocesses *through* the shared executor — so a corrupted executor breaks the
very check that might catch it.

[#marshal-drift-gap]
[NOTE]
====
**The same blind spot applies to `marshal.json` content drift.** The
phase-handshake `main_dirty` invariant captures `git status --porcelain` against
the main checkout *filtered to exclude `.plan/` paths*
(`workflow-integration-git/standards/worktree-handling.md`), so a `manage-config`
side-effect that dirties `.plan/marshal.json` in the main checkout is invisible
to Layer D — the phase agent reports `main_dirty: 0` while the shared file is in
fact modified. The only guard is the **refine-only** named-recovery assertion in
`plan-marshall/workflow/planning.md` (a `git -C . status --porcelain` check after
the phase-2-refine dispatch, with a `.plan/marshal.json` named-recovery revert).
`planning-outline.md` (3-outline, 4-plan) and the remaining phase boundaries
carry no equivalent assertion, so outside refine the leak passes silently — it
was caught during this design pass only by a manual `git status` between phases.
This is the config-state analogue of the executor blind spot above: the existing
invariant filters out the very surface that drifts, and the runtime check that
would catch it runs at only one of the boundaries where the leak occurs.
====

[#proposed-invariant]
== Proposed invariant

[IMPORTANT]
====
**A plan-scoped (worktree) operation must resolve and confine itself to its own
worktree/plan boundary. It must never silently fall through to the shared
main-checkout root, write into a tree other sessions share, or compute against
moving shared state.**

The dual of this for meta-tools (Bucket C): **a meta-tool that writes
host-global / shared state (the executor, the plugin cache, `marshal.json`) must
either run against the main checkout or write its worktree-true output to an
isolated scratch path — never overwrite the shared artifact with worktree-bound
content.**
====

This is a generalization of cwd-policy.md Bucket C from "meta-tools refuse
worktree paths" to "no plan-scoped op leaks onto the shared side, and the rule is
*enforced*, not merely documented."

== Enforcement options

The discussion to settle is *how* to make the invariant executable. These are
not mutually exclusive; the likely answer is a layered combination.

=== Option 1 — Hard guard in the producer (`generate_executor`)

When `output_path is None` (about to write the shared `executor_path()`) **and**
the resolved `marketplace_root` / `PM_MARKETPLACE_ROOT` is under
`.plan/local/worktrees/`, refuse (or force a scratch path).

* **Pro:** Makes the Bucket-C rule executable at the single choke point; protects
  *every* caller (plugin-doctor, marshall-steward, and any future one) in one
  move. Smallest blast radius.
* **Con:** Only covers the executor. marshall-steward's *intent* (a persistent
  worktree-bound executor for interactive dev) needs a deliberate answer — refuse
  outright, or route to `--output`?

=== Option 2 — Generalize the scratch-output primitive

The in-flight plan adds `--output PATH` to `generate_executor`. Make the same
"emit worktree-true output to an isolated scratch path, never the shared
artifact" pattern the standard mechanism for *all* meta-tools that have a
worktree-true mode (plugin cache regeneration, inventory scans).

* **Pro:** One primitive, consistently applied; the shared artifact is never
  mutated from a worktree by anyone.
* **Con:** Touches multiple tools; needs a convention for scratch-path naming and
  cleanup (ties into the `.plan/temp` lifecycle gap, lesson `2026-05-31-20-003`).

=== Option 3 — A plugin-doctor lint rule

A static rule that flags any workflow/skill text passing
`--marketplace-root {worktree_path}` (or any `.plan/local/worktrees/` path) to a
Bucket-C tool without `--output`, analogous to the existing
`manage-invocation-invalid` / `ARGUMENT_NAMING_*` cluster.

* **Pro:** Edit-time prevention; catches the violation in the source prose
  (exactly where marshall-steward's bug lives) before it ships.
* **Con:** Static text analysis can't see runtime-resolved paths; complements but
  does not replace a runtime guard.

=== Option 4 — A phase-handshake invariant

Add an executor-binding invariant to the phase-handshake registry that records
(and at guarded boundaries, verifies) that the shared executor's embedded paths
resolve to the main checkout, not a worktree.

* **Pro:** Runtime detection at phase boundaries; fits the existing invariant
  registry.
* **Con:** The handshake runs *through* the executor, so a corrupted executor may
  break the check itself — needs a self-contained probe that doesn't depend on
  the thing it validates. Detection, not prevention.

[#scope-boundary-for-follow-up-plans]
== Scope boundary for follow-up plans

The corpus shows this class is broader than the executor. The following are
**instances of the same invariant** but are **independent work** and should NOT
be folded into the executor fix:

* **Finalize-step discovery resolves against main-checkout cwd** — a project-local
  finalize step on a feature branch is invisible to `_discover_all_finalize_steps()`
  (lesson `2026-05-31-00-001`, UNBUILT). Needs a worktree-context override.
* **`.claude/skills/` project-local scripts escape the module-scoped quality-gate**
  (lesson `2026-06-01-19-001`, UNBUILT). Needs holistic-gate routing.
* **`.plan/temp` has no lifecycle reaper; working state leaks to harness memory**
  (lesson `2026-05-31-20-003`, UNBUILT, explicitly "design discussion required").
* **Two-dot diff against the advancing `origin/main` tip** instead of the plan
  branch's merge-base (lesson family `2026-05-28-20-002`, UNBUILT). Needs a shared
  `base...HEAD` plan-branch-only-diff primitive.
* **Fresh-worktree venv resolves a newer-than-CI toolchain** that lints the whole
  shared corpus (lesson `2026-06-02-10-004`, UNBUILT). Needs a pinned lint
  toolchain in the build env.
* **`manage-config` re-canonicalizes / back-fills `.plan/marshal.json` from
  worktree-deferred phases** — `save_config()`'s key-reorder + default back-fill
  dirties the shared main-checkout `marshal.json` as a side effect of
  read-only/effort-resolution calls during states 1–2 (observed live this design
  pass; not yet captured as a lesson). The `main_dirty` handshake invariant
  excludes `.plan/`, so the leak is undetected outside the refine-only
  named-recovery assertion (<<marshal-drift-gap>>). Candidate fixes: a
  no-op-write guard in `save_config()` (skip the write when the serialized
  content is byte-identical), and/or extension of the named-recovery assertion to
  the analytical-phase (3-outline / 4-plan) boundaries.

The two already-shipped narrow fixes — `PLAN_BASE_DIR`-honoring resolvers
(`2026-05-29-22-001`) and the opt-in-vs-autouse config fixture
(`2026-06-02-10-001`), plus the `rglob` depth-limit (`2026-06-02-10-002`) —
generalized their rule in prose but added **no enforcement**. They are evidence
that the invariant is real and recurring, and that "fix + durable-rule-in-prose"
has not stopped the recurrence.

== Recommended path

1. **Settle the invariant** (the box in <<proposed-invariant>>) and record it as
   an ADR.
2. **Enforce it for the executor first** — Option 1 (hard guard) + Option 3 (lint
   rule), because the executor is the highest-blast-radius shared artifact and has
   recurred 3×. This subsumes and completes the in-flight plan
   `worktree-bound-executor-regeneration-is-never-rest` (which currently fixes
   only one of two callers and adds no guard).
3. **Migrate the second offender** (marshall-steward) in the same plan as the
   guard, so the guard does not simply start rejecting the steward's documented
   workflow.
4. **Spawn independent plans** for the broader instances in
   <<scope-boundary-for-follow-up-plans>>, each gated on its own design where the
   lesson flags one (notably `2026-05-31-20-003`).
