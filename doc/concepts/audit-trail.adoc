= Audit Trail
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:index.adoc[Concepts]

Every Plan Marshall run leaves a structured, machine-readable record of *everything* it did — every dispatch, every script call, every skill load, every configuration decision, every finding, every phase boundary, every artifact produced. The record lives next to the plan on local disk, never leaves the developer's machine, and is detailed enough that the `plan-retrospective` skill can audit a plan automatically.

== Overview

Three telemetry layers, all produced as a side-effect of plan execution:

* **Logs** — line-oriented event stream covering status changes, dispatches, skill loads, decisions, script calls.
* **Structured artifacts** — every milestone document (`request.md`, `solution_outline.md`, `tasks/TASK-N.json`, `status.json`, `references.json`, `execution.toon`, `handshakes.toon`, `metrics.md`).
* **Findings** — append-only JSONL records for every Q-Gate verification, deliverable assessment, and PR review comment, with hash-id dedup and resolution tracking.

Storage layout under `.plan/local/{plans|archived-plans}/{plan_id}/` — gitignored, local-only.

== What gets recorded

=== Logs

Three line-oriented log files under `{plan_id}/logs/`, each with the same timestamp / hash-id / severity / context prefix shape:

[cols="1,3", options="header"]
|===
| File | Content

| `work.log` | The narrative event stream. Status changes, dispatches, artifact creation, skill loads, refine steps. Filterable by tag prefix.
| `decision.log` | Every config read, route decision, gate resolution, and judgement call the LLM logged. The "why this branch was taken" record.
| `script-execution.log` | Every `python3 .plan/execute-script.py` invocation — notation, args, exit status. The full record of which scripts ran.
|===

Tag prefixes inside `work.log` provide structure:

[cols="1,3", options="header"]
|===
| Tag | Meaning

| `[STATUS]` | Phase milestones — "Starting init phase", "Init phase complete", "Refine phase complete".
| `[DISPATCH]` | Every `Task: execution-context-{level}` subagent dispatch — target, level, role, workflow doc, plan id.
| `[SKILL]` | Every `Skill:` load inside an `execution-context` envelope — see xref:skill-handling.adoc#_auditing_what_loaded[Skill Handling § Auditing what loaded].
| `[ARTIFACT]` | Every artifact creation — "Created request.md", "Snapshotted architecture into architecture-pre/".
| `[REFINE:N]` | Phase-2 refine-iteration markers.
|===

Real samples from an archived plan:

[source]
----
[2026-05-15T09:42:13Z] [INFO] [17dda4] [ARTIFACT] (plan-marshall:phase-1-init) Created request.md from description
[2026-05-15T09:44:54Z] [INFO] [94549d] [DISPATCH] (plan-marshall:plan-marshall) target=execution-context-high level=high role=phase-2-refine workflow=plan-marshall:phase-2-refine/SKILL.md plan_id=worktree-integration-analysis
[2026-05-15T10:25:51Z] [INFO] [c52dc6] (plan-marshall:phase-3-outline:qgate-resolve) hash_id=882466 resolved as taken_into_account — Phasing Rationale block added to solution_outline.md…
----

The dispatch log contract (what fields every `[DISPATCH]` line must carry) is canonical at link:../../marketplace/bundles/plan-marshall/skills/ref-workflow-architecture/standards/dispatch-logging.md[`ref-workflow-architecture/standards/dispatch-logging.md`].

=== Structured artifacts

Every plan accumulates a fixed set of structured files. Each file is produced by a specific phase and carries a contract-validated shape. The canonical inventory lives in link:../../marketplace/bundles/plan-marshall/skills/ref-workflow-architecture/standards/artifacts.md[`ref-workflow-architecture/standards/artifacts.md`]; this section calls out the major artefacts and their telemetry role.

[cols="1,1,3", options="header"]
|===
| File | Producer | Purpose

| `request.md` | phase-1-init | The captured task / issue description. Plan's reason for existing.
| `solution_outline.md` | phase-3-outline | The deliverable breakdown. Each deliverable names a module, profile, and applicable skills.
| `tasks/TASK-N.json` | phase-4-plan | Materialized task records — `module`, `profile`, `skills[]` (per xref:skill-handling.adoc[Skill Handling]), `depends_on`, status.
| `references.json` | phase-1-init … phase-6-finalize | Branch, base branch, `modified_files[]`, lesson references, related plan ids.
| `status.json` | every phase | Phase lifecycle state plus `metadata.phase_handshake` invariants captured at each boundary.
| `execution.toon` | phase-4-plan | The per-plan finalize-step manifest — which verify steps run in phase-5, which finalize steps run in phase-6.
| `handshakes.toon` | every phase | Each phase boundary's captured invariants (e.g. `pending_findings_by_type`, `pending_findings_blocking_count`).
| `metrics.md` | phase-1 → phase-6 | Per-phase duration, token usage, tool-use counts; rolled up by `manage-metrics`.
| `architecture-pre/` | phase-1-init | Snapshot of `.plan/project-architecture/` at plan start. Lets retrospective detect mid-plan architecture drift.
| `quality-verification-report.md` | plan-retrospective | The retrospective output (when run). Optional, opt-in.
|===

All these files are read on retrospective via `manage-*` scripts — never by ad-hoc filesystem access. The `manage-*` interface is the audit layer's API.

=== Findings

Append-only JSONL files under `{plan_id}/artifacts/findings/`. One file per producer × phase.

Observed in archived plans:

[source]
----
artifacts/findings/qgate-3-outline.jsonl
artifacts/findings/qgate-4-plan.jsonl
artifacts/findings/assessments.jsonl
artifacts/findings/pr-comment.jsonl
----

Each line is a complete finding record:

[source,json]
----
{
  "hash_id": "882466",
  "timestamp": "2026-05-15T10:23:22Z",
  "phase": "3-outline",
  "source": "qgate",
  "type": "anti-pattern",
  "title": "Self-modifying + breaking plan requires phasing rationale or split",
  "detail": "…",
  "resolution": "taken_into_account",
  "resolution_detail": "…",
  "resolution_timestamp": "2026-05-15T10:25:45Z",
  "severity": "warning"
}
----

The `hash_id` is content-derived: re-running a Q-Gate that produces the same finding text won't duplicate the entry. Resolution fields (`fixed` / `suppressed` / `accepted` / `taken_into_account` / `pending`) record the disposition; the `pending` set feeds the phase-handshake `pending_findings_blocking_count` invariant that gates phase-5 → phase-6 transitions.

The findings layer is the structural counterpart to the prose `decision.log`: decisions explain *why*; findings record *what was found and what was done about it*.

== Why it's comprehensive

Two structural reasons the audit trail is exhaustive rather than best-effort:

1. **The script-executor proxy is the audit point.** Every `python3 .plan/execute-script.py` call routes through one Python entry point, which logs the notation, args, and exit status to `script-execution.log` before dispatching to the underlying script. Nothing operational slips past. See xref:security.adoc[Security] for the executor-proxy contract.
2. **Phase handshakes make state checkable.** Every phase boundary captures a registry of *invariants* into `status.json.metadata.phase_handshake` and verifies them on entry to the next phase. Drift between captured and live state surfaces immediately as a `status: drift` return — visible in `work.log`, persisted in `handshakes.toon`.

The two together mean any run can be replayed from the logs and re-evaluated from the artifacts — no information needs to be re-derived from chat history or memory.

== Evaluation — `plan-retrospective`

`plan-retrospective` is the opt-in skill that consumes the audit trail and produces a `quality-verification-report.md`. It is the canonical answer to "did this plan run well?"

The design has two layers:

* **12 aspect scripts** produce deterministic TOON *fragments* under `{plan_id}/work/` — pure facts derived from the audit trail. Examples seen in archived plans:
+
[source]
----
work/fragment-artifact-consistency.toon
work/fragment-dispatch-audit.toon
work/fragment-log-analysis.toon
work/fragment-permission-prompt-analysis.toon
work/fragment-request-result-alignment.toon
work/fragment-script-failure-analysis.toon
work/fragment-plan-efficiency.toon
work/fragment-llm-to-script-opportunities.toon
work/fragment-logging-gap-analysis.toon
work/fragment-invariant-outcomes.toon
work/fragment-direct-gh-glab.toon
work/fragment-lessons-proposal.toon
work/fragment-manifest-decisions.toon
----
* **The LLM synthesizes** the report from those fragments — judgement on top of facts. Aspect references (`references/artifact-consistency.md`, `references/log-analysis.md`, …) define what each aspect means and how to read its fragment.

The intent is captured in the skill's own design statement: "Python scripts produce deterministic TOON fragments (facts). The orchestrator loads aspect reference docs on-demand so the LLM can synthesize judgement from those facts. Scripts never judge; references never run code."

Invocation paths:

* During a live finalize phase: dispatched automatically as the `post-run-review` finalize step (when configured in `execution.toon`).
* As a standalone audit: `Skill: plan-marshall:plan-retrospective`, or `/plan-marshall action=lessons` to surface proposed lessons.
* Against an archived plan: the same skill invoked in "archived mode" — writes the report next to the archived plan without mutating any plan state.

Outputs:

* `quality-verification-report.md` — the synthesized retrospective (with a `-audit-{timestamp}` suffix when run in archived mode).
* Proposed lesson drafts under `work/lesson-body-*.md` — the LLM's candidate lessons-learned, ready for review and promotion via `manage-lessons`.

== Retention and privacy

The audit trail is **strictly local**.

* **Storage** — `.plan/local/{plans,archived-plans}/{plan_id}/`. Gitignored by `.plan/*` (see xref:../user/getting-started.adoc#_where_things_live[Where things live]). Never pushed, never shared with the team automatically.
* **Active vs. archived** — completed plans move from `plans/` to `archived-plans/` via `/plan-marshall action=cleanup`. Archived plans are read-only from the retrospective's perspective.
* **Sharing** — if a retrospective is worth sharing, copy `quality-verification-report.md` out manually. Nothing under `.plan/local/` is ever surfaced to remotes by Plan Marshall itself.

This is by design: the audit trail records local LLM behaviour, model usage, decisions, and per-developer working state — content that should not propagate to a shared repository.
