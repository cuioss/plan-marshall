This document is the single sanctioned place to record where plan-marshall's design
was informed by prior art. The planning and reference documents elsewhere in the
tree describe plan-marshall on its own terms and deliberately carry no external
references; when an idea was prompted by studying another tool, the attribution
lives here rather than scattered through those documents.

== Why attribution is centralised

Naming an external tool inside a skill, script, or planning document couples that
document to something outside the project and dates quickly. The project's
documents therefore state *what plan-marshall does and why*, in plan-marshall's own
vocabulary. The intellectual debt is real, though, and worth recording once — this
file is that record.

== Workflow-as-process tools

Several capability ideas were sharpened by reviewing external AI-development
workflow toolkits (for example, gstack — a collection of opinionated Claude Code
slash-command skills covering the think → plan → build → review → test → ship →
reflect lifecycle). The useful observation common to that class of tool is that the
differentiator is *encoded process*, not the model: standalone, composable
capabilities a developer can reach for individually.

plan-marshall already realises the deeper machinery that class of tool lacks —
deterministic state, worktree isolation, a structured findings pipeline,
multi-reviewer review, and the architecture-hints store. Where the comparison was
productive was in identifying *cheaper paths onto* that machinery and *more signal
fed back into* it. As those ideas land they fold into the permanent concept docs
(xref:recipes.adoc[Recipes], xref:planning-workflow.adoc[Planning Workflow],
xref:security.adoc[Security], xref:automatic-reviews.adoc[Automatic Reviews]). The
plan-marshall realisations differ in substance: audits emit into the closed
findings taxonomy rather than printing prose reports; shortcuts run inline yet
still create a plan-directory; verification stays encoded (committed e2e tests, not
a live browser daemon), and interactive exploration is left to the user's own
tools.

== Security-methodology skillset

The cross-cutting security knowledge in `persona-security-expert` (the
authoritative index of standards lives in
xref:../../marketplace/bundles/plan-marshall/skills/persona-security-expert/SKILL.md[the persona-security-expert SKILL.md])
was informed by several published security frameworks and code-analysis skillsets.
The debt is conceptual and structural — recognised taxonomies, a threat-modeling
method, a cross-mapping convention, and a knowledge-skill organisation pattern —
never copied prose. Every standards document was authored fresh in plan-marshall's
own vocabulary; the sources below shaped *what categories exist* and *how the
documents are structured*, not the wording.

Nature of the influence, by area:

* *Risk taxonomy.* The OWASP Top 10 (2025 numbering) supplies the recognised
  category baseline that every standards document maps findings to (the `Maps to:`
  lines). This is a *learned-from* influence: the category names and IDs are factual
  references, and the explanatory prose under each is original.

* *Threat-modeling method.* STRIDE supplies the per-threat decomposition method
  behind `threat-modeling-stride.md`. *Learned-from* — the method is applied, the
  worked examples are original.

* *Supply-chain frameworks.* SLSA (build-provenance levels), the OWASP SCVS control
  families, and the NIST SSDF / SP 800-57/800-52 guidance shaped the structure of
  `dependency-supply-chain.md` and `cryptography-key-management.md`. *Learned-from* —
  the level definitions and control-family names are factual references; the
  mitigations are paraphrased and re-expressed.

* *Cross-mapping convention.* The inline `Maps to: CWE-NNN · OWASP AXX · ASVS Vn`
  convention adopts the *structure* of the Semgrep rule-metadata cross-map and the
  CodeQL `external/cwe/cwe-NNN` tag pattern, plus the agentskills.io frontmatter
  `metadata` map. *Structure-adopted* — the convention's shape is reused; no rule
  text is copied.

* *Knowledge-skill organisation.* The cheatsheet-first progressive-disclosure shape
  (a concise index up front, full per-topic depth loaded on demand) and the
  definition → attack-scenario → mitigation → detection unit structure echo the
  Trail of Bits `skills-curated` / `scv-scan` template and the Anthropic
  claude-code-security-review methodology. *Structure-adopted* — these are knowledge
  skills, not scanners, so the organisation is adopted, not the scanner pipeline.

== Security-source licenses and attribution

The sources below were *learned from and attributed*, never copied verbatim. Several
are share-alike (CC-BY-SA) — their *prose* must not be copied into this
differently-licensed project, so all authored content is paraphrased and original.
One source, the Semgrep Rules License v1.0, is the single non-OSI,
commercial-use-restricted entry; only its rule-metadata *structure* was learned
from, and no rule text was reused.
