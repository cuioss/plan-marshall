= User Configuration — Per-Role Models
:toc:
:toclevels: 3

How project authors tune which model runs for each LLM-judgement workflow in plan-marshall.

== The 6-Group Phase-Scoped Registry

The role registry is phase-scoped, six groups (`phase-1-init` through `phase-6-finalize`). Every group is polymorphic — its value may be a plain string (single-level shorthand for every workflow under that phase) or an object whose sub-keys map to specific workflows under that phase. There are zero mandatory keys: a minimal config is `{}` and every dispatch resolves via `effort` → `inherit`.

Sub-key whitelist per phase:

[cols="1,3", options="header"]
|===
| Group | Recognised sub-keys

| `phase-1-init` … `phase-4-plan` | `default`, `research`
| `phase-5-execute` | `default`, `verification-feedback`, `research`
| `phase-6-finalize` | `default`, `verification-feedback`, `post-run-review`, `research`
|===

The canonical map (which workflow doc each sub-key drives) lives in link:../marketplace/bundles/plan-marshall/skills/plan-marshall/standards/effort-roles.md[`effort-roles.md`]. Retired pre-rewrite keys (`cross.*` and the five retired `phase-6-finalize.{create-pr,pre-submission-self-review,lessons-capture,retrospective,pr-doctor}`) return `status: error` with a remediation message naming the new target.

== Polymorphic JSON Shape

The `.plan/marshal.json` per-phase `effort` attributes accept polymorphic values. A string applies that level to every workflow under the phase; an object sets per-sub-key overrides plus an optional `default` slot that serves as the in-phase fallback. The plan-wide fallback is `plan.effort` (a single string):

[source,json]
----
{
  "plan": {
    "effort": "medium",
    "phase-2-refine": { "effort": "high" },
    "phase-3-outline": {
      "effort": { "default": "high" }
    },
    "phase-5-execute": {
      "effort": { "verification-feedback": "high" }
    },
    "phase-6-finalize": {
      "effort": {
        "default": "medium",
        "verification-feedback": "high",
        "post-run-review": "xxhigh"
      }
    }
  }
}
----

When a phase's `effort` is a plain string (e.g. `"phase-2-refine": { "effort": "high" }`), every sub-key under it resolves to that value. When `effort` is an object, sub-key lookups walk to the explicit override first, then to the `default` slot, then to `plan.effort`.

Resolution order, deepest first:

1. The explicit sub-key (e.g. `plan.phase-6-finalize.effort.verification-feedback`)
2. The phase's `default` slot (e.g. `plan.phase-6-finalize.effort.default`)
3. The phase's plain-string `effort` value (e.g. `"phase-2-refine": { "effort": "high" }`)
4. `plan.effort` — plan-wide default
5. `inherit` — implicit fallback (canonical no-suffix variant; inherits parent's model)

== Four Accepted Lookup Forms

The resolver accepts four input forms, mapping to the same resolution:

[source,bash]
----
manage-config effort resolve-target --phase phase-2-refine                              # bare group
manage-config effort resolve-target --role phase-6-finalize.verification-feedback         # dotted
manage-config effort resolve-target --phase phase-6-finalize --role verification-feedback # two-flag
manage-config effort resolve-target --default                                    # zero-role fallback
----

== Preset Shortcuts

Three named presets cover most usage. Apply via `/marshall-steward` Effort submenu (preset-picker) or directly:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  effort apply-preset --preset {economic|balanced|high-end}
----

Payloads live in link:../marketplace/bundles/plan-marshall/skills/plan-marshall/scripts/effort_presets.py[`effort_presets.py`] (`EffortPresets.ECONOMIC`, `BALANCED`, `HIGH_END`). The wizard delegates to `apply-preset`, which writes the per-phase `effort` attributes and the plan-wide `plan.effort` fallback.

== No-Restart Semantics

The resolver reads `.plan/marshal.json` fresh on every dispatch. An `effort` edit (manual or via the wizard) takes effect on the **next** dispatch — no Claude Code restart, no `/sync-plugin-cache`, no target regeneration required. The seven emitted `execution-context` variants (canonical + six levels) are static; only which variant the resolver selects changes.

A restart IS required when target/claude/ is regenerated (e.g., variant frontmatter changes), but that is a developer-side meta-project flow, not a user-side configuration flow.

== Worked Example

A project author wants the research workflow at the absolute top tier in phase-3-outline, Q-Gate checks on phase-3-outline/phase-4-plan at Sonnet-high (their phase defaults), the verification-feedback envelopes on Sonnet-high in both phase-5-execute and phase-6-finalize, and everything else on the default. Their `.plan/marshal.json`:

[source,json]
----
{
  "plan": {
    "effort": "medium",
    "phase-3-outline": {
      "effort": { "default": "high" }
    },
    "phase-4-plan": { "effort": "high" },
    "phase-5-execute": {
      "effort": { "verification-feedback": "high" }
    },
    "phase-6-finalize": {
      "effort": {
        "default": "medium",
        "verification-feedback": "high"
      }
    }
  }
}
----

After saving:

* Q-Gate dispatches from phase-3-outline (`--phase phase-3-outline`, no `--role`) hit `execution-context-high` via the phase-3-outline `default` slot. Phase-4 Q-Gate dispatches hit `execution-context-high` via the bare-string at `phase-4-plan.effort`.
* Every `--phase phase-5-execute --role verification-feedback` dispatch (Step 11 / Step 11b triage) hits `execution-context-high`.
* Every `--phase phase-6-finalize --role verification-feedback` dispatch (sonar / pr-comment / plugin-doctor / pr-state) hits `execution-context-high`.
* Every other phase-6-finalize dispatch (create-pr, pre-submission-self-review, post-run-review, enrich-module) hits `execution-context-medium` via the phase-6-finalize `default` slot.
* Every other dispatch (phase-1-init, phase-2-refine, phase-5-execute body) hits `execution-context-medium` via the `plan.effort` plan-wide fallback.

== Migration note — `xhigh` / `xxhigh` rebind

Plan-marshall is pre-1.0. The recent palette expansion (insert `xhigh = opus-medium`, promote `max` to live) rebinds the existing `xhigh` and `xxhigh` keywords to weaker primitives than they previously resolved to. Consumer `marshal.json` files that already set per-role levels are unaffected at the resolver level (the keywords still resolve), but the *primitive* the variant pins is now weaker. User-side action: any role that previously needed *Opus, high* under `xhigh` should be re-pointed at `xxhigh`; any role that previously needed *Opus, xhigh* under `xxhigh` should be re-pointed at `max`. There is no auto-migration.

| Level | Previous binding | New binding |
| `xhigh` | `opus, high` | `opus, medium` |
| `xxhigh` | `opus, xhigh` | `opus, high` |
| `max` | (reserved — refused) | `opus, xhigh` (Opus-4.7-only) |
