= Architecture
:toc: left
:toclevels: 2

Plan Marshall is structured as a Claude Code marketplace: ten production bundles, each contributing skills, agents, and (occasionally) commands to a shared three-tier component model.

== Three-Tier Component Model

[cols="1,3", options="header"]
|===
|Tier |Purpose

|Skills
|Domain knowledge, standards, reference documentation, and the workflow/standards/scripts that implement a capability. Loaded into context on demand via the `Skill:` directive or invoked through a script notation.

|Agents
|Autonomous task executors. Plan Marshall ships exactly one role-eligible canonical agent — `plan-marshall:execution-context` — which is emitted in six effort-level variants by the Claude target build. Every plan-marshall `Task:` dispatch routes through one of those variants.

|Commands
|User-invocable slash commands that orchestrate workflows (e.g. `/marshall-steward`, `/plan-marshall`).
|===

The dispatcher pattern that lets one canonical agent stand in for many role-pinned variants is described in xref:execution-context.adoc[Execution Context].

== Repository Layout

[source]
----
plan-marshall/
├── marketplace/
│   ├── .claude-plugin/marketplace.json   # master marketplace configuration
│   ├── adapters/                         # multi-assistant export adapters
│   └── bundles/                          # 10 production bundles (source of truth)
├── target/                               # build output, per target
│   └── claude/                           # populated by marketplace/targets/generate.py
├── test/                                 # pytest tests for marketplace scripts
├── doc/                                  # the documentation tree you are reading
├── .plan/                                # partially tracked — marshal.json + project-architecture/ committed, .plan/local/ + execute-script.py + .plan/temp/ ignored
└── .claude/                              # project-level Claude Code configuration
----

Each bundle under `marketplace/bundles/` follows the same layout: a `.claude-plugin/plugin.json` manifest, optional `agents/` and `commands/`, and a `skills/` tree where each skill is a `SKILL.md` accompanied by `standards/`, `scripts/`, `references/`, `templates/`, or `workflow/` as needed.

== Per-Plan Runtime Layout

When `/marshall-steward` first runs, it materialises `.plan/` inside the project:

[source]
----
.plan/
├── execute-script.py             # generated executor (do not edit by hand)
├── marshal.json                  # project configuration (effort, build systems, domains)
├── run-configuration.json        # runtime settings
├── project-architecture/         # module analysis
└── plans/{plan-id}/              # one directory per active plan
----

All access to `.plan/` goes through `python3 .plan/execute-script.py {notation}` — never via direct file I/O.
