= Execution-Context Overview
:toc:

Why plan-marshall ships exactly one role-eligible agent (`plan-marshall:execution-context`), and how every plan-marshall `Task:` invocation dispatches through it.

== One Generic Dispatcher

`plan-marshall:execution-context` is the sole role-eligible canonical agent in the marketplace. It owns the dispatcher boilerplate (load `dev-general-practices`, load caller-specified skills, locate workflow doc, execute, emit return TOON) exactly once and parameterises the workflow body at dispatch time. The caller passes five prompt-body fields:

* `name` — human label for logging and `mark-step-done`
* `plan_id` — plan identifier (or sentinel `none` for free-standing dispatches)
* `skills[]` — caller-loaded skills, in load order
* `workflow` — bundle-prefixed notation for the workflow doc to run (e.g. `plan-marshall:phase-1-init/SKILL.md`, `plan-marshall:plan-marshall/workflow/triage.md`)
* `WORKTREE` — repo-relative working directory (`.` for main checkout)

The dispatcher loads `dev-general-practices` implicitly, then the caller-specified `skills[]`, then reads the resolved workflow path and follows it. Workflow-specific runtime inputs (`finding_type`, `track`, `scope`, etc.) flow through additional prompt-body fields the workflow doc declares.

A new LLM-judgement workflow worth pinning its own model is added by writing a new workflow doc that declares `implements: ext-point-execution-context-workflow` and registering a new role key in the registry. No new agent file, no new emitted variants.

== Level Variants

The Claude target emits six variants per ordinal level — `execution-context.md` (canonical, inherit), `execution-context-low.md`, `-medium.md`, `-high.md`, `-xhigh.md`, `-xxhigh.md`. Each variant has `model:` and `effort:` baked into its frontmatter. Dispatch sites resolve which variant to call via `manage-config effort resolve-target --role <key>`, which returns the canonical name when level is `inherit`/empty and the matching variant otherwise. The resolver reads `.plan/marshal.json` fresh per dispatch, so an edit to any `plan.<phase>.effort` attribute takes effect on the next call — no Claude Code restart required.
