After installing Plan Marshall, run the setup wizard to wire it into your project:

[source,bash]
----
/marshall-steward
----

== First-run wizard

When the executor (`​.plan/execute-script.py`) is missing, `/marshall-steward` runs in wizard mode and walks you through the full 17-step setup. The material prompts you will see:

1. **Configure `.gitignore`** — adds a `.plan/*` block plus targeted re-includes (`!.plan/marshal.json`, `!.plan/project-architecture/`) so the shared configuration is tracked while your private working area under `.plan/local/` and the generated executor stay local. See <<_where_things_live>> for the full layout.
2. **Update project documentation** — appends idempotent guidance blocks (Temporary Files, Tool Usage, Build Commands) to `CLAUDE.md` and `agents.md`. See xref:#_what_the_wizard_writes_to_claude_md[What the wizard writes to `CLAUDE.md`] below.
3. **Ensure executor permission** — verifies (and adds if missing) the `Bash(python3 .plan/execute-script.py *)` allow entry in `.claude/settings.json`.
4. **Generate the script executor** — emits `.plan/execute-script.py` with the script mappings embedded for every installed bundle.
5. **Discover project modules** — walks the codebase via the extension API to identify build modules.
6. **Initialise `marshal.json`** — seeds the per-project configuration file with sensible defaults.
7. **Configure skill domains** — chooses which bundles' skill domains apply to which modules.
8. **Set review gates** — interactive prompts for the four review-gate booleans (`init_without_asking`, `plan_without_asking`, `execute_without_asking`, `finalize_without_asking` + the asymmetric `loop_back_without_asking`). See xref:configuration.adoc#review-gates[Configuration › Review gates].
9. **Detect CI provider and tools** — identifies GitHub vs. GitLab, Sonar/SonarCloud, and any other integrations.
10. **Configure architecture-refresh tier knobs** — sets `tier_0` (deterministic discover) and `tier_1` (LLM re-enrichment) policies in `run-configuration.json`.
11. **Credential setup** — interactively prompts for credentials for each registered provider (GitHub, GitLab, Sonar, Git) and stores them under `~/.plan-marshall-credentials/`. The credentials never enter the LLM's context.
12. **Permission setup** — applies any marketplace-recommended permission updates from `tools-permission-fix`.

Plus several smaller steps (configuration validation, terminal-title opt-in, etc.). The wizard is idempotent — re-running it on a fresh checkout produces the same setup.

== Interactive menu (returning users)

Once the executor exists, `/marshall-steward` opens an interactive menu instead:

* **Maintenance** — regenerate the executor (after bundle changes), clean execution logs.
* **Health Check** — verify the setup, diagnose missing scripts or stale configuration.
* **Configuration** — adjust build systems, skill domains, the per-role effort/model registry, terminal-title integration, and other project-level knobs.

Use `--wizard` to force the first-run wizard explicitly:

[source,bash]
----
/marshall-steward --wizard
----

== Creating your first plan

Once setup is complete, the simplest way to start a plan is to invoke `/plan-marshall` with no arguments and pick "Create new plan" from the interactive menu:

[source,bash]
----
/plan-marshall
----

If you prefer a one-liner, pass a `task=` description directly — the `init` action is inferred from the presence of `task=`:

[source,bash]
----
/plan-marshall task="Add user authentication"
----

Or seed the plan from a GitHub issue:

[source,bash]
----
/plan-marshall issue="https://github.com/cuioss/plan-marshall/issues/42"
----

`/plan-marshall` drops you into the six-phase planning flow (init → refine → outline → plan → execute → finalize) described in xref:../concepts/planning-workflow.adoc[Planning Workflow]. Resume an existing plan with `/plan-marshall plan="<plan-id>"` — it auto-detects the current phase. See xref:commands.adoc[Commands] for the full parameter reference and the action-resolution rules.

== Where things live

After setup, expect the following inside your project:

[source]
----
.plan/
├── marshal.json                # ✅ committed — project configuration
├── project-architecture/       # ✅ committed — per-module enriched.json + _project.json
│   ├── _project.json
│   └── {module}/
│       └── enriched.json       # ✅ committed
├── execute-script.py           # ❌ ignored — generated by /marshall-steward, do not edit
├── local/                      # ❌ ignored — your private working area
│   ├── plans/{plan-id}/        #    active plans live here
│   ├── worktrees/{plan-id}/    #    per-plan git worktrees
│   ├── lessons-learned/        #    captured lessons before promotion
│   ├── logs/                   #    execution logs
│   ├── archived-plans/         #    completed plans pruned from active list
│   └── …
└── temp/                       # ❌ ignored — scratch space, safe to delete
----

`.plan/` is **partially** tracked. Two paths are committed and reviewed alongside code:

* **`.plan/marshal.json`** — the per-project configuration that drives `/plan-marshall`, build resolution, the per-role effort registry, and skill domains. Changes belong in a normal commit.
* **`.plan/project-architecture/`** — the LLM-enriched module-analysis artifacts. The `enriched.json` files are committed so a fresh checkout has the architecture map without re-running the enrichment.

Everything else under `.plan/` is gitignored by `.plan/*` in the repository's `.gitignore` and is yours alone — never push it, never expect a teammate's copy to match. Active plans, worktrees, lessons in progress, logs, and temporary outputs all live under `.plan/local/` or `.plan/temp/`.

[NOTE]
====
The current `manage-architecture` code (see `_architecture_core.py`) computes per-module *derived* data (paths, packages, dependencies, file inventories) **on demand by crawling the live worktree on every read**. There is no `derived.json` to ignore or to track — only `enriched.json` (committed) and the on-the-fly crawl result (never written to disk).
====

For the canonical rules, see the project's `.gitignore` — the `.plan/*` block plus the two `!.plan/...` re-includes is the source of truth.

== What the wizard writes to `CLAUDE.md`

In addition to scaffolding `.plan/`, the first-run wizard *appends* a small set of guidance blocks to your project's `CLAUDE.md` (and, where applicable, `agents.md`) so the agents running in your project have the conventions they need. All edits are **idempotent and additive** — the wizard detects a marker substring per block and skips the edit if it's already present, so re-running the wizard never duplicates content. The only destructive path is explicit user confirmation under the Build Commands conflict prompt described below.

[cols="1,1,2,2", options="header"]
|===
| Block | Target file(s) | Skip condition | What it appends

| Temporary Files
| `CLAUDE.md`, `agents.md`
| File already contains the substring `.plan/temp`
| A `## Temporary Files` section pointing all temporary and generated files at `.plan/temp/` (covered by the `Write(.plan/**)` permission).

| Tool Usage
| `CLAUDE.md`
| File already contains the substring `use Glob, Read, Grep`
| A `## Tool Usage` section reminding agents to prefer `Edit`/`Read`/`Write` over shell `echo`/`cat`, and `Glob`/`Read`/`Grep` over `find`/`grep`/`cat`/`ls`.

| Build Commands
| `CLAUDE.md`
| File already contains a `### Build Commands` heading
| A `### Build Commands` block (under a "Development Notes" section) with the resolved canonical commands for this project — `Compile`, `Quality gate`, `Tests`, `Full verify`, plus `Integration tests` / `E2E` / `Coverage` / `Benchmark` where they resolve on the default module, plus a bullet per child-module-only command. Includes a "Never hard-code" preamble, a reminder to call builds with a 10-minute Bash timeout (`600000` ms), and a reminder to analyze the TOON return (`status`, `errors[]`, `log_file`).
|===

[NOTE]
====
If `CLAUDE.md` already contains hand-written build patterns (`mvn`, `mvnw`, `gradle`, `npm run`, `./pw`, or the phrase "build command") *and* no `### Build Commands` heading, the wizard will pause and ask whether to **Replace existing** or **Keep existing** before writing the Build Commands block. Replace overwrites the hand-written section with the resolved canonical commands; Keep leaves your `CLAUDE.md` untouched.
====

The canonical implementations live in `marshall-steward`'s scripts and references:

* link:../../marketplace/bundles/plan-marshall/skills/marshall-steward/scripts/determine_mode.py[`determine_mode.py`] — the `CONTENT_CHECKS` and `FIX_CONTENT` tables that drive Temporary Files and Tool Usage.
* link:../../marketplace/bundles/plan-marshall/skills/marshall-steward/references/wizard-flow.md[`wizard-flow.md`] § Step 9b — the Build Commands resolution and conflict handling.

The Maintenance menu surfaces these same checks on demand — run `/marshall-steward` → Health Check to see which markers are present and which are missing without writing anything, or → Maintenance → Update Project Documentation to apply any missing blocks.
