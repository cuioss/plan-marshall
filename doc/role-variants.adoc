= Role Variants and Per-Role Models
:toc:
:toclevels: 3

Overview and entry-point map for the per-role model variant-emission system. The canonical specifications live next to the code they govern; this document is a navigational summary that points there.

Authors configure `models.roles.<phase>[.<subkey>]` (hierarchical kebab-case keys like `phase-3.research`, `phase-6.verification-feedback`) in `.plan/marshal.json`. The Claude target emits six `execution-context[-{level}].md` agent variants for the sole role-eligible agent. Dispatch sites resolve the target via `manage-config models resolve-target --phase <phase> [--role <subkey>]` and call the matching variant by name. End users get the right model per role at dispatch time without any user-side build step or Claude Code restart.

== Quick reference

* **Configure a role** — set `plan.models.roles.<phase>[.<subkey>]` in `.plan/marshal.json` to one of `low | medium | high | xhigh | xxhigh | max | inherit` (a string value at the phase applies that level to every workflow under it; an object value sets per-sub-key overrides plus an optional `default` slot). See `role-variants.md` for the bubbling resolution rules.
* **Resolve a role at dispatch** — `python3 .plan/execute-script.py plan-marshall:manage-config:manage-config models resolve-target --phase <phase> [--role <subkey>]` returns the canonical or variant name directly (e.g. `execution-context` for `inherit`, `execution-context-high` otherwise). See `model-roles.md`.
* **Add a new workflow doc** — place it under `{bundle}:{skill}/workflow/{file}.md` (single-phase consumer) or `plan-marshall/skills/plan-marshall/workflow/{file}.md` (multi-phase / cross-cutting), declare `implements: plan-marshall:extension-api/standards/ext-point-execution-context-workflow` in frontmatter, and document the Output contract (`status` + `display_detail`).
* **Inspect emitted variants** — generate the Claude target output (`python3 marketplace/targets/generate.py --target claude --output target/claude`) and look under `target/claude/plan-marshall/agents/execution-context[-{level}].md`. The canonical no-suffix file is the `inherit` target.

== Rationale for keeping the canonical specs in `marketplace/bundles/`

Each standard is co-located with the skill or extension that owns it so it stays under the same review lifecycle as the code it governs. This `doc/` page is intentionally a thin summary — duplicating the substance here would create two places to update on every change and the `marketplace/bundles/` copy is the one that downstream consumers (plugin-doctor, the build target, dispatch sites) read.
