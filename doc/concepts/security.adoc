= Security
:nofooter:
:toc: left
:toclevels: 2

How Plan Marshall reduces the prompt-vs-blast-radius trade-off that Claude Code's stock permission modes force on developers — and a frank statement of what it does **not** solve.

== TL;DR

Claude Code ships two extremes:

* **Accept-edits / interactive mode** — every meaningful action prompts the developer. Safe, but flow-killing on any non-trivial task.
* **YOLO (`--dangerously-skip-permissions`)** — no prompts. Fast, but a single misstep by the LLM can do anything within the user's shell.

Plan Marshall sits between them with a different approach: **narrow what the LLM does at all**, so the permission surface is small enough to pre-approve with confidence. Two layers cooperate — a normative baseline (`dev-general-practices` hard rules) and a technical funnel (the script-executor proxy at `python3 .plan/execute-script.py`).

[CAUTION]
====
**Does this solve the problem completely? No.**

`dev-general-practices` is skill-loaded *guidance*, not kernel-level enforcement. The LLM can still ignore the rules; nothing in Claude Code's runtime forces a subagent to honour a skill's `Prohibited actions` list. Plan Marshall lowers the probability of unsafe behaviour and bounds the blast radius via narrow allow-lists — it does not eliminate the risk.

The model is **way fewer prompts than accept-edits, way more secure than YOLO**. For the remaining cases where Claude Code violates the rules anyway, you still get the prompt — just much less often than you would interactively, and the prompt itself is the safety net.
====

== Claude Code's stock modes — recap

For orientation:

[cols="1,3,1", options="header"]
|===
| Mode | Behaviour | Prompt frequency

| Interactive (default) | Every Edit / Write / Bash / WebFetch outside the `allow` list pops a permission prompt. | High
| Accept-edits | Skips prompts for Edit / Write inside the cwd, prompts for everything else. | Medium
| Auto | Heuristic mid-tier — narrower auto-accepts than YOLO, broader than accept-edits. | Low
| YOLO (`--dangerously-skip-permissions`) | Skips every prompt. | None — unbounded blast radius
|===

Plan Marshall operates inside interactive mode, but the prompt rate stays low because the permission lists pre-approve the patterns plan-marshall actually uses.

== Layer 1 — `dev-general-practices` as the normative baseline

Every plan-marshall subagent loads `plan-marshall:dev-general-practices` as Tier-1 *before* anything else (see xref:skill-handling.adoc[Skill Handling]). Its `Workflow Discipline (Hard Rules)` section in `standards/general-development-rules.md` narrows what the subagent is allowed to *try*:

* **No unconstrained generic subagents** — every phase-1 through phase-6 dispatch routes through `execution-context-{level}`. No raw `Task: general-purpose`.
* **`.plan/` access through scripts only** — never direct Read/Write/Edit on `.plan/` files (except via documented workflow exceptions).
* **One Bash command per call** — no `&&`, `;`, `&`, or newlines. Compound commands are refused at review time.
* **No shell constructs** — no `for`/`while` loops, `$()` substitution, subshells, or heredocs with `#` lines (which trigger security prompts).
* **Structured queries first** — `architecture files --module X` before `Glob`/`Grep` for module-scoped discovery.
* **Build commands via architecture-resolve** — never hard-coded `mvn`/`gradle`/`npm`/`./pw`; always resolved through `manage-architecture:architecture resolve`.
* **CI operations via abstraction** — `tools-integration-ci` instead of direct `gh`/`glab`.

These rules cut off the gnarliest LLM failure modes (shell injection through `$()`, runaway loops, ad-hoc subagents with full tool access) before they can fire. They are guidance, not enforcement — but loaded at every dispatch, they are present in the subagent's context at every decision.

== Layer 2 — the script-executor proxy as a technical funnel

Almost every action plan-marshall takes routes through one Bash pattern:

[source,bash]
----
python3 .plan/execute-script.py {bundle}:{skill}:{script} [args...]
----

The executor is a Python proxy generated by `/marshall-steward` that resolves the `{bundle}:{skill}:{script}` notation to an actual script under `marketplace/bundles/`, sets `PYTHONPATH`, and runs the script with the caller's args forwarded. The list of `{bundle}:{skill}:{script}` notations is embedded — only registered scripts resolve; unknown notations error out.

This shape means a *single broad Claude Code permission* covers the bulk of plan-marshall's work:

[source,json]
----
"Bash(python3 .plan/execute-script.py *)"
----

The `*` is bounded — it can only match arguments that the executor itself accepts, and the executor only dispatches to registered scripts. The LLM cannot smuggle arbitrary commands through this allow rule because the executor doesn't `exec` shell strings; it dispatches to Python.

**Net effect**: hundreds of distinct plan-marshall operations (every `manage-*`, every build, every triage, every finalize step) collapse into one allow entry. Without the executor proxy, the same coverage would require either a permission prompt per operation, or a dozens-of-entries-long allow list that's easy to get wrong.

== Permission settings — shape and split

Plan Marshall uses Claude Code's standard two-file split:

[cols="1,1,3", options="header"]
|===
| File | Tracked? | Role

| `.claude/settings.json` | ✅ committed | The project-shared allow list. Small (~10 entries) — just the executor proxy, narrow `Edit`/`Write` against `.plan/`, Skill access for finalize-step plugin synchronisation. Same for every developer on the team.
| `.claude/settings.local.json` | ❌ gitignored | Per-developer customisation. Larger (~50 entries) because individual developers approve patterns the team-shared file doesn't (WebFetch domains, local tool paths, ad-hoc Bash patterns).
|===

Sample allow entries from a real plan-marshall checkout:

[source,json]
----
"Bash(python3 .plan/execute-script.py *)",   // the executor proxy — covers ~90% of operations
"Bash(rm -rf .plan/logs/*)",                 // narrow cleanup, scoped to ignored paths
"Edit(.plan/**)",                            // local plan files
"Write(.plan/**)",                           // local plan files
"Read(~/.claude/plugins/cache/**)",          // installed plugin cache (read-only)
"WebFetch(domain:github.com)"                // narrow web access
----

The list is short by design — every entry maps to a known operation pattern plan-marshall produces. There are no `deny` or `ask` entries in plan-marshall's recommended settings; the *narrowness* of the allow list is the safety mechanism.

== When prompts still happen

Plan Marshall reduces prompt frequency; it does not eliminate prompts. Note that git itself does *not* prompt — commits and pushes flow through link:../../marketplace/bundles/plan-marshall/skills/workflow-integration-git/SKILL.md[`workflow-integration-git`]'s `git_workflow` script, which runs under the executor proxy, so the broad `Bash(python3 .plan/execute-script.py *)` allow covers them. Prompts you should still expect:

* **Bash file-ops (`find`, `cat`, `grep`, `ls`, `head`, `tail`)** — banned by hard rule, but if the LLM tries them anyway, the absence of a matching allow rule surfaces a prompt before damage.
* **Compound shell commands** — `&&`, `;`, `&`, newlines, `$()`, heredocs, `for`/`while` loops. All banned by hard rule, all unmatched by any allow pattern, so they prompt.
* **Direct `gh` / `glab` calls** — the CI abstraction (`tools-integration-ci`) is the documented route. Direct provider calls aren't pre-approved.
* **New WebFetch domains** — the WebFetch allow list is per-domain; novel domains prompt.
* **Ad-hoc Bash that doesn't match the executor proxy** — anything the LLM tries that bypasses the executor and doesn't match a narrow allow rule surfaces a prompt. The fallback when the LLM ignores a hard rule.

**Treat every prompt as a signal.** A prompt for something plan-marshall should never have tried means a hard-rule violation made it through the skill-load layer. The right response is usually to deny, redirect, and (if it recurs) capture as a lesson.

== Permission tooling

Two user-invocable skills manage the settings files themselves:

* link:../../marketplace/bundles/plan-marshall/skills/tools-permission-doctor/SKILL.md[`tools-permission-doctor`] — read-only analysis. Detects redundant patterns, security anti-patterns (overly broad allow rules, conflicting deny lists), validates syntax. Use *before* applying fixes.
* link:../../marketplace/bundles/plan-marshall/skills/tools-permission-fix/SKILL.md[`tools-permission-fix`] — write operations. Adds, removes, consolidates patterns; applies marketplace-recommended permission updates after installing a new bundle; runs the executor-migration sweep that converts legacy direct-script allows into the executor-proxy form.

Both expose `--dry-run` for verification before any settings file is rewritten.

== Audit trail

Because every action routes through the script-executor proxy, every script invocation is logged automatically. Combined with the structured `[STATUS]` / `[DISPATCH]` / `[SKILL]` / `[ARTIFACT]` log emitter inside `execution-context`, the per-plan findings JSONL, and the phase-handshake invariants, this produces a comprehensive, machine-readable record of every plan run. The full story — what's logged, where, how `plan-retrospective` evaluates it — is its own concept page: xref:audit-trail.adoc[Audit Trail].
