== Configuration surfaces — at a glance

[cols="1,1,3", options="header"]
|===
| Surface | Tracked? | What it controls

| `.plan/marshal.json`
| ✅ committed
| Per-project behaviour: per-phase fields, review gates, the effort/model registry, verify- and finalize-step lists, build-system mappings, skill domains. See <<marshal-json>>.

| `.plan/run-configuration.json`
| ❌ ignored
| Per-developer command-execution state: adaptive build-command timeouts, acceptable-warning patterns, the `architecture_refresh` tier knobs, and last-execution metadata. Never committed.

| `CLAUDE.md`, `agents.md`
| project decision (usually ✅)
| Agent instructions. The wizard appends three idempotent guidance blocks — see xref:getting-started.adoc#_what_the_wizard_writes_to_claude_md[Getting Started → What the wizard writes to `CLAUDE.md`].

| `.claude/settings.local.json`
| ❌ ignored (per-developer)
| Local Claude Code settings — hooks, `statusLine`, env vars. Plan Marshall writes the terminal-title hooks here; see xref:terminal-title.adoc[Terminal Title].

| `.gitignore`
| ✅ committed
| The `.plan/*` block plus `!.plan/marshal.json` and `!.plan/project-architecture/` re-includes (the `.plan/*` exclusion also covers `.plan/local/worktrees/{plan-id}/` — no separate carve-out needed). See xref:getting-started.adoc#_where_things_live[Getting Started → Where things live].

| VS Code user settings
| n/a
| One-time opt-in for terminal-title integration. See xref:terminal-title.adoc[Terminal Title] § VS Code.
|===

[#marshal-json]
== `.plan/marshal.json`

The bulk of project configuration. Canonical schema and field reference: link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`].

`marshal.json` is organised by phase under `plan.phase-1-init` … `plan.phase-6-finalize`, plus the top-level `providers`, `project`, `skill_domains`, and `system` sections. Each phase block holds fields specific to that phase, including the run-at-all gates and the three auto-continuation knobs, each homed under the phase whose decision it governs. The sections below cover the fields project authors tune regularly; for everything else (`branch_strategy`, `use_worktree`, `confidence_threshold`, `compatibility`, `simplicity`, `commit_and_push`, `per_deliverable_build`, `per_task_budget_reserve_tokens`, `max_iterations`, `review_bot_buffer_seconds`, `pr_merge_strategy`, the `plan.phase-6-finalize.*` auto-continuation knobs, `auto_rebase_threshold`, `drop_review_on_scope_gate`, `open_in_ide`, the `system.retention.*` knobs, `plan.phase-6-finalize.checks_wait_timeout_seconds`, the `project.*` branch-naming knobs, …) see the canonical per-key reference in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`].

[#review-gates]
=== Review gates

Boolean fields under each phase that decide whether the next-phase transition pauses for user review or auto-continues. Each gate lives under the phase whose **outbound** transition it governs.

[cols="1,1,3", options="header"]
|===
| Config key | Default | Governs the transition…

| `plan.phase-1-init.init_without_asking` | `true` | …from `phase-1-init` to `phase-2-refine`. Set to `false` to stop after init and review the captured request before refinement starts.
| `plan.phase-3-outline.plan_without_asking` | `false` | …from the outline review checkpoint to task creation.
| `plan.phase-4-plan.execute_without_asking` | `true` | …from `phase-4-plan` to `phase-5-execute`.
| `plan.phase-6-finalize.finalize_without_asking` | `true` | …from `phase-5-execute` to `phase-6-finalize` (forward direction).
| `plan.phase-6-finalize.loop_back_without_asking` | `false` | …from a `phase-6-finalize` `loop_back` outcome back into `phase-5-execute` (reverse direction). Defaults differ from `finalize_without_asking` to prevent silent unattended loops.
| `plan.phase-6-finalize.auto_merge_after_ci` | `true` | …from a green CI run to the `pr merge --delete-branch` step. `true` (default) merges automatically once CI passes, serialised across concurrent plans by the cross-plan merge-lock; `false` prompts the operator before merging.
|===

Every gate lives under the phase block whose outbound transition it governs; the three auto-continuation knobs (`finalize_without_asking`, `loop_back_without_asking`, `auto_merge_after_ci`) are flat fields under `plan.phase-6-finalize`. Set any gate via `/marshall-steward` → Configuration → Review Gates, or directly with the standard phase `set` verb:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-1-init set --field init_without_asking --value false

python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-6-finalize set --field auto_merge_after_ci --value false
----

[#run-at-all-gates]
=== Run-at-all gates

Separate from the auto-continuation knobs above (which decide *whether to pause* once a gate has run), the per-phase run-at-all gates decide *whether each gate runs at all*. Each gate is a tri-state enum — `auto` (default), `always`, or `never`:

* `auto` — defer to the existing decision machinery (the phase-1-init lane router for the planning gates, the manifest composer for the finalize gates). This is the historical behaviour.
* `always` — force the gate in, overriding the machinery that would otherwise skip it. For the finalize gates this overrides the manifest composer's `scope_gated_finalize` drop on surgical / single-module plans.
* `never` — skip the gate. For a finalize gate that guards a safety net this can mask real failures; `finalize.qgate` is the highest-risk (it can mask real build/test failures and push a red tree).

[cols="1,1,3", options="header"]
|===
| Config key | Default | What the gate controls

| `plan.phase-1-init.deep_lane` | `auto` | Whether the precondition-driven deep planning lane runs. `always` forces deep; `never` forces light.
| `plan.phase-1-init.escalation` | `auto` | Whether the hard-escalation safety ratchet stays live. `never` is the explicit full-speed-full-risk opt-in.
| `plan.phase-2-refine.revalidation` | `auto` | Whether the premise / narrative-vs-code safety check runs.
| `plan.phase-3-outline.qgate` | `auto` | Whether the planning-time Q-Gate validation runs.
| `plan.phase-6-finalize.self_review` | `auto` | Whether the pre-submission structural + cognitive self-review is selected into the execution manifest.
| `plan.phase-6-finalize.qgate` | `auto` | Whether the finalize blocking-findings re-capture runs. Highest-risk gate — `never` can push a red tree.
| `plan.phase-6-finalize.plugin_doctor` | `auto` | Whether the structural marketplace lint runs before push.
| `plan.phase-6-finalize.simplify` | `auto` | Whether the holistic post-implementation simplification sweep runs. `always` forces it in even when the composer's `simplify_inactive` pre-filter would drop it; `never` skips it.
|===

Read or set the run-at-all gates with the standard phase `get`/`set` verb against the owning phase block:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-6-finalize get --field self_review

python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-6-finalize set --field plugin_doctor --value always
----

See the canonical per-key reference in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`] for the full per-phase gate catalogue.

[#step-configuration]
=== Step configuration (verify + finalize pipelines)

`plan.phase-5-execute.steps` and `plan.phase-6-finalize.steps` are flat ordered lists of step identifiers that run sequentially during their phase. Each step ID resolves to one of three sources:

[cols="1,3", options="header"]
|===
| Step kind | Example

| **Built-in** (prefix `default:`)
| `default:quality_check`, `default:build_verify`, `default:coverage_check` (phase-5 — underscore form); `default:pre-push-quality-gate`, `default:commit-push`, `default:create-pr`, `default:ci-verify`, `default:automated-review`, `default:sonar-roundtrip`, `default:lessons-capture`, `default:branch-cleanup`, `default:record-metrics`, `default:archive-plan` (phase-6, kebab-case, canonical default order).

| **Project-local** (prefix `project:`)
| Loaded from `.claude/skills/finalize-step-<step>/` — a project-specific finalize step (e.g. a custom deploy or notification).

| **Extension-provided**
| Contributed by an installed bundle via `provides_verify_steps()` (phase-5) or `provides_finalize_steps()` (phase-6). The bundle's `extension.py` declares them; the wizard appends them in domain-defined order.
|===

Order is significant — phase-5 runs verify steps in list order until one fails; phase-6 runs finalize steps in list order until the pipeline completes or a `loop_back` outcome resets it. Configure via `/marshall-steward` → Configuration, or `manage-config plan phase-N set --field steps`.

Reference: `data-model.md` §§ phase-5-execute / phase-6-finalize; link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-verify-steps.md[`ext-point-verify-steps.md`]; link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-finalize-steps.md[`ext-point-finalize-steps.md`].

[#per-task-budget-reserve]
=== Per-task budget reserve

`plan.phase-5-execute.per_task_budget_reserve_tokens` (default `"50K"`) is the per-task budget **reserve** — the minimum context-window margin that must remain free before the `phase-5-execute` task loop is allowed to start another task. It governs the loop's continue-vs-yield sentinel. The `_tokens` suffix names the unit; the human-friendly value form (`"50K"`) is parsed to the integer `50000` by the consumer via `sensible_number.parse_sensible_int`.

The dispatch model is **budget-bounded**, neither per-task nor per-deliverable. One `execution-context` envelope runs as many tasks as the reserve permits — so it bundles several small deliverables into one envelope and may span a single large deliverable across several envelopes — then yields, and the orchestrator re-dispatches a fresh envelope to resume the loop. The reserve is the operator-tunable knob for that granularity: raise it to make each envelope yield earlier (smaller dispatches), lower it to pack more tasks into each envelope.

Set via:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-5-execute set --field per_task_budget_reserve_tokens --value 50K
----

For the conceptual home of per-dispatch context isolation see xref:../concepts/token-management.adoc[Concepts › Token Management]. Canonical schema: link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`].

[#commit-and-build-depth]
=== Commit strategy and per-deliverable build depth

Two `plan.phase-5-execute` knobs govern how often the execute phase commits and how deeply it builds at each deliverable boundary:

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `plan.phase-5-execute.commit_and_push` | `true` | When `true` (default), the execute loop commits per-deliverable and finalize pushes + opens a PR; when `false`, the run is local-only (the commit-push/push/PR finalize steps are stripped by the manifest `commit_push_disabled` pre-filter).
| `plan.phase-5-execute.per_deliverable_build` | `compile+scoped-test` | Build depth at each per-deliverable chain-tail. `off` skips the focused build (end-of-phase sweep only); `compile-only` type-checks the changed module; `compile+scoped-test` adds the module's scoped tests; `full` runs a whole-tree quality-gate per deliverable (legacy, opt-in).
|===

Set via `/marshall-steward` → Configuration, or directly:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-5-execute set --field per_deliverable_build --value compile-only
----

[#merge-strategy]
=== PR merge strategy and auto-rebase

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `plan.phase-6-finalize.pr_merge_strategy` | `squash` | The merge method the branch-cleanup step passes to `pr merge` — `squash`, `merge`, or `rebase`.
| `plan.phase-6-finalize.auto_rebase_threshold` | `no_overlap_only` | Gates the pre-merge auto-rebase decision. `no_overlap_only` lets the auto-rebase proceed only when it would touch a disjoint file set; any overlap defers to the operator.
| `plan.phase-6-finalize.drop_review_on_scope_gate` | `false` | Escape hatch for the manifest composer's scope-gating pre-filter. `false` keeps the bot-review invariant intact; `true` opts into additionally dropping `automated-review` on scope-gated plans.
|===

NOTE: The `default:pre-push-quality-gate` finalize step has no dedicated activation key. Its activation is *derived from* `build.map` (see <<build-map>>): the step activates whenever a changed path matches a glob registered in the build_map.

[#open-in-ide]
=== Open in IDE

`plan.open_in_ide` (default `true`) controls whether Plan Marshall attempts to open the plan/worktree in your IDE on creation. A missing key is treated as `true`. Set to `false` to suppress IDE auto-open.

[#retention-and-ci]
=== Retention and CI wait-policy

Top-level `system.retention` knobs plus the finalize CI-wait policy that project authors tune occasionally:

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `system.retention.logs_days` | `1` | Days to keep execution logs before cleanup.
| `system.retention.archived_plans_days` | `5` | Days to keep archived plans before cleanup.
| `system.retention.lessons_superseded_days` | `0` | Days to keep superseded lessons before removal (`0` = remove immediately).
| `system.retention.temp_on_maintenance` | `true` | Whether the maintenance pass clears `.plan/temp/`.
| `plan.phase-6-finalize.checks_wait_timeout_seconds` | `600` | Default ceiling (seconds) the CI-completion polling commands wait for run completion. Overridden by an explicit `--timeout` flag when supplied. A finalize wait-policy owned by phase-6-finalize.
|===

[#project-branch-naming]
=== Base branch and branch-naming

The top-level `project` section holds the project's base branch and the closed branch-prefix sets:

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `project.default_base_branch` | `main` | The project's canonical base branch. `phase-1-init` seeds `references.base_branch` from it; per-plan overrides via `manage-references set --field base_branch`.
| `project.working_prefixes` | `["feature/", "fix/", "chore/"]` | The closed set of allowed working-branch prefixes. `manage-status create` validates `--worktree-branch` against this set. A structural test asserts every prefix is covered by a `.github/workflows/python-verify.yml` push trigger, so a dropped prefix that would make a PR unmergeable fails CI.
|===

[#build-systems]
=== Build systems and profiles

The wizard discovers project modules and their build systems automatically (Maven / Gradle / npm / pyprojectx) via the extension API. The discovery output (which build commands a given module supports, the resolved canonical mapping, optional per-module profile filtering) is stored in `.plan/project-architecture/{module}/enriched.json`; derived data is computed on demand on every read and is never persisted.

For Maven projects, unmatched profiles are surfaced for review (Ignore / Skip / Map) during Step 10 of the wizard. Canonical commands (`Compile`, `Quality gate`, `Tests`, `Full verify`, …) are resolved per module and written into `CLAUDE.md` under the `### Build Commands` heading so agents call builds via canonical names rather than hard-coding the underlying tool.

Tune via `/marshall-steward` → Configuration → Build Systems. Canonical contracts: link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-build.md[`ext-point-build.md`] and link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/canonical-commands.md[`canonical-commands.md`].

[#skill-domains]
=== Skill domains

`marshal.json` carries a `skill_domains` block that maps each project module to the bundles whose skills apply to it. Domains are detected automatically by the wizard from each bundle's `plan-marshall-plugin` manifest, then assigned per module either by detection (e.g. Java files → `pm-dev-java`) or by user confirmation.

Canonical contracts: link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/skill-domains.md[`skill-domains.md`] and link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/skill-domains-operations.md[`skill-domains-operations.md`].

Tune via `/marshall-steward` → Configuration → Skill Domains.

[#build-map]
=== Build map

`marshal.json` carries a `build.map` block at the top level — the file-to-build contract. It maps every changed path to the build action that path requires, so when a plan runs only the relevant build fires: a production-code change compiles, a test change runs the tests, and a docs-only change runs no build at all (documentation has no build owner). The block is required and always seeded.

The `build.map` is seeded automatically. Each domain bundle declares which path globs it owns and what build action each glob needs; the wizard aggregates those declarations into `build.map`. Seeding is write-once — a re-seed never clobbers an existing block — so a correction you make survives.

Each entry classifies a glob into a closed build-class vocabulary:

* `compile` — production code; compiles the changed module.
* `module-tests` — test code; compiles and runs the changed module's tests.
* `verify` — build configuration; runs the full verify pipeline for the module.
* `none` — no build at all (the path is changed but needs no build action).

==== Correcting a wrong mapping

When the seeded mapping is wrong for your project, correct the entry directly in `build.map` — there is no separate override layer. Because seeding is write-once, your correction survives a re-seed.

The common case is a generated source file that the seed treats as production code but that should never trigger a build. Set its entry's `build_class` to `none`:

[source,json]
----
{
  "build": {
    "map": {
      "python": [
        { "glob": "marketplace/bundles/**/scripts/generated_*.py", "role": "production", "build_class": "none" }
      ]
    }
  }
}
----

The canonical schema (the field table, the build-class enum source of truth, the longest-glob-wins precedence) lives in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`] § Section: build.map.

[#worktrees]
=== Branch strategy and worktrees

How a plan decides whether to allocate a feature branch + isolated worktree, where the worktree lives on disk, and when materialization happens. Plan Marshall reads `plan.phase-1-init.branch_strategy` to drive the decision:

* `feature` (default for new plans) — the orchestrator allocates a feature branch named `feature/{plan-id}` and an isolated worktree at `<project_root>/.plan/local/worktrees/{plan-id}/`. `status.metadata.use_worktree` is set to `true`.
* `main` — the plan runs against the main checkout without a worktree (intended for docs-only patches that do not need isolation).

`phase-1-init` records only `use_worktree`. The feature branch (`feature/{plan-id}`) and the worktree directory are created together during `phase-5-execute` Step 2.5, immediately before the first task dispatch. Phases 2-4 (refine, outline, plan) run against the main checkout because they only produce `.plan/` artifacts and have no source-code changes to isolate.

The `.plan/*` exclusion in `.gitignore` covers `.plan/local/worktrees/{plan-id}/` automatically — no per-worktree carve-out is needed. The conceptual overview (why each plan runs in plan-bound isolation, and how the post-execute shipping flow lands the work on `main`) lives in xref:../concepts/branches-and-worktrees.adoc[Concepts › Branches and Worktrees]; the canonical contract (path resolution, dispatch protocol, cleanup ordering) lives in link:../../marketplace/bundles/plan-marshall/skills/workflow-integration-git/standards/worktree-handling.md[`workflow-integration-git/standards/worktree-handling.md`].

== Configuration outside `.plan/`

=== `CLAUDE.md` and `agents.md`

The first-run wizard appends three idempotent guidance blocks — Temporary Files, Tool Usage, Build Commands. All edits are marker-gated and never duplicate. Full breakdown in xref:getting-started.adoc#_what_the_wizard_writes_to_claude_md[Getting Started → What the wizard writes to `CLAUDE.md`].

=== `.claude/settings.local.json`

Per-developer Claude Code configuration — gitignored by convention. Plan Marshall writes the terminal-title hooks plus a `statusLine` entry here when you opt in via `/marshall-steward` → Configuration → Terminal title. See xref:terminal-title.adoc[Terminal Title] for the full hook chain and platform caveats.

=== `.gitignore`

The wizard adds a `.plan/*` block plus targeted re-includes for `marshal.json` and `project-architecture/`. See xref:getting-started.adoc#_where_things_live[Getting Started → Where things live] for the full layout.

=== VS Code user settings

VS Code's integrated terminal does not honor OSC title sequences by default — opt in by setting `terminal.integrated.tabs.title = "${sequence}"` in your VS Code user settings. The wizard cannot patch this file (it belongs to VS Code, not Claude Code). See xref:terminal-title.adoc[Terminal Title] § VS Code for the verification steps and troubleshooting.

== Editing configuration

Three paths, in increasing order of automation:

1. **Wizard (recommended)** — `/marshall-steward` opens an interactive menu covering every section above. Inputs are validated at save time and conflicts are surfaced before writing.
2. **`manage-config` CLI** — script-driven, useful for automation or for fields the wizard does not surface. Full command reference: link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/api-reference.md[`api-reference.md`].
3. **Direct file edit** — `marshal.json` is JSON; edit it in any editor. The resolver reads it fresh on every dispatch, so changes take effect on the next call without a Claude Code restart, no plugin reinstall.
