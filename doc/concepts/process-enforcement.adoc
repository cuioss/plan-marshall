= Process Enforcement
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:index.adoc[Concepts]

How Plan Marshall keeps the LLM on the documented process — four cooperating mechanisms, from skill-loaded guidance the model can still ignore, to a mechanical gate that exits non-zero on drift.

== Overview

LLMs are not reliable process followers. They sidestep documented steps, invent plausible-but-wrong CLI verbs, shortcut around checks they consider "obviously satisfied", and report success against state they never produced. Telling them not to does not stop them.

Plan Marshall responds with four cooperating mechanisms ordered from soft (skill-loaded text) to hard (mechanical refusal). Each layer assumes the layers above it can be bypassed; the stack is designed so a drift terminates as high up as possible, and so the deeper a violation tries to reach, the more mechanical the resistance becomes.

image::../resources/diagrams/enforcement-layers.svg[Process enforcement — four layers stacked from soft (skill-loaded text) to hard (mechanical gate). Layer 1 is the dev-general-practices hard rules. Layer 2 is the script-executor proxy plus manage-* funnel for all .plan/ access. Layer 3 is the Q-Gate findings checkpoint resolved at the next phase's entry. Layer 4 is the phase-handshake invariant registry that captures fingerprints at completion and verifies them at entry, refusing to advance on drift., align=center]

Each layer assumes the layer above it can be bypassed. The four together are small in surface area — one hard-rules file, one script proxy, one Q-Gate gate, one handshake registry — but load-bearing at every phase boundary.

== Layer 1 · Hard rules in `dev-general-practices`

Every plan-marshall subagent loads link:../../marketplace/bundles/plan-marshall/skills/dev-general-practices/SKILL.md[`plan-marshall:dev-general-practices`] as Tier-1 before anything else (see xref:skill-handling.adoc[Skill Handling]). Its `Workflow Discipline (Hard Rules)` section in link:../../marketplace/bundles/plan-marshall/skills/dev-general-practices/standards/general-development-rules.md[`standards/general-development-rules.md`] is the closest thing the marketplace has to "rules in the LLM's context window every time it dispatches." Representative rules:

* `.plan/` access goes through `python3 .plan/execute-script.py` — never direct Read/Write/Edit on `.plan/` files.
* One Bash command per call — no `&&`, `;`, `&`, or newlines.
* No shell constructs — no `for`/`while` loops, no `$(...)` substitution, no subshells.
* Build commands resolve via `manage-architecture:architecture resolve`, never hard-coded `mvn`/`gradle`/`npm`/`./pw`.
* CI operations go through `tools-integration-ci`, never direct `gh`/`glab`.
* Triage findings flow through `manage-findings` + `ext-triage-{domain}`.

**Strength**: skill-loaded *guidance*. The LLM might still violate the rules. Layer 1 reduces the probability; the layers below it bound the damage when it doesn't.

== Layer 2 · The script-executor proxy + manage-\* funnel

The mechanism: virtually every action Plan Marshall takes routes through one Bash pattern.

[source,bash]
----
python3 .plan/execute-script.py {bundle}:{skill}:{script} [args...]
----

This is generated by `/marshall-steward` and contains an embedded notation-to-script map — only registered scripts resolve; unknown notations error out. A single Claude Code `Bash(python3 .plan/execute-script.py *)` allow rule covers hundreds of operations, while everything *not* matching that pattern surfaces a prompt.

**Bounded artifacts** are the natural counterpart. The set of files Plan Marshall maintains under `.plan/` is fixed and per-file owned by a single `manage-*` skill — `manage-status`, `manage-tasks`, `manage-findings`, `manage-architecture`, `manage-references`, `manage-execution-manifest`, `manage-run-config`, etc. Each `manage-*` is the only legitimate writer for its file; readers and writers all go through it. The contract — TOON in, TOON out, structured errors — lives in link:../../marketplace/bundles/plan-marshall/skills/ref-workflow-architecture/standards/manage-contract.md[`ref-workflow-architecture/standards/manage-contract.md`].

The hard rule (Layer 1) forbids direct file ops; the proxy + `manage-*` interface (Layer 2) is the *path* the rule directs the LLM to. Together they collapse "all the ways the LLM might touch state" down to one auditable surface.

**Strength**: a permission-allowlist enforced by Claude Code. The LLM is free to try a direct `Read .plan/foo.json` — the absence of a matching allow surfaces a prompt before damage. Operations through the proxy carry their notation, args, and exit status into `script-execution.log` automatically (see xref:audit-trail.adoc[Audit Trail]).

The broader compromise — why narrowing the runtime surface is the central security idea — lives in xref:security.adoc[Security].

== Layer 3 · Q-Gate findings · checkpoint at phase entry

Three phases write Q-Gate findings into the per-plan findings store:

* `phase-2-refine` for lesson-derived plans (Step 13.5).
* `phase-3-outline` Step 11 — outline-time Q-Gate (bypassable when `scope_estimate=surgical` AND `change_type ∈ {bug_fix, tech_debt, verification}` AND `deliverable_count=1`).
* `phase-4-plan` Step 9b — plan-time Q-Gate (always).

The shape is producer / consumer: the Q-Gate workflow body produces findings into `artifacts/findings/qgate-{phase}.jsonl`; the **next phase's entry protocol** queries them and refuses to start work until every pending finding has a resolution (`fixed`, `taken_into_account`, `deduplicated`, or `reopened`). The protocol is documented in link:../../marketplace/bundles/plan-marshall/skills/ref-workflow-architecture/standards/phase-lifecycle.md[`phase-lifecycle.md`] § Phase Entry Protocol.

**Strength**: a *phase checkpoint*. The LLM cannot enter the next phase until each finding is resolved through `manage-findings qgate resolve`. The data lives in the findings store, hash-id deduplicated, so re-running the Q-Gate doesn't drop pre-resolved entries. Full architectural synthesis (producers, store, consumers, the four resolution outcomes) is in xref:automatic-reviews.adoc[Automatic Reviews].

== Layer 4 · Phase handshake invariants

The mechanical gate. Every phase boundary captures a fingerprint of key invariants at completion; the next phase's entry re-evaluates each fingerprint against live state and refuses to continue on mismatch. This is the difference between *the LLM said it edited 5 files* and *the worktree shows 5 modified files at the expected SHA*.

image::../resources/diagrams/handshake-protocol.svg[Sequence diagram of the phase-handshake protocol. At phase N completion the orchestrator runs phase_handshake capture, which evaluates an invariant registry (main_sha, dirty count, task_state_hash, qgate_open_count, config_hash, pending_findings_blocking_count, phase_steps_complete) and writes a row to handshakes.toon. At phase N+1 entry the orchestrator runs phase_handshake verify with strict mode; the script reads the captured row, re-runs the invariants against live state, diffs, and returns either ok (continue), drift (surface diffs, exit 1, refuse to advance), or skipped (log warning, continue)., align=center]

The invariant registry lives in `_invariants.py` as `(name, applies_fn, capture_fn)` tuples. Today's registry:

[cols="1,3", options="header"]
|===
| Invariant | What it catches

| `main_sha` | Any commit change on the main checkout since the prior snapshot.
| `main_dirty` | Uncommitted drift on main between phases.
| `worktree_sha` / `worktree_dirty` | Worktree/main confusion or uncommitted drift inside the worktree.
| `task_state_hash` | Silent mutation of `TASK-*.json` between phases (number, status, step outcomes, dependencies).
| `qgate_open_count` | Q-Gate findings still pending — the integer counterpart to Layer 3's structural check.
| `config_hash` | `marshal.json` config swapped mid-run.
| `pending_tasks_count` | A premature transition while fix tasks are still pending.
| `phase_steps_complete` | Required intra-phase steps silently skipped — reads each phase's static `standards/required-steps.md` and refuses capture when any required step is missing or recorded as anything other than `done`.
| `pending_findings_by_type` | Per-type retrospective count of pending findings across the 12-type taxonomy.
| `pending_findings_blocking_count` | **Guarded.** Sum of pending counts across the per-phase `blocking_finding_types` partition. On guarded boundaries (`5-execute → 6-finalize`) capture *refuses* when non-zero.
|===

Two of those invariants — `phase_steps_complete` and `pending_findings_blocking_count` — *raise* at capture rather than recording a value, so the boundary protocol cannot just dismiss them. The full contract (storage format, drift envelope, override path, "adding a new invariant" extension contract) lives in link:../../marketplace/bundles/plan-marshall/skills/plan-marshall/references/phase-handshake.md[`phase-handshake.md`].

**Strength**: a *mechanical gate*. On drift the script returns `status: drift` with a `diffs[invariant, captured, observed]` table; `--strict` makes the exit non-zero. For the guarded `5-execute → 6-finalize` boundary, `manage-status transition --completed 5-execute` *inlines* the strict verify so refusal and advance are one atomic call — the transition refuses to write the new phase state when drift is present.

== What the layers add up to

The four layers compose to a probabilistic-then-mechanical funnel. Most of the time the LLM follows Layer 1's rules and Layers 2-4 sit quietly. When it slips:

* A direct `Edit .plan/foo.json` surfaces a Claude Code prompt (Layer 2 catches what Layer 1 didn't enforce).
* An attempt to enter phase N+1 with unresolved Q-Gate findings is blocked at the entry protocol (Layer 3).
* A misreported "tasks complete" against a dirty worktree is caught when the handshake re-evaluates `worktree_dirty` and `task_state_hash` (Layer 4).

[CAUTION]
====
**Does this guarantee compliance? No.**

Layer 1 is skill-loaded text, not kernel enforcement. Layer 2 narrows the surface but does not prevent the LLM from trying a prohibited operation — it surfaces a prompt as fallback. Layer 3 depends on the LLM actually invoking `manage-findings qgate query` at phase entry (the phase skill always does, but a rogue workflow body could skip it). Layer 4 is the most mechanical of the four and still depends on the orchestrator running `phase_handshake verify --strict` at the right moment (the lifecycle workflows always do; ad-hoc dispatches do not).

The point is not "the LLM can't break the process." The point is that breaking the process is *progressively harder* the deeper a violation tries to reach. Most violations terminate at Layer 1 or Layer 2 with a prompt the developer sees. The handful that reach Layer 4 surface as a `drift` TOON the developer reads on screen — at which point the choice is an authorized override (`capture --override --reason X`) or a manual fix, both of which leave an audit trail.
