= Security
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:README.adoc[Concepts]

Claude Code ships two extremes: **accept-edits** prompts the developer on every meaningful action (safe but flow-killing) and **YOLO** (`--dangerously-skip-permissions`) skips every prompt (fast but a single LLM misstep can do anything within the user's shell). Plan Marshall sits between them with a different approach — **narrow what the LLM does at all**, so the permission surface is small enough to pre-approve with confidence. Three layers cooperate: a normative baseline (`persona-plan-marshall-agent` hard rules), a technical funnel (the script-executor proxy at `python3 .plan/execute-script.py`), and reader/orchestrator/writer isolation for untrusted external content (a read-only reader plus a deterministic validator script).

[CAUTION]
====
**Does this solve the problem completely? No.**

`persona-plan-marshall-agent` is skill-loaded *guidance*, not kernel-level enforcement. The LLM can still ignore the rules; nothing in Claude Code's runtime forces a subagent to honour a skill's `Prohibited actions` list. Plan Marshall lowers the probability of unsafe behaviour and bounds the blast radius via narrow allow-lists — it does not eliminate the risk.

The model is **way fewer prompts than accept-edits, way more secure than YOLO**. For the remaining cases where Claude Code violates the rules anyway, you still get the prompt — just much less often than you would interactively, and the prompt itself is the safety net.

== Layer 1 — `persona-plan-marshall-agent` as the normative baseline

Every plan-marshall subagent loads `plan-marshall:persona-plan-marshall-agent` as Tier-1 *before* anything else (see xref:skill-handling.adoc[Skill Handling]). The hard-rules block narrows what the subagent is allowed to *try*: no unconstrained generic subagents (every dispatch routes through `execution-context-{level}`), `.plan/` access through scripts only, one Bash command per call, no shell constructs (`for`/`while`/`$()`/subshells/heredocs), structured queries before `Glob`/`Grep`, build commands resolved via the architecture API rather than hard-coded, CI operations through the abstraction rather than direct `gh`/`glab`.

These rules cut off the gnarliest LLM failure modes — shell injection through `$()`, runaway loops, ad-hoc subagents with full tool access — before they can fire. The rules are guidance, not enforcement, but loaded at every dispatch they sit in the subagent's context at every decision point.

== Layer 2 — The script-executor proxy as a technical funnel

Almost every action plan-marshall takes routes through one Bash pattern: `python3 .plan/execute-script.py {bundle}:{skill}:{script} [args...]`. The executor is a Python proxy with an embedded notation-to-script map: only registered scripts resolve, unknown notations error out, and the executor never `exec`s shell strings — it dispatches to Python directly.

This shape means a single broad Claude Code permission (`Bash(python3 .plan/execute-script.py *)`) covers the bulk of plan-marshall's work. The wildcard is bounded: it can match only arguments the executor itself accepts, and the executor only dispatches to registered scripts. Without the proxy, the same coverage would require either a prompt per operation or a dozens-of-entries-long allow list that's easy to get wrong. With it, hundreds of distinct plan-marshall operations (every `manage-*`, every build, every triage, every finalize step) collapse into one allow entry.

The settings hierarchy itself follows Claude Code's standard two-file split — `.claude/settings.json` is the team-shared allow list (small, version-controlled, same for every developer) and `.claude/settings.local.json` is the per-developer customisation (gitignored, larger, individual). The canonical sample patterns and the resolution rules between them live in link:../../marketplace/bundles/plan-marshall/skills/tools-permission-doctor/standards/permission-architecture.md[`permission-architecture.md`].

== Layer 3 — Reader/orchestrator/writer isolation for untrusted external content

Layers 1 and 2 narrow what a plan-marshall subagent *tries* and *can reach*. They do not address a distinct threat: bytes the project does not author and cannot trust — web pages fetched during research, GitHub issue/PR/comment bodies fetched during CI/review, Sonar issue messages fetched during triage. Any of these can carry a prompt-injection payload crafted to read as instructions and hijack a write-capable LLM context. Layer 3 contains that threat by splitting untrusted ingestion across three roles defined in the shared link:../../marketplace/bundles/plan-marshall/skills/untrusted-ingestion/SKILL.md[`untrusted-ingestion`] standard (threat model, read-only-reader contract, output-schema discipline):

* **Reader** (`execution-context-reader-{level}`) — a read-only variant whose tool surface is exactly `WebSearch, WebFetch, Read, Grep` (no `Write`/`Edit`/`Bash`/`Skill`). Its only job is *semantic extraction*: parsing raw external text into a CANDIDATE struct. A reader that an injection payload hijacks still has no tool with which to write, edit, execute, or load a skill — the blast radius of a reader-side hijack is bounded to emitting a (possibly malformed) candidate.
* **Validator script** (`plan-marshall:untrusted-ingestion:validate_struct`) — the *containment boundary*. This is the load-bearing point: the read-only tool surface alone is NOT the containment. The deterministic script is. The orchestrator runs it on the reader's candidate before any write-capable context consumes it. It enforces the output schema (`additionalProperties:false` + `maxLength` + `maxItems` + `pattern`), length-caps/truncates strings and arrays, and performs the WebFetch domain-allowlist check by reusing link:../../marketplace/bundles/plan-marshall/skills/workflow-permission-web/SKILL.md[`workflow-permission-web`] logic (`permission_web.categorize_domain` / `permission_web.check_red_flags`). It returns `status: success` with the clamped struct, or `status: error` on any schema or domain violation.
* **Orchestrator / writer** (`execution-context-{level}`) — the write-capable variant. It consumes ONLY the script-validated, clamped struct — never the raw bytes, never an unvalidated candidate. On `status: error` it aborts.

The reader does semantic extraction only; its candidate struct is untrusted until script-validated. Security does not rest on the reader behaving — it rests on a deterministic script the orchestrator runs before consumption. Each ingestion surface (the web research workflow, the GitHub/Sonar CI/review surfaces, the external-issue-body refine path) applies this reader-dispatch-then-validate pipeline, gaining one extra dispatch hop plus the script gate. The architectural decision — reader isolation as a tool-surface lever distinct from the level/effort lever, and the deterministic script as the containment — is recorded in xref:../adr/003-Reader_isolation_as_a_tool-surface_lever.adoc[ADR-003].

== When prompts still happen

Plan Marshall reduces prompt frequency; it does not eliminate prompts. Expect prompts for: Bash file-ops that the LLM tries despite the hard-rule ban (`find`, `cat`, `grep`, `ls`, `head`, `tail`), compound shell commands the rules forbid (`&&`, `;`, `$()`, heredocs, `for`/`while`), direct `gh`/`glab` calls (the abstraction is the documented route), and novel WebFetch domains.

Treat every prompt as a signal. A prompt for something plan-marshall should never have tried means a hard-rule violation made it through the skill-load layer. The right response is usually to deny, redirect, and — if it recurs — capture as a lesson.

== On-demand security audit: recipe-security-audit

The three layers above are *preventive* — they narrow what the LLM tries, funnel its actions, and contain untrusted input. They do not, on their own, proactively *look for* security defects in the code a plan produces. The link:../../marketplace/bundles/plan-marshall/skills/recipe-security-audit/SKILL.md[`recipe-security-audit`] recipe is the on-demand entry point for that: the operator invokes it (`/plan-marshall recipe=security-audit`) to run a security review over the current footprint, bounded to an optional path/file scope and an optional explicit base branch (defaulting to the current-branch-vs-base diff).

The audit gathers *best-effort, action-general* security context — exactly three skills:

* link:../../marketplace/bundles/plan-marshall/skills/persona-security-expert/SKILL.md[`persona-security-expert`] — the action-general security identity: OWASP Top Ten, STRIDE, trust-boundary and secure-coding principles.
* link:../../marketplace/bundles/plan-marshall/skills/untrusted-ingestion/SKILL.md[`untrusted-ingestion`] — the untrusted-external-content ingestion contract (the same Layer-3 standard above).
* link:../../marketplace/bundles/plan-marshall/skills/workflow-permission-web/SKILL.md[`workflow-permission-web`] — web-domain permission analysis.

It runs the cognitive review across the footprint and emits each finding as a `bug` (a concrete defect) or `anti-pattern` (a risky-but-not-yet-exploitable structural pattern) into the same `manage-findings` store every producer writes to — there is no `security-issue` finding type; the taxonomy is closed and security maps onto existing types. Each finding dispatches to its domain `ext-triage-{domain}` extension, so a security audit flows through the identical triage envelope and four-outcome resolution model as every other producer (see xref:automatic-reviews.adoc[Concepts › Automatic Reviews]).

This is the *best-effort* security pass, not the proactive pre-ship gate. The latter — a `finalize-step-security-audit` that runs automatically at finalize and resolves per-domain `skills_by_profile.security` skills through a dedicated `security` profile — is workstream 05's deliverable. It *reuses* the same shared five-stage engine this recipe runs (authored at link:../../marketplace/bundles/plan-marshall/skills/recipe-security-audit/standards/audit-engine.md[`recipe-security-audit/standards/audit-engine.md`]), plugging its per-domain skills in additively without reshaping the procedure.

== Audit trail

Because every action routes through the script-executor proxy, every script invocation is logged automatically. Combined with the structured `[STATUS]` / `[DISPATCH]` / `[SKILL]` / `[ARTIFACT]` tags emitted by `execution-context`, the per-plan findings JSONL, and the phase-handshake invariants, this produces a comprehensive, machine-readable record of every plan run. The full story — what's logged, where, how `plan-retrospective` evaluates it — is its own concept page: xref:audit-trail.adoc[Audit Trail].
