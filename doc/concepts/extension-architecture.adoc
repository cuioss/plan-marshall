= Extension Architecture
:nofooter:
:toc: left
:toclevels: 2

How Plan Marshall is built as a small core plus a set of domain bundles that plug in through eight named hooks. The contract every bundle satisfies, the discovery mechanism, and how to add a new domain.

== Overview

The Plan Marshall *core* â€” the orchestrator, the six phase skills, the `execution-context` dispatcher, the `manage-*` services â€” never decides what `Java` or `documentation` or `OCI containers` means. Every domain-specific behaviour lives in a separate *bundle* that the core discovers, registers, and calls through a fixed set of hooks.

image::../resources/diagrams/extension-topology.svg[Extension topology. The Plan Marshall core (orchestrator + phase-1-init through phase-6-finalize + execution-context dispatcher + manage-* services + the `extension_discovery.py` scanner) sits on the left. The right column shows the eight named extension hooks the core exposes: build, triage, recipe, outline, provider, execution-context-workflow, verify-steps, and finalize-steps. Each hook is annotated with its `ExtensionBase` method name and a current implementation count. Two hooks are placeholders today (verify-steps, finalize-steps). All ten production bundles implement `get_skill_domains()`; the other hooks are optional., align=center]

The seam between core and bundle is intentionally narrow â€” eight hook methods on a Python base class, plus one filesystem convention for discovery. Once you know the seam, every domain bundle ships exactly the same way.

== The seam â€” `ExtensionBase`

Every bundle that wants to extend the core ships exactly one file:

[source]
----
marketplace/bundles/{bundle}/skills/plan-marshall-plugin/extension.py
----

The directory name `plan-marshall-plugin` is hardcoded into the discovery scanner â€” `find_extension_path()` in link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/extension-contract.md[`extension-contract.md`] looks for that exact path in every installed bundle. The file declares a single class:

[source,python]
----
from extension_base import ExtensionBase

class Extension(ExtensionBase):
    def get_skill_domains(self) -> list[dict]:
        ...
----

`ExtensionBase` declares one abstract method (`get_skill_domains`) and seven optional hooks with default no-op implementations. A bundle only overrides the hooks it actually needs.

A bundle doesn't have to fill every slot. A frontend bundle might only override `get_skill_domains`, `provides_triage`, and ship a profile-overriding workflow body â€” three hooks of eight. The unfilled hooks return the default no-op and the core uses its generic behaviour.

== Discovery

At install time, the executor's `extension_discovery.py` scans every bundle under `marketplace/bundles/` (or the equivalent plugin-cache path) and asks "does `skills/plan-marshall-plugin/extension.py` exist?" If yes, it imports the file, instantiates the `Extension` class, and calls `get_skill_domains()` to learn the domain identity.

The returned domain key (e.g. `java`, `javascript`, `oci`) is registered in `marshal.json` under `skill_domains.{key}`, with a `bundle` field added by `skill-domains configure` that reverse-maps from the domain key back to the source bundle. From that point on every other hook is invoked through the same `Extension` instance whenever its host workflow needs it.

The full marshal.json schema â€” every path each hook writes, every consumer that reads it â€” is catalogued in link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/marshal-json-reference.md[`marshal-json-reference.md`].

== Adding a new domain

Three steps:

1. **Create the directory** `marketplace/bundles/{bundle}/skills/plan-marshall-plugin/` and add an `extension.py` that subclasses `ExtensionBase` and implements `get_skill_domains()`.
2. **Add `skills/`** for the domain knowledge the bundle provides (`{bundle}/skills/{skill}/SKILL.md` per skill). At minimum a `core` profile skill set so something loads during dispatches.
3. **Run `/marshall-steward`** in the consuming project. The wizard discovers the new bundle, calls `get_skill_domains`, writes the registration into `marshal.json`, and prompts for any optional configuration (credentials, profile overrides, finalize steps).

Adding a hook to an existing bundle is even smaller â€” override the relevant method on the existing `Extension` class. The wizard picks the new declarations up on re-run.

The reference implementation of "what minimum looks like" is link:../../marketplace/bundles/pm-dev-frontend-cui/skills/plan-marshall-plugin/[`pm-dev-frontend-cui`] â€” `get_skill_domains` only, no other overrides, useful as a starting skeleton.

== Where this fits

Extension Architecture is the *plumbing* answer to "what is Plan Marshall." The xref:planning-workflow.adoc[Planning Workflow] is the *user-visible* answer. The eight hooks are the contact points between the two: every domain decision a plan makes â€” which build skill to dispatch (xref:build-management.adoc[Build Management]), which triage skill loads for a Sonar issue (xref:automatic-reviews.adoc[Automatic Reviews]), which skills enter the subagent context for a Java testing task (xref:skill-handling.adoc[Skill Handling]) â€” flows through one of the eight hooks.
