How Plan Marshall is built and tested locally.

== Overview

This project uses https://pyprojectx.github.io/[Pyprojectx] for Python development tooling, following the Maven/Gradle wrapper pattern. Only Python 3 is required on the host system — and that host interpreter *is* the build interpreter: Pyprojectx creates its tool venv on the same interpreter that invoked `./pw`. See <<Which Interpreter Runs the Build>>.

== Rationale

Pyprojectx was chosen because it provides a proven, mvnw/gradlew-style wrapper for Python that requires only Python 3 on the host system. Unlike custom solutions, it is actively maintained (since 2022), supports multiple tools (uv, ruff, mypy, pytest), and provides reproducible builds via lock files. This approach avoids reinventing the wheel while matching industry best practices for 2025.

== Canonical Commands

[IMPORTANT]
====
Never hard-code `./pw` in workflow or skill code. Always resolve build commands via the architecture API:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:build-pyproject:pyproject_build run --command-args "{command}"
----

The `./pw` shortcuts below are appropriate for ad-hoc local development only.
====

[source,bash]
----
# Canonical commands with optional module filtering (see build.py)
./pw compile                      # mypy marketplace/bundles/
./pw compile pm-dev-frontend      # mypy single bundle
./pw test-compile                 # mypy test/
./pw test-compile plan-marshall   # mypy single test directory
./pw module-tests                 # pytest test/
./pw module-tests pm-dev-frontend # pytest single directory
./pw quality-gate                 # ruff check all
./pw quality-gate pm-dev-java     # ruff on single bundle + tests
./pw coverage                     # pytest with coverage
./pw verify                       # Full: quality-gate + test-compile + module-tests
./pw verify pm-dev-frontend       # Full verification on single bundle
./pw clean                        # Remove build artifacts
----

Always call build commands with a Bash timeout of at least 10 minutes (600000ms) when running them from inside a Plan Marshall workflow. The build emits TOON; analyse `status`, `errors[N]{file,line,message,category}`, and `log_file` as needed.

== Quality-Gate Coverage

The `quality-gate` build target (root `build.py:cmd_quality_gate`) is the fast-feedback stage developers run on every iteration. It enforces the following marketplace-wide invariants in seconds, so violations are caught before push rather than at CI time:

[cols="1,2,2", options="header"]
|===
| Stage | Tool / Rule | Scope

| Type-check
| `mypy`
| Production sources (`marketplace/bundles`, `.claude` when present)

| Lint
| `ruff check`
| `marketplace/bundles`, `test`, `.claude` (full tree) or scoped paths (module run)

| Static-analysis invariants
| `pm-plugin-development:plugin-doctor:doctor-marketplace quality-gate`
| Marketplace-wide (full-tree run only)
|===

The plugin-doctor `quality-gate` subcommand runs only the rules whose violations are currently enforced as build-failing invariants by the pytest suite (real marketplace must produce zero findings):

* `scan_argparse_safety` — every `argparse.ArgumentParser(...)` and `subparsers.add_parser(...)` call must specify `allow_abbrev=False`. Prevents prefix-abbreviation matching from silently accepting truncated flags.
* `validate_extension_contracts` — extension-point implementations must declare the required contract sections (severity guidelines, acceptable-to-accept lists, etc.) per `plan-marshall:extension-api`.
* `analyze_argument_naming` — notation/subcommand/flag/canonical-forms cluster. Unconditionally active under `quality-gate` (build-failing invariant). The same cluster is *gated off by default* under the `analyze` subcommand and opt-in only via `--rules argument_naming` (or the `--enable-argument-naming` alias) on `plugin-doctor analyze`. The companion `verb_chain` cluster follows the same shape: opt in with `--rules verb_chain` or `--enable-verb-chain`.

Module-scoped quality-gate runs (`./pw quality-gate <module>`) skip the marketplace-wide plugin-doctor sweep because they target a single bundle.

*Coverage scope*: `quality-gate` does NOT run pytest tests. Invariants enforced by pytest fixtures (helper-classification rules, fixture-driven detection tests, integration-test contracts) still require `module-tests` or `verify`. New plugin-doctor static-analysis rules are picked up automatically once their `scan_*` / `validate_*` entry point is wired into `cmd_quality_gate` in `doctor-marketplace.py`; no per-rule registration in `build.py` is needed.

== Tool Configuration

Tool configuration is in `pyproject.toml`:

* `[tool.pyprojectx.main]` — dev tool requirements (uv, pytest, ruff, mypy)
* `[tool.pyprojectx.aliases]` — command shortcuts
* `[tool.pytest.ini_options]` — pytest configuration
* `[tool.coverage.run]` / `[tool.coverage.report]` — coverage data-collection + reporting settings. The HTML output directory (`.plan/temp/htmlcov/`) is wired in `build.py` via `--cov-report=html:…`, not in `pyproject.toml`.
* `[tool.ruff]` — linter/formatter configuration
* `[tool.mypy]` — type checker configuration

=== Which Interpreter Runs the Build

**The build interpreter is whichever `python3` invokes `./pw`.** No entry in `pyproject.toml` selects it, and no `[tool.pyprojectx]` key can.

The chain, established by reading the installed Pyprojectx 3.3.2 sources:

. `pw` bootstraps its own Pyprojectx venv from `sys.executable` — the interpreter that ran `pw`.
. `pyprojectx/env.py::IsolatedVirtualEnv.install()` then creates the tool venv with `uv venv --python {sys.version_info.major}.{minor}` — the version of the *Pyprojectx process*, i.e. still the invoking interpreter. The venv directory is named accordingly (`.pyprojectx/venvs/main-<hash>-py<X.Y>`), which is the observable check for which interpreter a checkout is actually building on.
. `pytest`, `mypy`, and `ruff` are console scripts inside that venv and are reached through `PATH`, so they run on that same interpreter. The `uv run` prefix in the `[tool.pyprojectx.aliases]` commands does *not* redirect them: `uv run` creates a project `.venv` from `[project].requires-python`, but the dev tools are not installed there, so `uv run pytest` falls through to the `PATH` entry in the Pyprojectx venv.

`[tool.pyprojectx].lock-python-version` does **not** select the build interpreter. `pyprojectx/config.py` reads it into `Config.lock_python_version`, and `pyprojectx/lock.py::_freeze()` passes it to `uv pip compile --python-version` — it governs *requirement resolution* when writing `pw.lock`, nothing else. Setting it to `3.12` on a host whose `python3` is 3.14 yields a `…-py3.14` tool venv; the value is inert with respect to the runtime.

CI's interpreter is likewise not declared in this repository. `.github/workflows/python-verify.yml` delegates to the SHA-pinned reusable workflow `cuioss/cuioss-organization/.github/workflows/reusable-pyprojectx-verify.yml@6278bbd7280a6e94b4914aa9e8c61cf23ea1abbe`, whose `actions/setup-python` step is conditional: it runs only when a Python version is supplied, either through the reusable workflow's `python-version` input or through the `pyprojectx.python-version` key of `.github/project.yml`, which the pinned `read-project-config` action reads. This repository sets neither, so `setup-python` is skipped and `./pw` runs on the runner's default `python3`.

Consequently the local and CI build interpreters are independent of each other and of `pyproject.toml`, and they can and do differ. Adding a `pyprojectx.python-version` entry to `.github/project.yml` is the sanctioned mechanism for pinning the CI side.

Three declarations exist and are related but distinct — none of them is the build interpreter:

* `[project].requires-python` — the minimum interpreter version the project supports, and the version `uv run` uses for the (tool-less) project `.venv`
* `[tool.pyprojectx].lock-python-version` — the `--python-version` passed to `uv pip compile` when regenerating `pw.lock`
* `[tool.mypy].python_version` — the language level mypy type-checks against, independent of the interpreter mypy itself runs on

A `.python-version` or `.tool-versions` file is deliberately absent — adding one would introduce a second, competing interpreter declaration that no part of this build consults.
