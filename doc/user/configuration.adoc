== Configuration surfaces — at a glance

[cols="1,1,3", options="header"]
|===
| Surface | Tracked? | What it controls

| `.plan/marshal.json`
| ✅ committed

| `.plan/run-configuration.json`

| n/a
| One-time opt-in for terminal-title integration. See xref:terminal-title.adoc[Terminal Title] § VS Code.

[#marshal-json]
== `.plan/marshal.json`

The bulk of project configuration. Canonical schema and field reference: link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`].

`marshal.json` is organised by phase under `plan.phase-1-init` … `plan.phase-6-finalize`, plus the top-level `providers`, `project`, `skill_domains`, and `system` sections. Each phase block holds fields specific to that phase, including the run-at-all gates and the flat auto-continuation knobs, each homed under the phase whose decision it governs. The phase-5/6 `verification_steps` / `steps` are id-keyed maps; step-owned parameters nest under their owning step's value (see <<step-configuration>>). The sections below cover the fields project authors tune regularly; for everything else (`branch_strategy`, `use_worktree`, `confidence_threshold`, `compatibility`, `simplicity`, `commit_and_push`, `per_deliverable_build`, `cost_size_token_table`, `per_envelope_budget_tokens`, `max_iterations`, the flat `plan.phase-6-finalize.*` auto-continuation knobs, `open_in_ide`, the `system.retention.*` knobs, the flat `plan.phase-6-finalize.checks_wait_timeout_seconds`, the step-owned params nested under `plan.phase-6-finalize.steps` — `review_bot_buffer_seconds`, `pr_merge_strategy`, `final_merge_without_asking`, `auto_rebase_threshold`, `touched_file_cleanup`, `do_transition`, `ce_wait_timeout_seconds` — the `project.*` branch-naming knobs, …) see the canonical per-key reference in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`].

[#review-gates]
=== Review gates

Boolean fields under each phase that decide whether the next-phase transition pauses for user review or auto-continues. Each gate lives under the phase whose **outbound** transition it governs.

[cols="1,1,3", options="header"]
|===
| Config key | Default | Governs the transition…

| `plan.phase-1-init.init_without_asking` | `true` | …from `phase-1-init` to `phase-2-refine`. Set to `false` to stop after init and review the captured request before refinement starts.
| `plan.phase-3-outline.plan_without_asking` | `false` | …from the outline review checkpoint to task creation.
| `plan.phase-4-plan.execute_without_asking` | `true` | …from `phase-4-plan` to `phase-5-execute`.
| `plan.phase-6-finalize.finalize_without_asking` | `true` | …from `phase-5-execute` to `phase-6-finalize` (forward direction).
| `plan.phase-6-finalize.loop_back_without_asking` | `false` | …from a `phase-6-finalize` `loop_back` outcome back into `phase-5-execute` (reverse direction). Defaults differ from `finalize_without_asking` to prevent silent unattended loops.
| `plan.phase-6-finalize.auto_merge_after_ci` | `true` | …from a green CI run to the `pr merge --delete-branch` step. `true` (default) merges automatically once CI passes, serialised across concurrent plans by the cross-plan merge-lock; `false` prompts the operator before merging.
|===

Every gate lives under the phase block whose outbound transition it governs; the three auto-continuation knobs (`finalize_without_asking`, `loop_back_without_asking`, `auto_merge_after_ci`) are flat fields under `plan.phase-6-finalize`. Set any gate via `/marshall-steward` → Configuration → Review Gates, or directly with the standard phase `set` verb:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-1-init set --field init_without_asking --value false

python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-6-finalize set --field auto_merge_after_ci --value false
----

[#run-at-all-gates]
=== Run-at-all gates

Separate from the auto-continuation knobs above (which decide *whether to pause* once a gate has run), the per-phase run-at-all gates decide *whether each gate runs at all*. Each gate is a tri-state enum — `auto` (default), `always`, or `never`:

* `auto` — defer to the existing decision machinery (the phase-1-init lane router for the planning gates, the manifest composer for the finalize gates). This is the historical behaviour.
* `always` — force the gate in, overriding the machinery that would otherwise skip it. For the finalize gates this overrides the manifest composer's `scope_gated_finalize` drop on surgical / single-module plans.
* `never` — skip the gate. For a finalize gate that guards a safety net this can mask real failures; `finalize.qgate` is the highest-risk (it can mask real build/test failures and push a red tree).

[cols="1,1,3", options="header"]
|===
| Config key | Default | What the gate controls

| `plan.phase-1-init.deep_lane` | `auto` | Whether the precondition-driven deep planning lane runs. `always` forces deep; `never` forces light.
| `plan.phase-1-init.escalation` | `auto` | Whether the hard-escalation safety ratchet stays live. `never` is the explicit full-speed-full-risk opt-in.
| `plan.phase-2-refine.revalidation` | `auto` | Whether the premise / narrative-vs-code safety check runs.
| `plan.phase-6-finalize.self_review` | `auto` | Whether the pre-submission structural + cognitive self-review is selected into the execution manifest.
| `plan.phase-6-finalize.qgate` | `auto` | Whether the finalize blocking-findings re-capture runs. Highest-risk gate — `never` can push a red tree.
| `plan.phase-6-finalize.plugin_doctor` | `auto` | Whether the structural marketplace lint runs before push.
| `plan.phase-6-finalize.simplify` | `auto` | Whether the holistic post-implementation simplification sweep runs. `always` forces it in even when the composer's `simplify_inactive` pre-filter would drop it; `never` skips it.
| `plan.phase-6-finalize.steps['default:finalize-step-security-audit'].security_audit` | `auto` | Whether the proactive `finalize-step-security-audit` pre-ship security sweep runs. `auto` defers to the composer's `security_audit_inactive` pre-filter; `always` forces it in even when that pre-filter would drop it; `never` skips it. The step contract (footprint scope, the `extra_security_skills` plug-in, hardening edits committed by the dispatcher's instrumentation) lives in link:../../marketplace/bundles/plan-marshall/skills/phase-6-finalize/standards/finalize-step-security-audit.md[`finalize-step-security-audit.md`].
|===

Read or set the run-at-all gates with the standard phase `get`/`set` verb against the owning phase block:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-6-finalize get --field self_review

python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-6-finalize set --field plugin_doctor --value always
----

See the canonical per-key reference in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`] for the full per-phase gate catalogue.

NOTE: The planning-time Q-Gate validation on `plan.phase-3-outline` / `plan.phase-4-plan` is *not* a run-at-all gate. It is governed by the separate `q_gate_validation` re-run knob (`off` \| `once` \| `until_clean`) documented in <<q-gate-validation>> below. The finalize `qgate` gate above is a different gate — the pre-push-quality-gate — and is unrelated.

[#q-gate-validation]
=== Planning-time Q-Gate validation

`plan.phase-3-outline.q_gate_validation` and `plan.phase-4-plan.q_gate_validation` control how much the planning-time Q-Gate validation re-runs over the emerging outline (phase-3) and task plan (phase-4). Each is a three-value re-run knob — *not* the `auto`\|`always`\|`never` run-at-all enum — tuned independently per phase:

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `plan.phase-3-outline.q_gate_validation` | `once` | How the outline Q-Gate re-runs. `off` skips Q-Gate validation entirely; `once` (default) runs a single validation pass and does not re-loop on findings; `until_clean` re-validates until the outline reports no blocking findings. The first pass always runs; re-runs fire only when the outline content changed since the last pass.
| `plan.phase-4-plan.q_gate_validation` | `once` | How the task-plan Q-Gate re-runs. Same `off`\|`once`\|`until_clean` semantics as the outline knob, tuned independently over the emerging task plan.
|===

The first validation pass always runs unconditionally (it is never gated by confidence); a re-validation is skipped when the outline/plan content is unchanged since the prior pass, so an already-clean outline does not trigger a fresh Q-Gate dispatch. Set either knob via the standard phase `set` verb:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-3-outline set --field q_gate_validation --value once
----

See the canonical per-key reference in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`] for the field semantics and validator.

[#recipe-routing]
=== Recipe-match routing

Tier 1 routing scores an informal request against the registered recipe set and may auto-route it into a matching recipe's known-shape path (see xref:../concepts/recipes.adoc[Concepts › Recipes]). Two flat `phase-1-init` knobs govern the auto-route gate:

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `plan.phase-1-init.auto_route_recipe` | `true` | Whether a strong recipe match auto-routes without prompting. `true` persists the matched recipe and shortcuts into its path when the top match clears the floor; `false` always surfaces the ranked matches for the user to pick from (or decline).
| `plan.phase-1-init.auto_route_recipe_threshold` | `0.6` | The confidence floor a top recipe match must clear to auto-route. Below the floor the matches are proposed rather than auto-selected.
|===

Set either knob via the standard phase `set` verb:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-1-init set --field auto_route_recipe --value false

python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-1-init set --field auto_route_recipe_threshold --value 0.7
----

The companion `aspect-classify` step labels a request as analysis, planning, or implementation so the execution-manifest composer can drop build / quality-gate / test verification steps for analysis- and planning-only requests. Its classification uses an independent confidence threshold (`0.7`) that is *not* the recipe-match auto-route floor above — the two thresholds are unrelated and must not be conflated. The `recipe-match` and `aspect-classify` verbs are documented in xref:../../marketplace/bundles/plan-marshall/skills/manage-config/SKILL.md[`manage-config/SKILL.md`] Canonical invocations.

[#micro-lane-fast-path]
==== Micro-lane fast path (surgical-fix recipe)

The `surgical-fix` recipe is a pre-diagnosed **micro-lane**: a request whose root cause and exact change are already known, bounded to a single module (≤~2 files / ~50 LOC, no cross-module behavioral delta), composed as a deterministic `scope_estimate: surgical` outline that skips the discovery pass and Q-Gate. It introduces no new config knob — it composes entirely from the two surfaces documented on this page:

* **Recipe-match auto-route (<<recipe-routing>>)** — a strong request match auto-routes into the recipe when `plan.phase-1-init.auto_route_recipe` is `true` and the top match clears `auto_route_recipe_threshold`; below the floor the match is proposed rather than auto-selected, exactly like any other recipe.
* **Lane seed (<<execution-profile-lane-selection>>)** — the recipe declares a `lane:` seed of `profile: minimal` with `plan-marshall:automatic-review` force-kept at `minimal`, so structural PR review stays in the loop even on the leanest posture. The `surgical` scope estimate also scores the *planning* lane `light` through the `phase-1-init` router (see <<run-at-all-gates,`deep_lane`>> for the separate planning-depth axis). The seed is the **lowest-precedence** posture input (recipe seed < operator posture < coverage-cell adversarial floor): when `lane_selection: ask`, the seed is surfaced as the pre-selected `minimal` recommendation and any explicit operator posture overrides it; when `lane_selection: auto`, the projection is taken silently.

The net effect is light planning plus a minimal execution profile with the bot-review safety net retained — the shortest safe path for a mechanical, pre-diagnosed change. The user-facing when-to-use / when-NOT-to-use contract lives in xref:recipes.adoc[User Guide → Recipes]; the recipe `lane:` seed contract is owned by link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-recipe.md[`ext-point-recipe.md`].

[#execution-profile-lane-selection]
=== Execution-profile lane selection

The **execution profile** is an operator-facing, cost-visible posture — `minimal`, `auto`, or `full` — chosen at plan init that decides how much ceremony (phase-6 finalize steps, adversarial validators, retrospectives) a plan runs. It is a separate axis from the `deep_lane` planning-depth gate above: `deep_lane: always` forces a deep *planning* lane without forcing a `full` *profile*. Every phase / finalize-step element self-declares its lane membership through a `lane:` frontmatter block (`class` / `tier` / `cost_size`); the postures are cutoffs over that lattice, so the posture→steps mapping stays derived, never a maintained list. The `manage-execution-manifest lanes preview` verb resolves all three postures (kept step set + summed token cost) for the init dialogue.

The three postures:

* **`minimal`** — only the tier-`minimal` floor (push, create-pr, ci-verify, branch-cleanup, record-metrics, archive, lessons capture, and the meta-project derived-state steps). No adversarial validators, no security audit, no retrospectives — appropriate for docs / mechanical changes.
* **`auto`** (the projection) — keeps the adversarial validators and prunes the structurally-empty prunable steps (e.g. sonar when there is no code delta); drops the `full`-only steps (security-audit, plan-retrospective).
* **`full`** — every configured step.

Four `marshal.json` knobs govern the mechanism:

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `plan.phase-1-init.lane_selection` | `ask` | Whether init PROMPTS for the posture (`ask`) or takes the computed `auto` projection silently (`auto`). Mirrors the `deep_lane` / `finalize_without_asking` ask/auto family.
| `plan.phase-<phase>.steps.<step>.lane` | _(per element)_ | Per-element override pinning any single lane element, value ∈ `off` \| `minimal` \| `auto` \| `full` \| `ask` (`off` = never run; `minimal` = force-keep in every posture; `ask` = always prompt). Nested under the owning step in the keyed map, like other step-owned params. The shipped default lives in each element's frontmatter `lane:` block; this override is the project-level pin.
| `plan.phase-1-init.lane_prune_thresholds` | `{confidence_complete: 95, linear_change_max_deliverables: 1}` | The tunable bounds for `auto`'s numeric prune predicates — `confidence_complete` (prune `refine` when the post-init confidence proxy is ≥ this value, int 0–100, matching the refine `confidence_threshold`) and `linear_change_max_deliverables` (treat the plan as a linear change, pruning 4-plan decomposition, when the deliverable count is ≤ this value). Tighten to prune more aggressively, loosen to keep more steps.
| `plan.phase-5-execute.cost_size_token_table` | `{XS: 5K, …, XXL: 520K}` | The six-size token table the dialogue sums for the per-posture cost preview — see <<per-envelope-packing-budget,the cost-size token table below>> (shared with the task bin-packer).
|===

[NOTE]
====
**Meta-project derived-state caveat.** For plan-marshall itself, the cache-deploy steps (`deploy-target`, `sync-plugin-cache`, on-main executor regeneration) are correctness-required derived state, not optional ceremony — they carry `lane.class: derived-state` and run under *every* posture by default. An operator may still override one to `off`, but that emits a loud correctness warning ("dropping sync-cache ships a broken cache") rather than a silent drop. `minimal` therefore cannot silently drop required derived state.
====

[NOTE]
====
**Independent of `init_without_asking`.** `lane_selection` and `plan.phase-1-init.init_without_asking` (see <<review-gates>>) are two independent `phase-1-init` gates that govern different decisions and do not interact. `lane_selection` gates only whether init *prompts* for the execution-profile posture; `init_without_asking` gates only the `phase-1-init` → `phase-2-refine` auto-continue. Concretely: `init_without_asking: true` does **not** suppress the `lane_selection: ask` posture dialogue — the posture prompt fires *within* init, before the transition the auto-continue governs — and `lane_selection: auto` does **not** change the init→refine auto-continue, since it only skips the posture dialogue and leaves the transition gate untouched.
====

Set a knob via the standard phase `set` verb (the per-element override goes through the step-param channel):

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-1-init set --field lane_selection --value auto
----

The closed `lane.class` enum, the class→default-tier table, the resolution lattice, and the named prune predicates are owned by link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-lane-element.md[`ext-point-lane-element.md`]; the per-key knob contracts (validators, defaults) live in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`].

[#step-configuration]
=== Step configuration (verify + finalize pipelines)

`plan.phase-5-execute.verification_steps` and `plan.phase-6-finalize.steps` are id-keyed maps of step identifiers (the key insertion order is the execution order). Each value is that step's nested parameter object: step-owned params (e.g. `pr_merge_strategy` under `default:branch-cleanup`, `review_bot_buffer_seconds` under `plan-marshall:automatic-review`, the prefix-stripped sonar params under `default:sonar-roundtrip`) nest under their owning step's value; verify steps and ownerless finalize steps map to the empty object `{}`. Each step ID resolves to one of three sources:

[cols="1,3", options="header"]
|===
| Step kind | Example

| **Built-in** (prefix `default:`)
| `default:verify:quality-gate`, `default:verify:module-tests`, `default:verify:coverage` (phase-5 — canonical-verify form); `default:pre-push-quality-gate`, `default:commit-push`, `default:create-pr`, `default:ci-verify`, `default:sonar-roundtrip`, `default:lessons-capture`, `default:branch-cleanup`, `default:record-metrics`, `default:archive-plan` (phase-6, kebab-case, canonical default order).

| **Project-local** (prefix `project:`)
| Loaded from `.claude/skills/finalize-step-<step>/` — a project-specific finalize step (e.g. a custom deploy or notification).

| **Bundle-optional**
| Opt-in bundle finalize steps (phase-6) — `{bundle}:{skill}` step docs declaring `implements: plan-marshall:extension-api/standards/ext-point-finalize-step` with `default_on: false`, surfaced via `extension_discovery.find_implementors` (e.g. `plan-marshall:plan-retrospective`). They surface in `list-finalize-steps` but are absent from the default step list, so operators must select them explicitly in the wizard.
|===

A step key therefore carries one of three forms, distinguished by prefix:

* `default:` — a *built-in* step backed by a standards/workflow doc in the `phase-6-finalize` (or `phase-5-execute`) skill, e.g. `default:create-pr`.
* `project:` — a *project-local* step loaded from `.claude/skills/finalize-step-<step>/`, e.g. `project:finalize-step-deploy-target`.
* `{bundle}:{skill}` — a *bundle* step resolved through the discovery registry, e.g. `plan-marshall:automatic-review`. Not every `bundle:skill` step is opt-in: `plan-marshall:automatic-review` is a *promoted built-in-equivalent* — a default-on review step that nonetheless carries the `bundle:skill` form (it lives in the `plan-marshall` bundle, not as a `default:` doc), so it appears in the default step list yet is keyed with the `bundle:skill` form `plan-marshall:automatic-review`.

[NOTE]
====
*Retired-key migration.* `manage-config sync-defaults` migrates *retired* step keys in the two keyed-map step containers (`plan.phase-5-execute.verification_steps` and `plan.phase-6-finalize.steps`) to their canonical ids via an explicit, extensible rename table. Its first (and currently only) entry maps both the built-in-prefixed and the bare legacy review-step forms — `default:automated-review` and `automated-review` — to the canonical `plan-marshall:automatic-review`, preserving the step's nested knob block byte-identically. The migration is idempotent and drops a retired duplicate when the canonical is already present, so a stale-key `marshal.json` never yields a double review step. Future renames extend the same table. See link:../../marketplace/bundles/plan-marshall/skills/manage-config/SKILL.md[`manage-config` § Workflow: Sync Defaults].
====

Order is significant — phase-5 runs verify steps in key order until one fails; phase-6 runs finalize steps in key order until the pipeline completes or a `loop_back` outcome resets it. Configure the step list via `/marshall-steward` → Configuration, or the step-list verbs `manage-config plan phase-N set-steps --steps …` / `add-step` / `remove-step` (these operate on the keyed map's keys, preserving insertion order and any existing per-step params, persisting under `verification_steps` for phase-5-execute and `steps` for phase-6-finalize).

==== Step parameters (two-tier source)

Step-owned parameters live in a two-tier source model:

* **Global default (marshal.json keyed map)** — read/written with the one-stop `step` verb: `manage-config plan phase-6-finalize step get --step-id default:branch-cleanup` returns the step's complete nested parameter object in one call; `manage-config plan phase-6-finalize step set --step-id default:branch-cleanup --param pr_merge_strategy --value rebase` writes one parameter. This is the compose-time default and the `/marshall-steward` wizard's write target.
* **Plan-local runtime (execution manifest snapshot)** — at plan-compose time the composer snapshots each selected step's resolved parameters into the per-plan execution manifest. The phase-5/6 runtime consumers read parameters from this plan-local snapshot via `manage-execution-manifest step-params get --phase {5-execute|6-finalize} --step-id {id}`, and a per-plan override is written with `manage-execution-manifest step-params set` (the manifest value then wins over the global default for that plan).

Reference: `data-model.md` §§ phase-5-execute / phase-6-finalize; link:../../marketplace/bundles/plan-marshall/skills/manage-execution-manifest/standards/manifest-schema.md[`manifest-schema.md`] § `step_params`.

[#per-envelope-packing-budget]
=== Per-envelope packing budget and the cost-size token table

The `phase-5-execute` dispatch model is **budget-bounded**, neither per-task nor per-deliverable. The continue-vs-yield decision is **pre-computed at plan time**: phase-4-plan derives each task's T-shirt `cost_size` and `predicted_cost_tokens`, then a deterministic bin-packer groups dependency-ordered tasks into execution envelopes. The executor runs only the tasks in its assigned envelope and yields when the group is exhausted — it performs no runtime cost measurement. Two knobs feed the plan-time packing:

`plan.phase-5-execute.per_envelope_budget_tokens` (default `"400K"`) is the per-envelope packing budget — the token ceiling the bin-packer accumulates `predicted_cost_tokens` against before opening a new envelope group. It is consumed at PLAN time by the bin-packer (`manage-tasks pack-envelopes`), NOT a runtime comparand. The `_tokens` suffix names the unit; the human-friendly value form (`"400K"`) is parsed to an integer by `sensible_number.parse_sensible_int`. Raise it to pack more tasks into each envelope (fewer, larger dispatches), lower it to yield earlier (smaller dispatches). The 400K default leaves headroom below a typical context window.

`plan.phase-5-execute.cost_size_token_table` (default `{XS: 5K, S: 25K, M: 60K, L: 130K, XL: 260K, XXL: 520K}`) maps each T-shirt `cost_size` to a predicted-token magnitude. The bin-packer reads it to map a task's derived `cost_size` to its `predicted_cost_tokens`, and the execution-profile lane dialogue (see <<execution-profile-lane-selection>>) sums each resolved lane element's `cost_size` through the same table to produce its per-posture cost preview. Keys must be exactly the six-size scale `XS`/`S`/`M`/`L`/`XL`/`XXL`; each value parses via `sensible_number.parse_sensible_int`. `XS` (deterministic ≈0-token bookkeeping) and `XXL` (the largest elements) are the two ends added so lane elements and tasks share one scale; the four middle magnitudes are calibrated to the forensic 134K–392K per-dispatch range. Tune them to recalibrate the cost model from observed post-return `<usage>`.

Set via:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-config:manage-config \
  plan phase-5-execute set --field per_envelope_budget_tokens --value 400K
----

[#merge-strategy]
=== PR merge strategy and auto-rebase

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `steps['default:branch-cleanup'].pr_merge_strategy` | `squash` | Step-owned param of `default:branch-cleanup`. The merge method the branch-cleanup step passes to `pr merge` — `squash`, `merge`, or `rebase`. Read/written via `step get` / `step set` (see <<step-configuration>> § Step parameters).
| `steps['default:branch-cleanup'].auto_rebase_threshold` | `no_overlap_only` | Step-owned param of `default:branch-cleanup`. Gates the pre-merge auto-rebase decision. `no_overlap_only` lets the auto-rebase proceed only when it would touch a disjoint file set; any overlap defers to the operator.
| `steps['default:branch-cleanup'].merge_hold_window` | `full_window_release_at_waits` | Step-owned param of `default:branch-cleanup`. Hold-scope mode for the merge mutex. `full_window_release_at_waits` acquires the lock before the pre-merge force-push and holds it through the CI wait, merge, and merge-CI-wait — releasing + FIFO-re-enqueueing at every operator-wait / loop-back boundary and re-validating after re-acquire. `pre_merge_only` is the legacy narrow hold (acquire only at the Pre-Merge Gate).
| `steps['default:branch-cleanup'].merge_hold_budget_seconds` | `3600` | Step-owned param of `default:branch-cleanup`. Bounds (in seconds, ~60 min) the maximum wall-clock the widened merge mutex may be held across the staleness window. Exceeding it during a legitimate wait releases + FIFO-re-enqueues the lock and escalates via a prompt, so a live-but-slow holder cannot monopolize the merge critical section.
| `steps['default:branch-cleanup'].use_merge_queue` | `false` | Step-owned param of `default:branch-cleanup`. Opt-in complement that routes the final merge through the platform merge queue (GitHub merge queue / GitLab merge train) instead of the immediate `pr safe-merge`, serializing a truly-external commit the session-scoped mutex cannot. Default `false` because engaging the platform merge queue is a repo-level branch-protection change affecting all PR workflows.
| `steps['default:branch-cleanup'].pre_merge_comment_barrier` | `fail_into_loopback` | Step-owned param of `default:branch-cleanup`. Governs the fail-closed pre-merge comment-completeness barrier that re-fetches bot comments immediately before merge/enqueue and blocks the merge on any still-unhandled comment (the final net for a comment posted after `plan-marshall:automatic-review` marked done). `fail_into_loopback` (default) loops back into the `plan-marshall:automatic-review` triage pipeline on any pending comment; `ask` fires an inline `AskUserQuestion` (re-triage now / merge anyway with a recorded reason / defer merge). The barrier gates BOTH the immediate `pr safe-merge` and the `use_merge_queue` merge-queue paths.
| `steps['default:pre-submission-self-review'].drop_review_on_scope_gate` | `false` | Step-owned param of the on-by-default `default:pre-submission-self-review` step. Escape hatch for the manifest composer's scope-gating pre-filter. `false` keeps the bot-review invariant intact; `true` opts into additionally dropping `plan-marshall:automatic-review` on scope-gated plans.
|===

NOTE: All `default:branch-cleanup` step params — `pr_merge_strategy`, `final_merge_without_asking`, `auto_rebase_threshold`, `merge_queue_wait_budget_seconds`, `merge_hold_window`, `merge_hold_budget_seconds`, `use_merge_queue`, `pre_merge_comment_barrier`, and `admin_merge_on_stuck_state` — are nested under the `default:branch-cleanup` step in the `plan.phase-6-finalize.steps` keyed map, not flat phase-level fields. Read the whole param object in one call with `manage-config plan phase-6-finalize step get --step-id default:branch-cleanup`.

NOTE: The `pre_merge_comment_barrier` is the final safety net that makes leaving `re_review_on_loopback` off (its `false` default, see <<automated-review>>) safe: even when a loop-back fix commit is not re-reviewed by the bots, the barrier re-fetches and blocks on any unhandled comment before the merge. It is independent of the initial-wait knobs (`review_bot_buffer_seconds`, `review_completion_poll_timeout_seconds`, and the opt-in `review_rate_window_await` rate-window handling): those bound how long the *initial* review wait polls, whereas the barrier is a distinct, always-on re-fetch at merge time — so a comment that lands after the initial wait (a late pass, or one posted once a bot's rate window reopens) is still caught.

NOTE: The `default:pre-push-quality-gate` finalize step has no dedicated activation key. Its activation is *derived from* `build.map` (see <<build-map>>): the step activates whenever a changed path matches a glob registered in the build_map.

[#merge-queue]
=== Platform merge queue (both providers)

The `use_merge_queue` opt-in (see the `default:branch-cleanup` param table above) routes the final merge through the *platform merge queue* so the platform re-tests-and-merges the PR against the latest base branch, serializing a truly-external commit (a manual `gh`/`glab` merge, a dependabot merge, a parallel remediation PR) that the session-scoped merge mutex cannot serialize on its own. It is supported on *both* providers:

* *GitHub* — the platform merge queue, configured as a `merge_queue` rule in a repository ruleset on the default branch. `pr merge-queue` engages it via `gh pr merge --auto`.
* *GitLab* — a *merge train* (a Premium/Ultimate-tier feature enabled per-project). `pr merge-queue` performs a real merge-train enqueue via `POST /projects/:id/merge_trains/merge_requests/:iid`.

==== Provisioning via `/marshall-steward`

Rather than hand-configuring the platform, use the steward's *Merge Queue* provisioning step (first-run wizard Step 13.5, or Configuration → Merge Queue in the maintenance menu). It follows a *probe → ask → configure-or-refuse* pattern and is idempotent (a re-run against an already-configured project surfaces nothing and mutates nothing):

. *Probe* the platform via `ci repo merge-queue probe`, which returns one of the shared eligibility discriminators: `eligible_configured`, `eligible_unconfigured`, `ineligible`, or `unsupported`.
. On `eligible_configured` — report and return silently; no change.
. On `eligible_unconfigured` — ask *once* whether to enable it; on *yes*, run `ci repo merge-queue enable` and persist `use_merge_queue=true`.
. On `ineligible`/`unsupported` — refuse with an actionable message naming the remedy (GitLab tier / GitHub Administration-scope or org policy), never a stack trace.

Setting `use_merge_queue=true` directly (via `step set`) is *probe-validated at set time*: enabling it is rejected with an actionable both-remedies message when a live probe reports `ineligible`/`unsupported`. At finalize time, a merge-queue enqueue failure aborts branch-cleanup with an actionable error naming both remedies (disable `use_merge_queue`, or run the steward provisioning step) — it never silently falls back to an immediate merge.

==== Eligibility is always probed live

There is *no static per-provider capability flag* recording whether the merge queue is available — eligibility is *always* probed live against the actual repository/project state at set time and at provisioning time. The `required_status_checks` "require branches to be up to date before merging" branch-protection setting is left as an *operator choice* (recorded in the steward prompt), not forced by the provisioning step.

==== Verification posture (fixture-vs-live)

The two provider paths carry different verification postures:

* *GitHub — fixture-tested verbs, live-eligible target.* The probe/enable verbs are unit-tested against API-shape-faithful fixtures (not the live `gh` CLI). GitHub merge queue is GA and this repository was confirmed eligible by a live platform probe, so — unlike GitLab — a real, accessible target exists: the steward provisioning exercises the verbs against the live GitHub API the first time it runs on this repo.
* *GitLab — fixture-verified only.* The GitLab merge-train probe/enable/enqueue path is verified against *API-shape-faithful fixtures only* (no live-GitLab-consumer verification is implied). The fixtures mirror the real Projects-API and merge-train response shapes so the handler logic is validated without a live GitLab instance.

[#automated-review]
=== Automated PR review and post-branch-update re-review

The `plan-marshall:automatic-review` finalize step owns the bot-review step-params, nested under `plan.phase-6-finalize.steps['plan-marshall:automatic-review']` in the keyed map. The ones project authors tune regularly govern the initial bot-review interaction (the buffer wait, the completion-aware poll bound, and the two re-review triggers) and the post-trigger await — how long the step waits for a bot to submit a re-review and what it does when that wait elapses.

[cols="1,1,3", options="header"]
|===
| Config key | Default | What it controls

| `steps['plan-marshall:automatic-review'].review_bot_buffer_seconds` | `180` | Step-owned param of `plan-marshall:automatic-review`. Buffer (seconds) the step waits after pushing before polling for bot review comments, giving the review bots time to start their pass. Also the fallback wait for a bot that publishes no completion check-run (empty registry `completion_check_name`), which is not eligible for the completion-aware poll below.
| `steps['plan-marshall:automatic-review'].review_completion_poll_timeout_seconds` | `600` | Step-owned param of `plan-marshall:automatic-review`. Bound (seconds) on the per-bot completion-aware poll: for each enabled bot (from `enabled_bots`) that publishes an in-progress check-run, the wait step polls that bot's completion check to done or until this budget elapses — so a slow bot is *awaited* rather than out-raced by the fixed `review_bot_buffer_seconds`. A bot with no completion check-run falls back to the `review_bot_buffer_seconds` settle; a bot still running at this bound is logged loudly and left to the pre-merge comment barrier (see <<merge-strategy>>).
| `steps['plan-marshall:automatic-review'].re_review_on_loopback` | `false` | Step-owned param of `plan-marshall:automatic-review`. When `true`, a loop-back that advances HEAD re-requests a bot review of the new commits; `false` (default) suppresses the loop-back re-review. Leaving it off is safe because the `pre_merge_comment_barrier` (see <<merge-strategy>>) re-fetches and blocks on any unhandled comment before the merge regardless.
| `steps['plan-marshall:automatic-review'].re_review_on_branch_cleanup` | `true` | Step-owned param of `plan-marshall:automatic-review`. When `true` (default), a branch-cleanup operation that advances HEAD (rebase/merge during finalize) triggers a re-review of the updated branch; `false` suppresses it.
| `steps['plan-marshall:automatic-review'].re_review_await_timeout_seconds` | `600` | Step-owned param of `plan-marshall:automatic-review`. Ceiling (seconds) the step waits for a requested re-review to be submitted by the bot before the `re_review_on_timeout` policy fires.
| `steps['plan-marshall:automatic-review'].re_review_on_timeout` | `ask` | Step-owned param of `plan-marshall:automatic-review`. Policy applied when `re_review_await_timeout_seconds` elapses without a submitted bot review. One of `ask` (default — halt and prompt the operator), `proceed` (advance the unreviewed HEAD to the merge gate), or `defer` (skip the merge so finalize is re-entered later).
|===

NOTE: These params are nested under the `plan-marshall:automatic-review` step in the `plan.phase-6-finalize.steps` keyed map, not flat phase-level fields. Read the whole param object in one call with `manage-config plan phase-6-finalize step get --step-id plan-marshall:automatic-review`; write one with `manage-config plan phase-6-finalize step set --step-id plan-marshall:automatic-review --param re_review_on_timeout --value proceed`.

NOTE: The `review_completion_poll_timeout_seconds` (here) and `pre_merge_comment_barrier` (see <<merge-strategy>>) defaults are seeded automatically from each owning step's `configurable:` frontmatter; existing consumer projects receive them non-destructively via `sync-defaults`' deep-merge — no manual `marshal.json` edit is required.

They govern two different scopes, so the knob you can edit depends on which surface you mean — both are documented here side by side so a consumer reader sees the one available to them next to the one that is not.

[cols="1,1,3", options="header"]
|===

[#build-map]
=== Build map

`marshal.json` carries a `build.map` block at the top level — the file-to-build contract. It maps every changed path to the build action that path requires, so when a plan runs only the relevant build fires: a production-code change compiles, a test change runs the tests, and a docs-only change runs no build at all (documentation has no build owner). The block is required and always seeded.

The `build.map` is seeded automatically. Each domain bundle declares which path globs it owns and what build action each glob needs; the wizard aggregates those declarations into `build.map`. Seeding is write-once — a re-seed never clobbers an existing block — so a correction you make survives.

Each entry classifies a glob into a closed build-class vocabulary:

* `compile` — production code; compiles the changed module.
* `module-tests` — test code; compiles and runs the changed module's tests.
* `verify` — build configuration; runs the full verify pipeline for the module.
* `none` — no build at all (the path is changed but needs no build action).

==== Correcting a wrong mapping

When the seeded mapping is wrong for your project, correct the entry directly in `build.map` — there is no separate override layer. Because seeding is write-once, your correction survives a re-seed.

The common case is a generated source file that the seed treats as production code but that should never trigger a build. Set its entry's `build_class` to `none`:

[source,json]
----
{
  "build": {
    "map": {
      "python": [
        { "glob": "marketplace/bundles/**/scripts/generated_*.py", "role": "production", "build_class": "none" }
      ]
    }
  }
}
----

The canonical schema (the field table, the build-class enum source of truth, the longest-glob-wins precedence) lives in link:../../marketplace/bundles/plan-marshall/skills/manage-config/standards/data-model.md[`data-model.md`] § Section: build.map.

[#worktrees]
=== Branch strategy and worktrees

How a plan decides whether to allocate a feature branch + isolated worktree, where the worktree lives on disk, and when materialization happens. Plan Marshall reads `plan.phase-1-init.branch_strategy` to drive the decision:

* `feature` (default for new plans) — the orchestrator allocates a feature branch named `feature/{plan-id}` and an isolated worktree at `<project_root>/.plan/local/worktrees/{plan-id}/`. `status.metadata.use_worktree` is set to `true`.
* `main` — the plan runs against the main checkout without a worktree (intended for docs-only patches that do not need isolation).

`phase-1-init` records only `use_worktree`. The feature branch (`feature/{plan-id}`) and the worktree directory are created together during `phase-5-execute` Step 2.5, immediately before the first task dispatch. Phases 2-4 (refine, outline, plan) run against the main checkout because they only produce `.plan/` artifacts and have no source-code changes to isolate.

The `.plan/*` exclusion in `.gitignore` covers `.plan/local/worktrees/{plan-id}/` automatically — no per-worktree carve-out is needed. The conceptual overview (why each plan runs in plan-bound isolation, and how the post-execute shipping flow lands the work on `main`) lives in xref:../concepts/branches-and-worktrees.adoc[Concepts › Branches and Worktrees]; the canonical contract (path resolution, dispatch protocol, cleanup ordering) lives in link:../../marketplace/bundles/plan-marshall/skills/workflow-integration-git/standards/worktree-handling.md[`workflow-integration-git/standards/worktree-handling.md`].
