Plan Marshall installs into Claude Code as a marketplace. Installs are `git-clone-from-ref` operations performed by Claude Code's `/plugin marketplace add` against `cuioss/plan-marshall` at the chosen ref — `@dist-claude` for the snapshot branch, `@claude/v{x.y.z}` for an immutable dist tag.

== Prerequisites

[IMPORTANT]
====
Python 3 must be available as `python3` in your PATH.

[source,bash]
----
python3 --version
----
====

=== Supported platforms

Plan Marshall targets a single POSIX runtime. **macOS** and **Linux** are supported natively.

On **Windows**, Plan Marshall runs **exclusively inside WSL2**. The reason is the one-POSIX-runtime requirement: the planning system, its `python3` executor, its git worktrees, and the build server all assume POSIX process, filesystem, and socket semantics, which the WSL2 Linux distro provides and native Windows does not. There is no native-Windows build. For the end-to-end WSL2 + Ubuntu install and editor-integration walkthrough, see xref:windows-wsl-setup.adoc[Windows / WSL Setup]; this section states the prerequisite and its scoping rules.

The all-in-distro rule: the **entire** runtime lives inside the WSL distro — the cloned repositories, `python3`, and the plan-marshall state directory (`~/.plan-marshall/`) all sit on the distro's own ext4 filesystem, never under the Windows-mounted `/mnt/c`. Editing a repo on the Windows side and running Plan Marshall against its `/mnt/c/...` path is unsupported.

One distro equals one machine. Plan Marshall's machine-global state is scoped **per WSL distro**: each distro has its own `~/.plan-marshall/` (credentials, the machine-global build-queue), its own build-server registry, and its own `marshalld` daemon instance. Two distros on the same physical Windows host are two independent Plan Marshall machines and never share state.

Daemon lifecycle on Windows is different from a long-lived Unix host. A `wsl --shutdown`, a Windows reboot, or the WSL idle timeout tears down the distro and with it any running `marshalld` daemon — so a build-server `down` status is **routine** on Windows, not an error. The xref:../concepts/build-server.adoc[phase-1-init build-server preflight] re-asks whether to start the daemon at the start of each planning session, so recovery after a shutdown is a one-prompt step.

[WARNING]
====
Do not place repositories or the plan-marshall state under `/mnt/c` (or any Windows-mounted drive). Cross-boundary filesystem access through the `9p` protocol is dramatically slower than the distro's native filesystem and will make builds, git operations, and plan I/O crawl. Keep everything on the distro filesystem (for example under `~/`).
====

== Snapshot (tracks main)

The `@dist-claude` ref tracks `main`: every push to `main` force-updates this branch to the freshly regenerated `target/claude/` tree.

[source,bash]
----
/plugin marketplace add cuioss/plan-marshall@dist-claude
/plugin install plan-marshall@plan-marshall
/reload-plugins
----

Restart Claude Code or run `/reload-plugins` to load the plugin.

=== Updating the snapshot

[source,bash]
----
/plugin marketplace update plan-marshall
----

Re-pulls `dist-claude` HEAD. Reinstall plugins to pick up updated `plugin.json` entries, then restart Claude Code or run `/reload-plugins`.

== Pinned release

The `claude/v{x.y.z}` tag is immutable — it remains pinned to the orphan commit produced for that release even after `dist-claude` is later overwritten by subsequent `main` pushes.

[source,bash]
----
/plugin marketplace add cuioss/plan-marshall@claude/v{x.y.z}
/plugin install plan-marshall@plan-marshall
/reload-plugins
----

Substitute `{x.y.z}` with the desired version, for example `@claude/v1.2.3`.

=== Updating to a newer pinned release

[source,bash]
----
/plugin marketplace remove plan-marshall
/plugin marketplace add cuioss/plan-marshall@claude/v{new.x.y}
----

Reinstall plugins and restart.

== Installing additional bundles

After the core `plan-marshall` plugin is installed, add the language- or domain-specific bundles you need:

[source,bash]
----
/plugin install pm-dev-java@plan-marshall
/plugin install pm-dev-frontend@plan-marshall
/plugin install pm-dev-python@plan-marshall
/plugin install pm-dev-oci@plan-marshall
/plugin install pm-documents@plan-marshall
/plugin install pm-requirements@plan-marshall
/plugin install pm-plugin-development@plan-marshall
----

The bundle table at the top of the project `README.md` links each bundle name to its `README.md`.
