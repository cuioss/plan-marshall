# Project Review & Remediation Plan

A full-project review of `plan-marshall`, with every finding grouped into an
independently-mergeable remediation PR. This document is a **plan and set of
fix hints only** — it prescribes *what* to change and *why*, not the code
changes themselves. Implementation is out of scope for the change that
introduces this document.

## 1. Scope & Method

The review swept the whole repository across five surfaces in parallel:

1. **Python correctness** — `build.py`, `marketplace/targets/**`, and a deep
   sample of the `plan-marshall` bundle's runtime scripts (executor, `manage-*`,
   `_providers_core`, `file_ops`, `ci_base`, findings, architecture).
2. **Test-suite health** — the `test/` tree (~545 `test_*.py` files),
   `conftest.py`, and `run-tests.py`, plus a script-to-test coverage cross-map.
3. **Bundle/skill structure** — the 10 bundles, their `plugin.json` manifests,
   `marketplace.json`, and every `SKILL.md` / agent / command.
4. **Documentation** — `doc/`, the root docs (`README`, `CLAUDE.md`,
   `AGENTS.md`, `CONTRIBUTING`, `SECURITY`), and all bundle `README.md` files,
   checked against the project's own documentation standards.
5. **Build / CI / supply chain** — `pyproject.toml`, `.github/workflows/**`,
   `pw`/lockfiles, `.gitignore`, and policy files.

**Objective baseline first.** The automated quality gate (`ruff` + `mypy` +
SPDX + `plugin-doctor`, 24 marketplace-wide invariant rules) was run and is
**fully green** — 0 lint issues, 0 type errors across 321 source files, 0
plugin-doctor findings. Every finding below is therefore a **semantic,
structural, or documentation** defect that the linters and static analyzers
cannot catch. There are no style findings.

**On line numbers.** Locations are cited as observed at review time. Line
numbers drift; treat each citation as a pointer to re-confirm against current
`main` when the fix lands, not as a fixed coordinate.

## 2. Executive Summary

The project is in strong mechanical health: the toolchain, invariant linter,
and type checker all pass, registration/count claims verify clean (145
registered / 136 unique skills, 2 agents, 2 commands), and cross-references
largely resolve. The findings are a long tail of real-but-non-blocking issues,
concentrated in three areas:

- **A cluster of latent runtime bugs** in error/edge paths of the plan-marshall
  scripts and the multi-target generator — code that works on the happy path
  but crashes or silently misbehaves on malformed input, HTTP edge cases, or
  unusual filesystem layouts.
- **Test-integrity gaps** — a false-green standalone runner, assertion-to-skip
  guards that let tests rot silently, and a tail of untested helper modules.
- **Documentation drift** — several bundle READMEs undercount their own skills
  (two omit a *security* skill entirely), and a few governance docs contradict
  each other or the code.

| Severity | Count | Character |
|---|---:|---|
| High | 3 | Unreachable fallback, potential source-tree deletion, false-green test runner |
| Medium | ~22 | Edge-path crashes, silent test skips, doc inaccuracies, CI hardening |
| Low | ~28 | Defensive-coding, dead code, cosmetic/consistency, optional polish |

Nothing here blocks release-readiness on its own, but the High items and the
security-skill documentation omissions warrant priority.

## 3. Remediation Plan — PR Clusters

Findings are partitioned into **seven PRs**, each cohesive by module/ownership,
independently reviewable, and independently mergeable. Recommended sequencing
and dependencies are in §4.

Per-finding format: **[ID] Title** — *Severity* · `location` · Problem · Fix hint.

### PR-1 — Runtime script correctness (plan-marshall bundle scripts)

**Scope:** Latent bugs in error/edge paths of the core runtime scripts. Highest
user-facing risk. Each fix should ship with a regression test that exercises the
specific edge path (most are currently unexercised — see PR-3).

**[R1] Executor-generation fallback is unreachable on its most common failure** — *High* ·
`marketplace/bundles/plan-marshall/skills/tools-script-executor/scripts/generate_executor.py:854-858`
Problem: `cmd_generate` wraps `discover_scripts()` in `try/except Exception` to
fall back to `discover_scripts_fallback()`, but `discover_scripts` signals
"inventory not found / non-zero return" via `sys.exit(2)` (raises `SystemExit`,
a `BaseException`, not `Exception`). The handler is bypassed, `safe_main`
re-raises, and the glob fallback never runs for exactly the failure it exists to
cover. The sibling `cmd_drift` (≈line 1013) correctly uses `except SystemExit`.
*Verified:* `sys.exit(2)` at lines 228 and 260.
Fix hint: catch `SystemExit` alongside `Exception` in `cmd_generate`, or have
`discover_scripts` raise a normal exception instead of `sys.exit`.

**[R2] `Retry-After` HTTP-date crashes the REST retry loop** — *Medium* ·
`marketplace/bundles/plan-marshall/skills/manage-providers/scripts/_providers_core.py:504`
Problem: `delay = int(retry_after) if retry_after else (2**attempt)`. Per HTTP
spec `Retry-After` may be an HTTP-date, not delta-seconds; `int()` on a date
string raises `ValueError`, which is outside the caught
`(ConnectionError, TimeoutError, OSError)` and propagates uncaught.
Fix hint: parse defensively — try `int()`, fall back to date parsing or
`2**attempt` on failure.

**[R3] Empty basic-auth password produces a half-configured header** — *Low* ·
`marketplace/bundles/plan-marshall/skills/manage-providers/scripts/_providers_core.py:595`
Problem: for `auth_type == 'basic'` the code validates username (missing /
placeholder) and rejects a placeholder password, but never rejects an *empty*
password — it base64-encodes `f'{username}:'` and emits a broken header. Token
auth rejects an empty token (line 584); basic auth should be symmetric.
Fix hint: add `if not password: raise ValueError(...)` mirroring the username check.

**[R4] `verify_system_auth` tokenizes the command two different ways** — *Low* ·
`marketplace/bundles/plan-marshall/skills/manage-providers/scripts/_providers_core.py:658,676`
Problem: the command runs via `shlex.split(verify_command)`, but the
`FileNotFoundError` handler reports the missing binary as
`verify_command.split()[0]` (naive `str.split`). For a quoted command the two
tokenizations disagree and the error names the wrong token.
Fix hint: compute `shlex.split(...)` once and reuse `tokens[0]` in the message.

**[R5] `get_metadata_content_split` returns overlapping metadata and body** — *Medium* ·
`marketplace/bundles/plan-marshall/skills/tools-file-ops/scripts/file_ops.py:862`
Problem: `body_start` defaults to `0` and is only reassigned on a blank line or
heading. For content that is entirely metadata (no blank line, no heading) the
loop never breaks, so `body` is the whole file *including* the metadata lines,
while `metadata_block` also contains them. `update_markdown_metadata` (line 848)
handles this correctly with a `for…else`.
Fix hint: add a `for…else` that sets `body_start = len(lines)` when no break occurs.

**[R6] Bulk finding-resolve erases `resolution_detail`** — *Low* ·
`marketplace/bundles/plan-marshall/skills/manage-findings/scripts/_findings_core.py:395`
Problem: `updates = {'resolution': to_resolution, 'resolution_detail': detail}`
writes `resolution_detail: None` whenever `detail` is omitted, wiping any stored
detail on bulk resolve. Single-finding `resolve_finding` (line 358) only sets it
when truthy — inconsistent semantics.
Fix hint: only include `resolution_detail` when `detail is not None`.

**[R7] `compute_total_elapsed` subtraction not guarded for naive/aware mismatch** — *Low* ·
`marketplace/bundles/plan-marshall/skills/tools-integration-ci/scripts/ci_base.py:1238`
Problem: the `try` wraps only `datetime.fromisoformat`; the final
`int((now - earliest).total_seconds())` is outside it. If `now` is tz-naive and
`earliest` tz-aware (parsed from a `…Z` stamp), the subtraction raises
`TypeError` uncaught. Sibling `compute_elapsed` guards its subtraction (lines
1206-1213).
Fix hint: move the subtraction inside the `try/except (ValueError, TypeError)`, or
normalize tz before subtracting.

**[R8] Executor validation interpolates a filesystem path into `python3 -c` source** — *Low* ·
`marketplace/bundles/plan-marshall/skills/tools-script-executor/scripts/generate_executor.py:739,766,796`
Problem: `verify_executor`/`get_executor_mappings` build Python source by
f-string-interpolating `real_executor` into
`spec_from_file_location('executor', '{real_executor}')` and run it with
`python3 -c`. A checkout path containing a quote or backslash breaks the
generated program (code-injection-shaped smell).
Fix hint: pass the path via `sys.argv`/env into the `-c` program instead of
interpolating it into the source string.

**[R9] `discover_scripts_fallback` drops any script whose name contains "test"** — *Low* ·
`marketplace/bundles/plan-marshall/skills/tools-script-executor/scripts/generate_executor.py:321`
Problem: the glob fallback skips
`if script_file.name.startswith('_') or 'test' in script_file.name.lower()`. The
bare substring also matches legitimate entrypoints (`latest.py`, `contest.py`,
`*_test_*`), silently omitting them — a discovery gap absent from the primary
inventory path.
Fix hint: restrict to a precise pattern (`test_` / `_test` / `*_test.py`), not a
bare substring.

**[R10] Root-module path collapses to empty string in architecture manager** — *Low* ·
`marketplace/bundles/plan-marshall/skills/manage-architecture/scripts/_cmd_manage.py:190`
Problem: `rel = Path(module_rel).as_posix().lstrip('./')`. For a repo-root module
(`module_rel == '.'`), `Path('.').as_posix()` is `'.'` and `lstrip('./')`
collapses it to `''`, so downstream `startswith`/`==` checks compare against an
empty path. (`lstrip('./')` is also a character-set strip — see [G3].)
Fix hint: use `removeprefix('./')` and special-case `'.'` before comparison.

### PR-2 — Multi-target generator robustness (`marketplace/targets/`)

**Scope:** Correctness of the Claude and OpenCode emitters and their equality
check. Affects generated distribution output, not the source of truth.

**[G1] Claude emitter's `rmtree` has no containment guard** — *High* ·
`marketplace/targets/claude/emitter.py:96-97`
Problem: `emit_bundle_verbatim` does `if dest_root.exists(): shutil.rmtree(dest_root)`
with no check that `dest_root` is inside the resolved `--output` dir. If
`--output` is ever pointed at (a tree containing) `marketplace/bundles`, a real
source bundle is destroyed. The OpenCode emitter added `_safe_rmtree(path,
output_dir)` (emitter.py:100, used at 112) precisely to refuse deletion outside
`output_dir`; the Claude emitter lacks the equivalent. *Verified:* bare
`shutil.rmtree` in Claude vs guarded helper in OpenCode.
Note: the emitter docstring argues the wipe is safe because `target/claude/` is
gitignored/regenerable — true for the *intended* destination, but the guard is
defense-in-depth against a mistyped `--output`, and closes the inconsistency
with OpenCode.
Fix hint: resolve `dest_root` and assert it is strictly within a resolved
`output_dir` before `rmtree` (reuse/share `_safe_rmtree`).

**[G2] OpenCode emitter never clears stale output** — *Medium* ·
`marketplace/targets/opencode/emitter.py:411` (`emit_bundles`)
Problem: `emit_bundles` only `mkdir(exist_ok=True)` and rewrites files in place;
it never prunes. A skill/agent/command removed from source leaves its previously
emitted `skill/{bundle}-{skill}/`, `agent/*.md`, or `command/*.md` behind, so
OpenCode output drifts past the source. The Claude emitter wipes each bundle
destination first (emitter.py:96) to avoid exactly this.
Fix hint: clear (guarded) the relevant `skill/`/`agent/`/`command/` subtrees at
emit start, or track written paths and prune leftovers.

**[G3] `lstrip('./')` used as a prefix-strip corrupts refs** — *Medium* ·
`marketplace/targets/opencode/emitter.py:316,332`; `.../opencode/plugin_discover.py:243,249,278,283,312,317`
Problem: `str(ref).lstrip('./')` treats `'./'` as a *character set*, stripping
every leading `.` and `/`. `'./.hidden/x'` becomes `'hidden/x'`; `'../x'`
collapses to `'x'` (traversal silently mangled rather than rejected). Same
anti-pattern recurs (see also [R10]).
Fix hint: replace every `lstrip('./')` with `removeprefix('./')`; reject or
normalize `..` segments explicitly.

**[G4] OpenCode frontmatter parser terminates on `---` inside a value** — *Medium* ·
`marketplace/targets/opencode/frontmatter.py:104`
Problem: `end = content.find('---', 3)` locates the closing fence by raw
substring, not a line-anchored `\n---\n`. A frontmatter value containing `---`
(a horizontal rule, or three hyphens as an em-dash) ends the block early,
dropping later fields — which then trips required-field validation or silently
loses data. The Claude variant emitter (variant_emitter.py:102) correctly
anchors on `\n---\n`.
Fix hint: match the closing fence line-anchored, as the variant emitter does.

**[G5] Per-bundle `plugin.json` read crashes the equality check on corrupt JSON** — *Medium* ·
`marketplace/targets/claude/equality_check.py:115`
Problem: `_read_emitted_plugin_json` calls `json.loads(...)` with no error
handling. In validate mode a corrupt emitted `plugin.json` raises
`JSONDecodeError`, which propagates through `run_equality_check` and crashes the
CLI instead of returning the documented "re-run emit" diagnostic. The
`marketplace.json` path (line 95) is already guarded.
Fix hint: wrap the read in `try/except JSONDecodeError` and surface it as a
failing `EqualityResult` with a structured summary.

**[G6] `_load_mapping` LRU-caches by path, ignoring on-disk changes** — *Low* ·
`marketplace/targets/claude/variant_emitter.py:301`
Problem: `@lru_cache(maxsize=8)` keys purely on the `Path`. In a long-lived
process (tests generating different `mapping.json` at the same path, or a
watch/regen loop) a modified file is not re-read; the docstring's `cache_clear()`
mitigation doesn't cover same-path content changes.
Fix hint: key the cache on `(path, st_mtime_ns)`, or enforce `cache_clear()` at
each generation entry point.

**[G7] Orphan-diff detection can double-count one root cause** — *Low* ·
`marketplace/targets/claude/equality_check.py:184`
Problem: `_diff_array` builds `declared = set(committed.get(subdir, []))` from the
emitted manifest and flags `on_disk - declared`. When source and manifest go out
of sync, this can report orphans while the manifest-drift layer above reports the
same cause, producing two overlapping `BundleDiff` entries and an inflated
failure count.
Fix hint: de-duplicate the two detection layers, or annotate orphan diffs
distinctly so the count reflects distinct causes.

### PR-3 — Test-suite integrity

**Scope:** Remove false-confidence patterns and close the direct-coverage tail.
This PR strengthens the safety net that PR-1/PR-2 fixes will rely on; landing it
early is valuable even though it is independent.

**[T1] `run-tests.py` reports "passed" for tests that execute zero tests** — *High* ·
`test/run-tests.py:96-119`
Problem: the runner executes each file as a script
(`subprocess.run([sys.executable, str(test_file)])`) and treats exit 0 as pass.
Only ~10 of ~545 `test_*.py` files have a `__main__` block that invokes pytest;
the other ~535 pytest-function-style files merely import and exit 0 having run
**zero** test functions — yet print `✓ passed`. Pure false-green. The canonical
CI path is pytest via `build.py`, so this is a standing *developer* trap, not a
CI hole. *Verified:* direct-execution + `returncode == 0` at lines 115-119.
Fix hint: shell out to `pytest <file>` per file, or delete `run-tests.py` in
favor of the `module-tests` build command.

**[T2] Config-content assertions silently skip when the config is missing** — *Medium* ·
`test/plan-marshall/workflow-shared/test_config_validation.py:77-78,90-91`
Problem: `test_required_keys_present` and `test_non_empty_arrays` do
`if not config_path.exists(): pytest.skip(...)`, so deleting/renaming a config
turns the meaningful checks into silent no-ops. Only `test_config_parses_as_valid_json`
(line 64) hard-asserts existence.
Fix hint: replace the skip guards with `assert config_path.exists()` to match the
parse test.

**[T3] Fixture-presence guards convert assertions into skips across plugin-doctor tests** — *Medium* ·
`test/pm-plugin-development/plugin-doctor/test_analyze.py:78-79,90-91,…,229-230` (20+ sites)
Problem: nearly every structure/coverage test opens with
`if not test_dir.exists(): pytest.skip('fixture not available')`. Any fixture
rename/removal converts the assertion into a silent pass; one guard even reads
`'…(depends on rename tasks completing earlier)'`, baking order/history
dependence into a skip.
Fix hint: `assert test_dir.exists()` instead of skipping; mark genuinely-optional
cases with an explicit decorator so they surface in the summary.

**[T4] Time-boundary tests depend on real `time.sleep` and the wall clock** — *Medium* ·
`test/plan-marshall/manage-metrics/test_manage_metrics_record_dispatch_boundary.py:155,159`;
`test/plan-marshall/plan-retrospective/test_compile_report.py:225`
Problem: these `time.sleep(1.05)`/`(1.1)` because the SUT distinguishes
records/filenames only at second granularity. They add wall-clock time and the
compile-report test's only distinguishing assertion
(`data_a['output_path'] != data_b['output_path']`) is purely a function of the
clock — it would flake if timestamp resolution changed and it masks a same-second
filename-collision design in the SUT.
Fix hint: monkeypatch the SUT clock / `datetime.now` to inject deterministic
increasing timestamps; assert ordering on the injected values.

**[T5] Direct unit tests missing for 17 helper modules** — *Medium* ·
`marketplace/bundles/*/skills/*/scripts/_*.py` (coverage cross-map)
Problem: skill-level coverage is dense and **no whole skill directory is
untested**, but 17 individual helper modules are never referenced by name in any
test (reachable only transitively, if at all). Largest: `_cmd_cleanup.py` (331
lines, `manage-run-config`), `_lessons_crud.py` (376), `_pyproject_cmd_discover.py`
(352); also the two `manage-providers` credential ops `_cred_list.py` /
`_cred_remove.py`. Their branches can regress without a failing test.
Fix hint: add direct unit tests (via `load_script_module`) for the unreferenced
helpers, or confirm+document that each is fully exercised through its parent
script's subcommand tests. (Full 17-item list captured in the analysis; prioritize
the four largest and the credential ops.)

**[T6] Dead `sys.modules.setdefault(...)` mocks that mock nothing** — *Low* ·
`test/plan-marshall/build-npm/test_npm_execute.py:13-14`; `.../build-maven/test_maven_execute.py:13`;
`.../build-gradle/test_gradle_execute.py:13`; `.../build-pyproject/test_pyproject_execute.py:30`;
`.../script-shared/test_build_config_contract.py:17`
Problem: these `sys.modules.setdefault('plan_logging', MagicMock(...))`, but
`conftest.py:187` pre-imports the real `plan_logging`/`run_config` first, so
`setdefault` is a guaranteed no-op — the tests run against the real modules while
implying isolation they don't have.
Fix hint: delete the `setdefault` lines, or switch to an explicit
`monkeypatch`/`sys.modules[...] =` fixture if stubbing is genuinely intended.

**[T7] Developer-specific absolute paths (and a username) baked into fixtures** — *Low* ·
`test/pm-dev-java/fixtures/sample-build-{warnings,errors,success}.log` (`/Users/oliver/…`);
`test/plan-marshall/build-npm/fixtures/log-test-data/*.log` (`/Users/dev/…`)
Problem: parser fixtures embed a specific user's absolute paths, leaking a real
username and coupling any future path-relative assertion to one machine's layout.
Fix hint: normalize fixture paths to a placeholder root (e.g. `/work/project`).

**[T8] Autouse pollution guard walks the real filesystem on every test** — *Low* ·
`test/conftest.py:558-611` (`_pollution_guard`, autouse) with `:481-490`, `:518-555`
Problem: an autouse fixture snapshots the real credentials dir (`rglob('*')`) and
`.plan/local/` before and after *every* test, purely as a backstop to sandbox
fixtures that already make the writes structurally impossible — real cost, low
marginal value, and it reads real developer state. (Already narrowed once for an
O(100k+) regression, per its docstring.)
Fix hint: scope the guard to credential/plan-touching tests via a marker/opt-in
fixture rather than autouse-on-everything.

**[T9] `PlanContext.__exit__` mutates shared process env and module globals** — *Low* ·
`test/conftest.py:924-976` (esp. `os.environ` save/restore `:962-972`, `_config_core` `:928-938`)
Problem: `PlanContext` manipulates process-global `os.environ` and `_config_core`
attributes, coexisting with newer autouse monkeypatch sandboxes that do the same
auto-reverted. Two overlapping mechanisms mutating the same globals is a
shared-state hazard: an exception between enter/exit, or interleaving with the
autouse fixture, can leave `PLAN_BASE_DIR` pointing at a torn-down temp dir for
later tests in the same worker.
Fix hint: migrate remaining `PlanContext`/`BuildContext` users to the
`plan_context` monkeypatch fixture and retire the manual save/restore path.

### PR-7 — CI, packaging & supply-chain hardening

**Scope:** `.github/workflows/**` and build/packaging config. The two
security-relevant items ([B1] injection, [B2] missing permissions) should be
prioritized within this cluster.

**[B1] Template injection via `github.ref_name` in a `run:` block** — *Medium* ·
`.github/workflows/claude-distribute.yml:71-79`
Problem: the "Create dist tag" step interpolates `${{ github.ref_name }}` directly
into a bash `run:` block that runs `git fetch`/`git tag` in a `contents: write`
context. Ref/tag names can contain shell metacharacters — a classic
template-injection surface.
Fix hint: pass `github.ref_name`/`github.sha` via an `env:` block and reference
`"$REF_NAME"` in the script instead of interpolating context into the shell.

**[B2] `opencode-generate-check.yml` declares no `permissions:` block** — *Medium* ·
`.github/workflows/opencode-generate-check.yml:9-12`
Problem: with no `permissions:` block the `GITHUB_TOKEN` inherits the
repo/org default (potentially read-write) for a job that only runs a generator —
violates least privilege.
Fix hint: add top-level `permissions: contents: read`.

**[B3] No `CODEOWNERS` file** — *Medium* ·
`.github/` (absent)
Problem: there is no `CODEOWNERS`. If the branch-protection ruleset enables
"Require review from Code Owners" (common alongside the `verify / verify`
requirement), the rule matches no owners and provides no enforced review routing.
Fix hint: add `.github/CODEOWNERS` mapping `marketplace/`, `.github/`, etc. to
maintainers — or confirm the ruleset does not depend on code owners.

**[B4] `uv.lock` locks zero dependencies while the build relies on `uv run`** — *Medium* ·
`uv.lock` vs `build.py:85,101,116,181` and `pyproject.toml:14-23`
Problem: tool deps (mypy, ruff, pytest, coverage) are declared under
`[tool.pyprojectx.main]`, not `[project.dependencies]`, so `uv.lock` contains only
the virtual root and pins nothing. Yet `build.py` invokes `uv run mypy/ruff/pytest`,
which resolve against the empty project env — it works only because pyprojectx's
separate venv (`pw.lock`) puts the tools on PATH. Two uncoordinated pin layers.
Fix hint: either add the tools to a dependency group and lock them in `uv.lock`,
or document that `uv.lock` is intentionally empty and `pw.lock` is the sole tool-pin source.

**[B5] `claude-distribute` grants workflow-wide `contents: write` unnecessarily** — *Low* ·
`.github/workflows/claude-distribute.yml:14-16`
Problem: workflow-level `contents: write` applies to every step, but pushes
actually use a GitHub App token / `personal_token`; the default `GITHUB_TOKEN`'s
write scope is unnecessary breadth for checkout/generate.
Fix hint: default to `contents: read`; grant `contents: write` only on the steps
that need it (or rely solely on the app token).

**[B6] Path-filtered check can wedge if ever made a required status** — *Low* ·
`.github/workflows/opencode-generate-check.yml:3-8`
Problem: the job runs only when `marketplace/bundles/**` or `marketplace/targets/**`
change. If added to required checks, PRs not touching those paths would sit in
"Expected — Waiting" forever.
Fix hint: if it must be required, split into a non-path-filtered job that emits a
neutral/success skip; otherwise keep it non-required and document that.

**[B7] `python-verify` grants no write scope for coverage/annotations** — *Low* ·
`.github/workflows/python-verify.yml:9-11`
Problem: `permissions` grants only `contents: read` + `pull-requests: read`. If
the reusable `reusable-pyprojectx-verify.yml` posts coverage summaries or check
annotations, it lacks `pull-requests: write` / `checks: write` and silently
degrades.
Fix hint: confirm the reusable workflow's needs; add the write scopes only if required.

**[B8] Duplicate CI runs for canonical-prefix branches** — *Low* ·
`.github/workflows/python-verify.yml:4-7`
Problem: for a `feature/*`/`fix/*`/`chore/*` branch with an open PR to `main`,
both `push` (branch matches) and `pull_request` (base is main) fire, running the
full verify suite twice per push.
Fix hint: drop the `push` trigger for those prefixes (keep `main`), or add a
concurrency group to cancel the redundant run.

**[B9] Vendored `pw` wrapper has a malformed uv-install fallback** — *Low* ·
`pw:211`
Problem: the non-Windows uv-install fallback builds
`curl … -LsSf irm {url}/uv-installer.sh | sh` — a stray `irm` (PowerShell's
Invoke-RestMethod) is spliced into the curl line, breaking that path. Only bites
when `venv` is unavailable / `PYPROJECTX_USE_UV` is set.
Fix hint: this is upstream pyprojectx wrapper code — regenerate via `pw --upgrade`
to a fixed release, or correct the line.

**[B10] Three divergent private-contact channels; verify the security mailbox is live** — *Low* ·
`SECURITY.md:13` vs `LICENSE.md:12`, `.github/ISSUE_TEMPLATE/config.yml:4`
Problem: `SECURITY.md` routes vulnerability email to `contact@cuioss.de`,
commercial-licensing goes to a Tally form, and LICENSE points to GitHub issues —
three mechanisms for adjacent private-contact purposes invite confusion; confirm
`contact@cuioss.de` is monitored for security intake.
Fix hint: cross-reference the private channels consistently and confirm the
security mailbox is live.

**[B11] No PR template** — *Low (optional)* ·
`.github/` (no `pull_request_template.md` / `PULL_REQUEST_TEMPLATE/`)
Problem: tooling and `CONTRIBUTING.md` assume a PR template may exist; none does,
so PRs get no structured description scaffold.
Fix hint: add `.github/pull_request_template.md` if a standardized description is desired.

## 4. Sequencing & Dependencies

The PRs are independent (no hard code dependencies), but a sensible order
maximizes safety and reviewer focus:

1. **PR-3 (test integrity)** first — landing the assertion-hardening and killing
   the false-green runner gives the correctness PRs a trustworthy safety net.
2. **PR-1 (runtime correctness)** and **PR-2 (generator robustness)** next — the
   highest user-facing risk. These can proceed in parallel; each fix carries its
   own regression test.
3. **PR-4 (skill references)** — quick, machine-consumed correctness; low risk.
4. **PR-7 (CI/supply-chain)** — prioritize [B1]/[B2] (security-relevant) even
   though the cluster is otherwise low-urgency.
5. **PR-5 (READMEs)** and **PR-6 (governance docs)** last — doc-only, no runtime
   effect. [D1]/[D2]/[D3] carry a mild elevation because they hide *security*
   skills.

PR-5, PR-6 are small doc/reference edits; PR-2 and PR-7 are medium.
