Operational guidance for the per-session terminal-title indicator that Plan Marshall configures via `/marshall-steward` → Configuration → Terminal title.

The wizard wires hooks and a `statusLine` entry into `./.claude/settings.local.json` so each Claude Code session tab shows a process icon — `➤` (active), `?` (waiting), `✓` (done), or `✅` (plan complete/archived) — optionally a lock glyph (`⏳` lock-waiting, `🔒` lock-owned), followed by the active plan and phase. It also sets `env.CLAUDE_CODE_DISABLE_TERMINAL_TITLE=1` to stop Claude Code's built-in OSC emitter from clobbering the title on every render cycle.

The canonical icon palette, glyph vocabulary, and the full three-way (state / composer / resolve+emit) architecture live in xref:../../marketplace/bundles/plan-marshall/skills/manage-terminal-title/standards/terminal-title-architecture.md[terminal-title-architecture.md].

iTerm2, Ghostty, Kitty, and Terminal.app pick up the OSC-set title automatically and need no additional configuration. **VS Code does not** — its integrated terminal ignores OSC title sequences unless you opt in. **JetBrains IDEs do not either** — and unlike VS Code, no client-side setting fixes it.

== Scope

Installation is project-local and per-developer. The wizard writes exclusively to `./.claude/settings.local.json`, which is gitignored by Claude Code convention so absolute per-user paths never enter version control. There is no global-scope option and no wizard path to uninstall — removing the feature is a manual edit of the file.

== VS Code

=== One-time user-settings change

Open VS Code user settings (`Cmd+,` → "Open Settings (JSON)" icon, or edit `~/Library/Application Support/Code/User/settings.json` directly) and add:

[source,jsonc]
----
"terminal.integrated.tabs.title": "${sequence}"
----

This is a VS Code-owned setting; the wizard cannot patch it because VS Code's user settings and Claude Code's `settings.local.json` are distinct files owned by different tools.

=== Why this is needed

VS Code's default tab-title template is `"${process}"` — the tab shows the running process name (`zsh`, `node`, `claude`) regardless of OSC escapes received from the shell. The `${sequence}` placeholder switches the template to use whatever was set via OSC 0/2, i.e. the title our hook emits.

If you have Settings Sync enabled, the change propagates across all your VS Code installs. Workspace-level `.vscode/settings.json` files that override `terminal.integrated.tabs.title` still win for that workspace only.

=== Verification

1. Restart VS Code (or reload the window) so the setting takes effect.
2. Open a fresh terminal tab whose working directory resolves to a plan worktree under `.plan/local/worktrees/<plan-id>/`, or `export PLAN_ID=<plan-id>` for a non-worktree shell.
3. Run `claude`. The tab should flip through `➤ <plan>:<phase>` → `?` → `✓` across a session turn.
4. Confirm Claude Code's version string (e.g. `2.1.114`) never reappears in the tab — that indicates `CLAUDE_CODE_DISABLE_TERMINAL_TITLE=1` is in effect.

=== Troubleshooting

* **Tab still shows the process name** — the user-settings edit didn't stick. Verify the key is present in `~/Library/Application Support/Code/User/settings.json` and no workspace override is masking it (`Cmd+Shift+P` → "Preferences: Open Workspace Settings (JSON)").
* **Tab shows `2.1.114` intermittently** — `env.CLAUDE_CODE_DISABLE_TERMINAL_TITLE` is missing or set to a non-`"1"` value in `./.claude/settings.local.json`. Re-run `/marshall-steward` → Configuration → Terminal title, or add the key manually under the `env` block.
* **Tab label is blank inside Claude Code** — the hook subprocess couldn't open `/dev/tty` and fell back to stdout (e.g. `➤ claude` appears literally); check `~/.claude/plugins/cache/plan-marshall/plan-marshall/<version>/skills/plan-marshall/scripts/set_terminal_title.py` is on disk and executable by your Python 3.

== JetBrains IDEs (IntelliJ IDEA, PyCharm, GoLand, WebStorm, RustRover, Rider, …)

JetBrains IDEs ship a built-in terminal tool window backed by JediTerm. Unlike VS Code, **there is no setting, registry flag, or shell hook that makes the JetBrains terminal tab title reflect OSC 0/2 title escape sequences.** The `/marshall-steward` wizard configures everything correctly, the script emits the OSC bytes correctly, JediTerm parses them correctly — and then the IDE discards them for tab-title purposes.

=== Why it cannot be fixed from our side

JediTerm consumes the `\033]0;...\007` sequence and stores the title on the internal `Terminal` session object. The visible tab name in the Terminal tool window, however, is owned by the IDE: the classic terminal renders it as `<shell>` / `<cwd>`, and the Reworked Terminal (default since 2024.3) renders it as the most recent command. Neither code path reads JediTerm's stored OSC title. The long-standing JetBrains tickets https://youtrack.jetbrains.com/issue/IDEA-156492[IDEA-156492] and https://youtrack.jetbrains.com/issue/IJPL-92196[IJPL-92196] track this limitation; until JetBrains resolves them, no client-side workaround can change tab titles in the JetBrains terminal.

=== What still works

The `statusLine` entry the wizard writes into `./.claude/settings.local.json` is the supported source of truth inside JetBrains IDEs. Claude Code renders the statusline as plain text in its TUI footer, so the same `<icon> pm:<phase>:<short>` label that drives the tab title elsewhere shows up at the bottom of the Claude pane regardless of terminal emulator. Look there for the live status when running `claude` inside the JetBrains terminal.

=== Optional workarounds

* **Rename the tab manually.** Right-click the Terminal tool window tab → `Rename Session…` and type the plan id. The name is static (won't update across phases) but at least disambiguates multiple plans open in parallel.
* **Run `claude` in an external terminal.** Use a standalone terminal emulator (iTerm2, Ghostty, Kitty, Terminal.app, etc.) and start `claude` there. The OSC-driven tab title works out of the box in those terminals.
* **Watch the statusline.** For most workflows the in-pane statusline is sufficient and you don't need the tab title at all.

=== Troubleshooting

* **Statusline missing inside Claude Code** — confirm `settings["statusLine"]` exists in `./.claude/settings.local.json` and points at the script with the `--statusline` flag. Re-run `/marshall-steward` → Configuration → Terminal title if absent.
* **OSC bytes leak as visible text in the terminal** (e.g. `]0;➤ claude` appearing as output) — the script ran fine; what failed is the `/dev/tty` write. The script then falls back to stdout, Claude Code captures that stdout as hook output, and the IDE renders the bytes literally because they never reached JediTerm as an in-band escape sequence. This is cosmetic and harmless — the tab title couldn't update anyway (see "Why it cannot be fixed from our side" above). If you want to silence the leak, run `claude` from a tty session where the hook subprocess can open `/dev/tty` directly, e.g. an external terminal.
