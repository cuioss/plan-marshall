Operating Plan Marshall with several plans in flight at once. Concurrent plans contend for two shared resources — the merge-to-`main` boundary and the build machine — and Plan Marshall serializes both with cooperative, main-anchored locks. This page covers the operator-tunable knobs, where the lock events are logged, and what to do when the build queue looks stuck. For how the locks work, see xref:../concepts/parallelism-and-locking.adoc[Concepts › Parallelism and Locking].

== Knobs

Three knobs govern build concurrency and stale-slot recovery.

The build-queue concurrency knobs live in `marshal.json`:

* `build.queue.max_slots` (default 5) — the maximum number of builds that may run concurrently across all sessions. A build that finds a free slot runs immediately; otherwise it is enqueued and promoted in FIFO order as slots free up.
* `build.queue.max_retries` (default 10) — how many times a waiting build's wrapper retries acquiring a slot before giving up. Raise it on a busy machine where builds routinely wait through several promotions.

The stale-slot threshold lives in the main-anchored `run-configuration.json` and is managed with the `manage-run-config build-queue-limit` verbs from xref:../concepts/parallelism-and-locking.adoc[the self-healing reaper]:

* `build.queue.upper_limit_seconds` (default and floor 600 s, ceiling 3600 s) — the adaptive estimate of the longest real build. The reaper treats an active slot older than `2 ×` this value as dead. The value is monotonic-up and clamped to `[600 s, 3600 s]`, so it grows toward the longest build observed but never ratchets past an hour. Read or set it explicitly:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:manage-run-config:run_config build-queue-limit get
python3 .plan/execute-script.py plan-marshall:manage-run-config:run_config build-queue-limit set --value 900
----

A `set` value below 600 is floored to 600; a value above 3600 is clamped to 3600; a non-positive value is rejected.

== The [LOCK] Log

Both locks emit `[LOCK]` lifecycle events — acquire, blocked, release, and the reaper's stale-reclaim — but they do *not* write to a plan's per-worktree work-log. Lock contention spans sessions, so the events go to a *single main-anchored global log* shared by every session, alongside Plan Marshall's other global logs under the main checkout's `.plan/`. Each line carries the lock id and, on contention, the holder and the waiter, so you can reconstruct who held what and for how long.

When two plans appear to serialize against each other — one waiting while another holds the merge mutex or a build slot — the main-anchored `[LOCK]` log is the place to look. Grep it for the waiting plan's id to find the `blocked` event and the `holder` it names.

== FAQ

=== No more builds are accepted / the queue appears stuck

When new builds stop being admitted, work through these steps in order:

1. *Inspect the `[LOCK]` events.* Open the main-anchored global lock log and find the active build ids and their `acquired` timestamps. A slot held far longer than a real build takes is the likely culprit; the `held` duration on a later `reaped-stale` event confirms it.

2. *Let the self-healing reaper clear it.* The reaper runs at the start of every acquire and every release, so the next queue operation auto-clears any active entry older than `2 ×` the adaptive `build.queue.upper_limit_seconds` and emits a `reaped-stale` `[LOCK]` event. In most cases simply triggering another build (or releasing one) frees the wedged slot without any manual action.

3. *Edit `build-queue.json` as a last resort.* If a slot must be freed immediately and no queue operation is imminent, inspect the main-anchored `.plan/local/build-queue.json` to identify the stale active entry, then remove that entry from the `active` list. Do this only when you are certain the holding build is genuinely dead — removing a live build's entry lets a second build claim its slot.
