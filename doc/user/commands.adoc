User-invocable slash commands provided by the marketplace.

== /marshall-steward

Project configuration wizard for the planning system.

[source,bash]
----
/marshall-steward           # Interactive menu, or first-run wizard if .plan/execute-script.py is missing
/marshall-steward --wizard  # Force first-run wizard
----

See xref:getting-started.adoc[Getting Started] for the first-run wizard steps. The interactive menu offers:

* **Maintenance** — regenerate the executor (after `/plugin install` / `/plugin uninstall`), clean execution logs.
* **Health Check** — verify setup, diagnose missing scripts or stale configuration.
* **Configuration** — adjust build systems, skill domains, the per-role effort/model registry (see xref:configuration.adoc[Configuration]), terminal-title integration (xref:terminal-title.adoc[Terminal Title]), and other project-level knobs.

== /plan-marshall

Unified plan lifecycle management — create, list, outline, execute, finalize, and clean up plans. The skill drives all six phases (`phase-1-init` … `phase-6-finalize`) through the xref:../concepts/execution-context.adoc[execution-context dispatcher]. See xref:../concepts/planning-workflow.adoc[Planning Workflow] for the phase sequence and boundaries.

=== Common calls

[source,bash]
----
/plan-marshall                                                  # List active plans (interactive menu)
/plan-marshall task="Add user authentication"                   # Create new plan from a task description (action=init implied)
/plan-marshall issue="https://github.com/org/repo/issues/42"    # Seed from a GitHub issue (action=init implied)
/plan-marshall action=outline plan="user-auth"                  # Run outline + plan phases for an existing plan
/plan-marshall action=execute plan="jwt-auth"                   # Resume execution of an existing plan
/plan-marshall action=finalize plan="jwt-auth"                  # Run phase-6-finalize (commit, push, PR)
/plan-marshall plan="jwt-auth"                                  # Continue an existing plan from its current phase
/plan-marshall action=cleanup                                   # Remove completed plans
/plan-marshall action=lessons                                   # List and convert lessons learned
/plan-marshall lesson="2026-05-18-11-001"                       # Convert a specific lesson to a plan (action=lessons implied)
/plan-marshall action=lessons-aggregate                         # Cross-lesson aggregation + superseded-stub prune
/plan-marshall recipe="refactor-to-standards"                   # Create plan from a predefined recipe (action=recipe implied)
/plan-marshall recipe="code-review"                             # On-demand diff-aware structural/quality review (emits lint-issue findings)
/plan-marshall recipe="security-audit"                          # On-demand security audit (emits bug/anti-pattern findings)
----

=== Parameters

[cols="1,1,3", options="header"]
|===
| Parameter | Type | Description

| `action` | optional | One of `list` (default), `init`, `outline`, `execute`, `finalize`, `cleanup`, `lessons`, `lessons-aggregate`, `recipe`. When omitted, the action is inferred from other parameters — see <<action-resolution>>.
| `task` | optional | Free-form task description used to create a new plan. Implies `action=init` when no explicit action is given.
| `issue` | optional | GitHub issue URL used to seed a new plan. Implies `action=init` when no explicit action is given.
| `lesson` | optional | Lesson ID to convert into a plan. Implies `action=lessons` when no explicit action is given.
| `recipe` | optional | Recipe key (e.g. `refactor-to-standards`). Implies `action=recipe` when no explicit action is given.
| `plan` | optional | Plan **name** (plan_id) for operations on an existing plan — not a path. When supplied without an explicit action, the action is auto-detected from the plan's current phase.
|===

[#action-resolution]
=== Action resolution

When `action=` is omitted, the effective action is resolved in this order — first match wins:

1. **`plan=` is supplied** → auto-detect from plan's current phase.
2. **`task=` or `issue=` is supplied** → `init` (the supplied value becomes the plan source).
3. **`lesson=` is supplied** → `lessons`.
4. **`recipe=` is supplied** → `recipe`.
5. **No source/target parameters at all** → `list` (interactive menu).

If two or more of `{task, issue, lesson, recipe, plan}` are supplied together without an explicit `action=`, the skill returns `status: error` naming the conflict. Pass an explicit `action=` or drop the conflicting parameter to resolve.

=== Recipe-match auto-routing of informal requests

You do not have to name a recipe to use one. When you create a plan from an informal description — `task="…"` or `issue=…` — `phase-1-init` scores the request against the registered recipe set and may *auto-route* it into a matching recipe's known-shape path, skipping the standard refine/outline discovery. This is the same Tier 1 recipe-match routing described in xref:../concepts/recipes.adoc[Concepts › Recipes].

Two configurable knobs govern the behaviour (full reference in xref:configuration.adoc#recipe-routing[Configuration › Recipe-match routing]):

* `plan.phase-1-init.auto_route_recipe` (default `true`) — when a request's top recipe match clears the confidence floor, route into it without prompting. Set to `false` to always be shown the ranked matches and choose.
* `plan.phase-1-init.auto_route_recipe_threshold` (default `0.6`) — the confidence floor a top match must clear to auto-route.

A request labelled analysis- or planning-only by the companion `aspect-classify` step has its build / quality-gate / test verification steps dropped from the execution manifest. An explicit `recipe=` selection always overrides recipe-match — naming a recipe is never second-guessed.

=== Audit recipe parameters

The audit recipes — `recipe=code-review` and `recipe=security-audit` — run a diff-aware review over the current footprint and emit findings into the triage pipeline (see xref:../concepts/recipes.adoc[Concepts › Recipes]). Both accept two optional parameters that bound the footprint:

[cols="1,1,3", options="header"]
|===
| Parameter | Type | Description

| `scope` | optional | Path or file restriction. When supplied, the review footprint is bounded to these paths. When omitted, the footprint is the full current-branch-vs-base diff.
| `base_branch` | optional | Explicit base ref for the diff. When omitted, defaults to the configured base branch (falling back to `main`).
|===

With neither parameter, both recipes audit the full current-branch-vs-base diff.

=== Personas

You never pass a persona on the command line — persona selection is automatic. Each task is dispatched **as** a named identity (a *persona*) resolved from the work it does: an implementation task runs as the Implementer, a module-testing task as the Module Tester, a security audit as the Security Expert. The persona determines which skills enter the subagent's context; the command surface stays the same regardless.

Where the binding comes from depends on the dispatch:

* **Plan tasks** derive their persona from the task's profile — no declaration needed.
* **Entry points** (recipes, finalize steps, and the slash commands themselves) declare one persona explicitly, so `/plan-marshall recipe=…` and the finalize phase each run under the identity their definition names.

Personas surface to you as the *names* in logs and status output ("the Auditor persona", "the Security Expert persona") rather than as a knob you set. For the full model — the persona / ref / profile archetypes, the composition resolver, and how a persona differs from the model-and-effort role/level — see xref:../concepts/personas.adoc[Concepts › Personas].
