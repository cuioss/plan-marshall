How Plan Marshall is built and tested locally.

== Overview

This project uses https://pyprojectx.github.io/[Pyprojectx] for Python development tooling, following the Maven/Gradle wrapper pattern. Only Python 3 is required on the host system to bootstrap the wrapper; the interpreter the build actually runs on is pinned independently of it, by `uv`. See <<Which Interpreter Runs the Build>>.

== Rationale

Pyprojectx was chosen because it provides a proven, mvnw/gradlew-style wrapper for Python that requires only Python 3 on the host system. Unlike custom solutions, it is actively maintained (since 2022), supports multiple tools (uv, ruff, mypy, pytest), and provides reproducible builds via lock files. This approach avoids reinventing the wheel while matching industry best practices for 2025.

== Canonical Commands

[IMPORTANT]
====
Never hard-code `./pw` in workflow or skill code. Always resolve build commands via the architecture API:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:build-pyproject:pyproject_build run --command-args "{command}"
----

The `./pw` shortcuts below are appropriate for ad-hoc local development only.
====

[source,bash]
----
# Canonical commands with optional module filtering (see build.py)
./pw compile                      # mypy marketplace/bundles/
./pw compile pm-dev-frontend      # mypy single bundle
./pw test-compile                 # mypy test/
./pw test-compile plan-marshall   # mypy single test directory
./pw module-tests                 # pytest test/
./pw module-tests pm-dev-frontend # pytest single directory
./pw quality-gate                 # ruff check all
./pw quality-gate pm-dev-java     # ruff on single bundle + tests
./pw coverage                     # pytest with coverage
./pw verify                       # Full: quality-gate + test-compile + module-tests
./pw verify pm-dev-frontend       # Full verification on single bundle
./pw clean                        # Remove build artifacts
----

Always call build commands with a Bash timeout of at least 10 minutes (600000ms) when running them from inside a Plan Marshall workflow. The build emits TOON; analyse `status`, `errors[N]{file,line,message,category}`, and `log_file` as needed.

== Quality-Gate Coverage

The `quality-gate` build target (root `build.py:cmd_quality_gate`) is the fast-feedback stage developers run on every iteration. It enforces the following marketplace-wide invariants in seconds, so violations are caught before push rather than at CI time:

[cols="1,2,2", options="header"]
|===
| Stage | Tool / Rule | Scope

| Type-check
| `mypy`
| Production sources (`marketplace/bundles`, `.claude` when present)

| Lint
| `ruff check`
| `marketplace/bundles`, `test`, `.claude` (full tree) or scoped paths (module run)

| Static-analysis invariants
| `pm-plugin-development:plugin-doctor:doctor-marketplace quality-gate`
| Marketplace-wide (full-tree run only)
|===

The plugin-doctor `quality-gate` subcommand runs only the rules whose violations are currently enforced as build-failing invariants by the pytest suite (real marketplace must produce zero findings):

* `scan_argparse_safety` — every `argparse.ArgumentParser(...)` and `subparsers.add_parser(...)` call must specify `allow_abbrev=False`. Prevents prefix-abbreviation matching from silently accepting truncated flags.
* `validate_extension_contracts` — extension-point implementations must declare the required contract sections (severity guidelines, acceptable-to-accept lists, etc.) per `plan-marshall:extension-api`.
* `analyze_argument_naming` — notation/subcommand/flag/canonical-forms cluster. Unconditionally active under `quality-gate` (build-failing invariant). The same cluster is *gated off by default* under the `analyze` subcommand and opt-in only via `--rules argument_naming` (or the `--enable-argument-naming` alias) on `plugin-doctor analyze`. The companion `verb_chain` cluster follows the same shape: opt in with `--rules verb_chain` or `--enable-verb-chain`.

Module-scoped quality-gate runs (`./pw quality-gate <module>`) skip the marketplace-wide plugin-doctor sweep because they target a single bundle.

*Coverage scope*: `quality-gate` does NOT run pytest tests. Invariants enforced by pytest fixtures (helper-classification rules, fixture-driven detection tests, integration-test contracts) still require `module-tests` or `verify`. New plugin-doctor static-analysis rules are picked up automatically once their `scan_*` / `validate_*` entry point is wired into `cmd_quality_gate` in `doctor-marketplace.py`; no per-rule registration in `build.py` is needed.

== Tool Configuration

Tool configuration is in `pyproject.toml`:

* `[dependency-groups].dev` — the dev toolchain (pytest, pytest-cov, pytest-xdist, pytest-timeout, ruff, mypy), resolved by uv into `uv.lock`
* `[tool.uv].default-groups` — declares `dev` as the group uv installs by default
* `[tool.pyprojectx.env]` — environment exported to every alias; carries `UV_PYTHON`, the build's interpreter pin
* `[tool.pyprojectx.main]` — Pyprojectx bootstrap requirements: `uv` and nothing else
* `[tool.pyprojectx.aliases]` — command shortcuts
* `[tool.pytest.ini_options]` — pytest configuration
* `[tool.coverage.run]` / `[tool.coverage.report]` — coverage data-collection + reporting settings. The HTML output directory (`.plan/temp/htmlcov/`) is wired in `build.py` via `--cov-report=html:…`, not in `pyproject.toml`.
* `[tool.ruff]` — linter/formatter configuration
* `[tool.mypy]` — type checker configuration

=== Which Interpreter Runs the Build

**`uv` pins the build interpreter, and `[tool.pyprojectx.env].UV_PYTHON` is the single declaration that sets it.** Local and CI resolve the same interpreter because they resolve it through the same tool from the same declaration.

==== Which tool owns what

Pyprojectx is the *entry point*; it is no longer the *tool environment*. `[tool.pyprojectx.main].requirements` lists `uv` and nothing else, so the venv Pyprojectx creates for itself contains no package that touches project code. `uv` owns everything downstream: it resolves `[dependency-groups].dev` into `uv.lock`, installs it into the project `.venv`, and selects that venv's interpreter.

This split matters because Pyprojectx *cannot* select an interpreter. Reading the installed Pyprojectx 3.3.2 sources: `pyprojectx/env.py::IsolatedVirtualEnv.install()` creates its venv with `uv venv --python {sys.version_info.major}.{minor}` — the version of the Pyprojectx process, i.e. whichever `python3` invoked `./pw` — and `pyprojectx/config.py` exposes no key that overrides it. That uncontrolled interpreter is now harmless: it runs only `uv`, which is a self-contained binary.

==== How the pin reaches every tool invocation

`[tool.pyprojectx.env]` is a map of environment variables that `pyprojectx/cli.py::_run_alias` merges into the environment of every alias, which `pyprojectx/env.py::IsolatedVirtualEnv.run()` layers over `os.environ`. Setting `UV_PYTHON = "3.12"` there means:

. Every `[tool.pyprojectx.aliases]` entry — all of which invoke `uv run` — resolves a 3.12 project `.venv`, downloading a uv-managed CPython 3.12 when the host has none.
. `build.py`'s own nested `uv run mypy` / `uv run ruff` / `uv run pytest` subprocesses inherit the variable, so they resolve the same `.venv`.

The observable check is the pytest header, which names both the version and the interpreter path:

[source]
----
platform darwin -- Python 3.12.3, pytest-9.1.1, pluggy-1.6.0 -- .../.venv/bin/python3
----

A `…/.venv/bin/python3` path reporting 3.12 means the pin held. A `.pyprojectx/venvs/main-<hash>-py<X.Y>` path would mean a tool escaped to the Pyprojectx venv — which is why every alias routes through `uv run` rather than calling a bare `ruff` or `pytest` from `PATH`.

==== CI

`.github/workflows/python-verify.yml` delegates to the SHA-pinned reusable workflow `cuioss/cuioss-organization/.github/workflows/reusable-pyprojectx-verify.yml@6278bbd7280a6e94b4914aa9e8c61cf23ea1abbe`, whose `actions/setup-python` step is conditional: it runs only when a Python version is supplied, through either the reusable workflow's `python-version` input or the `pyprojectx.python-version` key of `.github/project.yml`. This repository sets neither, so `setup-python` is skipped and `./pw` bootstraps on the runner's default `python3`.

That is deliberate and sufficient. `setup-python` would pin only the *bootstrap* interpreter — the one that runs Pyprojectx, which now installs nothing but `uv` — while `UV_PYTHON` pins the interpreter the tools actually run on. Adding a `pyprojectx.python-version` key would therefore introduce a second interpreter declaration that governs a layer whose version no longer matters, inviting exactly the misreading this section previously documented: that the Pyprojectx venv is where the build happens. The single pin is the honest one.

==== The related declarations

* `[tool.pyprojectx.env].UV_PYTHON` — **the build interpreter.** The only declaration that selects it.
* `[project].requires-python` — the *minimum* interpreter the project supports; a consumer-facing support claim and a resolution floor, not a pin.
* `[tool.mypy].python_version` — the language level mypy type-checks against, independent of the interpreter mypy itself runs on.

`[tool.pyprojectx].lock-python-version` was removed. It reached only `pyprojectx/lock.py::_freeze()` as `uv pip compile --python-version` when rewriting `pw.lock`, so it governed the resolution of the Pyprojectx bootstrap requirements and nothing else; with `uv` as the sole bootstrap requirement it selected nothing at all, while its name implied it pinned the build.

A `.python-version` or `.tool-versions` file is deliberately absent — either would be a second, competing interpreter declaration alongside `UV_PYTHON`.
