A repeatable methodology for measuring test-suite duration and coverage, and — as a first-class part of the method — a stated detection limit that says which effects the protocol can and cannot resolve.

== Overview

Claims about "the suite got faster" or "coverage went up" are only trustworthy when the measurement that backs them is repeatable and its noise floor is known. This page defines that measurement discipline. It has three parts:

* a CI-history method for suite-level duration, using the free, accumulating verify-job signal over a *median of several runs*;
* a per-test attribution method, using `--durations` tables that are near-immune to fixed per-run overhead;
* an explicit *detection limit* — the smallest effect the protocol can resolve — treated here as a valued deliverable, not a caveat.

The rule that governs every duration claim: a suite-level wall-clock number never stands alone as evidence. It is either backed by per-test `--durations` deltas or by a median over multiple CI runs, or it is not a claim.

== Prerequisite: deterministic local runs

Local measurement is only trustworthy when a run's own machinery does not perturb it. Each `module-tests` and `coverage` pytest invocation is given its own per-session `--basetemp` under `.plan/temp/pytest-basetemp/{session}` (see `build.py`), so concurrent worktrees and a killed-then-restarted session never share a temp root and never race pytest's own cleanup. Without that isolation a cleanup race can perturb — or, under `filterwarnings=["error"]`, kill — an otherwise-green run, which contaminates any duration reading taken from it. Treat the per-session basetemp as the precondition for every local measurement below.

== Method 1 — Suite-level duration from CI history

The verify job records its own elapsed time on every run. That signal is free (it already exists on every PR), accumulating (it grows with each run rather than costing a fresh measurement), and taken on a controlled runner rather than a developer laptop.

Read it through the CI abstraction, never `gh`/`glab` directly:

[source,bash]
----
python3 .plan/execute-script.py plan-marshall:tools-integration-ci:ci checks status --pr {number}
----

Discipline for a duration claim from this signal:

* Collect the verify-job `elapsed_sec` from *at least five* runs, then report the *median* — never a single run. A single run mixes in runner scheduling, cache state, and concurrent-load variance that the median suppresses.
* Compare medians of comparable populations (for example, five pre-change runs against five post-change runs), not a before/after pair.
* Hold the confounds fixed or state them: cache state (cold vs warm), runner class, and any toolchain shift within the sample window.

== Method 2 — Per-test attribution with `--durations`

Suite wall-clock cannot tell you *which* test moved. The per-test `--durations` table can, and it is near-immune to fixed per-run overhead: a constant startup or teardown cost lands outside the per-test timings, so the ranked table isolates the tests that actually dominate.

[source,bash]
----
./pw module-tests {bundle} -- --durations=25
----

(Resolve the wrapper through the architecture API in workflow code — never hard-code `./pw`; see xref:build.adoc[Build].)

Discipline for a per-test duration claim:

* Attribute a hotspot from the ranked `--durations` table, scoped per module so an instrumented run fits under the local time ceiling. Whole-tree instrumented runs are long enough to be orchestrator-tier / CI-only — never blind-backgrounded locally.
* Back every applied fix with a *per-test before/after `--durations` delta*, recorded in the plan artifacts. The per-test delta is the evidence; the suite wall-clock is not.
* Fix the specific cause the table points at — an over-broad fixture, an unnecessary subprocess spawn, a real `sleep()`, a session-vs-function scope mismatch — without dropping or weakening any assertion.

== Detection limit (stated, first-class)

A protocol that cannot say what it *cannot* see invites false claims. The detection limit below is therefore a deliverable in its own right, not a disclaimer.

*A sub-20-second suite-level duration effect is below this protocol's noise floor and is undetectable at the suite level.* Two independently observed sources of variance each dwarf a 20-second effect:

* a cold-versus-warm swing of roughly 59 seconds between an uncached and a cached run of the same suite;
* a verify-job range of about 116 seconds across recent PRs on the CI runner.

Because either source alone is several times larger than a 20-second change, no suite-level measurement — however many runs are medianed — can attribute a change that small to the code rather than to cache state or runner variance.

Consequences for how results are reported:

* A change whose only expected effect is a sub-20-second suite-level duration shift *should not* be claimed from suite wall-clock. Use Method 2 (per-test deltas) to attribute it, or report it as below the detection limit.
* A *bounded or negative* suite-level duration outcome, honestly reported against this stated limit, is an accepted result — not a failure of the work.
* Coverage is the axis expected to move measurably; duration at the suite level often is not. Size expectations accordingly before running the measurement.

== Applying the protocol

For a change that touches test duration or coverage:

. Confirm deterministic local runs (per-session basetemp in place).
. Re-derive the live population first — the current per-test `--durations` ranking, or the current module coverage percentage — never an inherited or stale list.
. Take the measurement by the matching method: Method 1 for suite-level duration, Method 2 for per-test attribution, and a module-scoped `coverage {bundle}` run for coverage percentage.
. Record the reading in the plan artifacts as evidence, with its confounds and its position relative to the detection limit.
. Report deltas against the freshly re-derived baseline, citing per-test `--durations` deltas or CI-history medians — never a standalone suite wall-clock number.
