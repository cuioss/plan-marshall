= Getting Started
:toc: left
:toclevels: 2

After installing Plan Marshall, run the setup wizard to wire it into your project:

[source,bash]
----
/marshall-steward
----

== First-run wizard

When the executor (`​.plan/execute-script.py`) is missing, `/marshall-steward` runs in wizard mode and walks you through:

1. **Configure `.gitignore`** — adds a `.plan/*` block plus targeted re-includes (`!.plan/marshal.json`, `!.plan/project-architecture/`) so the shared configuration is tracked while your private working area under `.plan/local/` and the generated executor stay local. See <<_where_things_live>> for the full layout.
2. **Generate the script executor** — emits `.plan/execute-script.py` with the script mappings embedded for every installed bundle.
3. **Discover project modules** — walks the codebase via the extension API to identify build modules.
4. **Initialise `marshal.json`** — seeds the per-project configuration file with sensible defaults.
5. **Configure skill domains** — chooses which bundles' skill domains apply to which modules.
6. **Detect CI provider and tools** — identifies GitHub vs. GitLab, Sonar/SonarCloud, and any other integrations.

The wizard is idempotent — re-running it on a fresh checkout produces the same setup.

== Interactive menu (returning users)

Once the executor exists, `/marshall-steward` opens an interactive menu instead:

* **Maintenance** — regenerate the executor (after bundle changes), clean execution logs.
* **Health Check** — verify the setup, diagnose missing scripts or stale configuration.
* **Configuration** — adjust build systems, skill domains, the per-role effort/model registry, terminal-title integration, and other project-level knobs.

Use `--wizard` to force the first-run wizard explicitly:

[source,bash]
----
/marshall-steward --wizard
----

== Creating your first plan

Once setup is complete, the simplest way to start a plan is to invoke `/plan-marshall` with no arguments and pick "Create new plan" from the interactive menu:

[source,bash]
----
/plan-marshall
----

If you prefer a one-liner, pass a `task=` description directly — the `init` action is inferred from the presence of `task=`:

[source,bash]
----
/plan-marshall task="Add user authentication"
----

Or seed the plan from a GitHub issue:

[source,bash]
----
/plan-marshall issue="https://github.com/cuioss/plan-marshall/issues/42"
----

`/plan-marshall` drops you into the six-phase planning flow (init → refine → outline → plan → execute → finalize) described in xref:../concepts/planning-workflow.adoc[Planning Workflow]. Resume an existing plan with `/plan-marshall plan="<plan-id>"` — it auto-detects the current phase. See xref:commands.adoc[Commands] for the full parameter reference and the action-resolution rules.

== Where things live

After setup, expect the following inside your project:

[source]
----
.plan/
├── marshal.json                # ✅ committed — project configuration
├── project-architecture/       # ✅ committed — per-module enriched.json + _project.json
│   ├── _project.json
│   └── {module}/
│       └── enriched.json       # ✅ committed
├── execute-script.py           # ❌ ignored — generated by /marshall-steward, do not edit
├── local/                      # ❌ ignored — your private working area
│   ├── plans/{plan-id}/        #    active plans live here
│   ├── worktrees/{plan-id}/    #    per-plan git worktrees
│   ├── lessons-learned/        #    captured lessons before promotion
│   ├── logs/                   #    execution logs
│   ├── archived-plans/         #    completed plans pruned from active list
│   └── …
└── temp/                       # ❌ ignored — scratch space, safe to delete
----

`.plan/` is **partially** tracked. Two paths are committed and reviewed alongside code:

* **`.plan/marshal.json`** — the per-project configuration that drives `/plan-marshall`, build resolution, the per-role effort registry, and skill domains. Changes belong in a normal commit.
* **`.plan/project-architecture/`** — the LLM-enriched module-analysis artifacts. The `enriched.json` files are committed so a fresh checkout has the architecture map without re-running the enrichment.

Everything else under `.plan/` is gitignored by `.plan/*` in the repository's `.gitignore` and is yours alone — never push it, never expect a teammate's copy to match. Active plans, worktrees, lessons in progress, logs, and temporary outputs all live under `.plan/local/` or `.plan/temp/`.

[NOTE]
====
The `.gitignore` also carries a defensive rule `.plan/project-architecture/**/derived.json`. The current `manage-architecture` code (see `_architecture_core.py`) computes the derived module data (paths, packages, dependencies, file inventories) **on demand by crawling the live worktree on every read**, so `derived.json` is no longer written by normal operation. The re-exclusion guards against (a) historical `derived.json` files left over from earlier code paths and (b) the test-only `save_module_derived` fixture writer accidentally producing committable files.
====

For the canonical rules, see the project's `.gitignore` — the `.plan/*` block plus the two `!.plan/...` re-includes is the source of truth.
