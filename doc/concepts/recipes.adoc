= Recipes
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:README.adoc[Concepts]

The standard pipeline re-discovers the scope of every plan from first principles — refine iterates on confidence, outline walks the architecture, plan creates tasks. Useful for the request the user just typed; wasteful for the transformation you have already done forty times. Recipes are the workflow's escape hatch for that case: predefined, repeatable transformations whose plan shape is already known, so the lifecycle skips the refine confidence loop and the outline-discovery work and produces deliverables from a deterministic mapping.

== Overview

The standard pipeline (xref:planning-workflow.adoc[Planning Workflow]) assumes a request whose scope must be *discovered*: phase-2-refine iterates on confidence, phase-3-outline's Complex Track walks the architecture and proposes deliverables, phase-4-plan creates tasks. That is the right shape for "build the thing the user just described." It is the wrong shape for recurring transformations like "refactor every package to the configured profile standards" or "verify every architecture diagram is up to date" — there the plan shape is *already known*, and asking the LLM to re-discover it from scratch each time burns tokens for no gain.

Recipes are the extension point for that case. A recipe declares "I know what this plan looks like; ask me, not the LLM."

== How the lifecycle changes

A recipe-driven plan still passes through every phase, but the work inside the early phases shortcuts. Refine bypasses its confidence iteration and returns immediately. Outline routes to the recipe skill instead of the Simple- or Complex-Track LLM analysis; the recipe skill produces `solution_outline.md` via its own deterministic discovery + deliverable mapping. The Q-Gate at outline is bypassed when the recipe yields a surgical single-deliverable shape and runs normally otherwise. From phase-4-plan onward, the plan flows through the same pipeline as any other.

The cost saving is real: a Complex-Track outline on a multi-module codebase is a multi-thousand-token LLM exercise. A recipe-driven outline producing the same N deliverables is mostly script invocations (`architecture modules`, `architecture module --full`, `manage-files discover`) plus a templated write. The shortcut shows up directly in the per-phase metrics for recipe-driven runs.

User entry point:

[source,bash]
----
/plan-marshall action=recipe
----

The list pulls from `manage-config list-recipes`, the user picks one, and `phase-1-init` writes the recipe metadata into `status.json` (`recipe_key`, `recipe_skill`, `recipe_domain`, `recipe_profile`, `recipe_package_source`).

== The contract

A recipe is a skill whose `SKILL.md` declares `implements: plan-marshall:extension-api/standards/ext-point-recipe`. It is registered for discovery by one of two routes — both fully documented in link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-recipe.md[`extension-api/standards/ext-point-recipe.md`]:

* **Extension-provided** — a domain bundle implements `provides_recipes()` and returns a list of recipe dicts (`key`, `name`, `description`, `skill`, `default_change_type`, `scope`).
* **Project-local** — the project ships its own recipe under `.claude/skills/recipe-*` (mirrors the finalize-step pattern).

At runtime the recipe skill is invoked under phase-3-outline Step 3 with `plan_id`, optional `recipe_domain`, optional `recipe_profile`, and optional `recipe_package_source`. It is responsible for:

1. Resolving the applicable skills for the recipe's profile (e.g. via `manage-config resolve-domain-skills`).
2. Querying the architecture for the input set (modules, packages, diagrams, ADRs, …).
3. Mapping the input set to deliverables — one row per "unit of repetition" (one deliverable per package, per diagram, …).
4. Writing `solution_outline.md` via `manage-solution-outline` so the file is structure-validated like any other outline.

The recipe skill is the *single source of truth* for what its deliverables look like — phase-3-outline does not second-guess it.

== Where this fits

Recipes are the workflow's *known-shape escape hatch*. The standard pipeline is what makes Plan Marshall safe for arbitrary requests; recipes are what make it efficient for recurring ones. The hooks the recipe pattern rides on (xref:execution-context.adoc[execution-context dispatch], xref:skill-handling.adoc[skill loading], xref:audit-trail.adoc[audit trail]) are the same — the recipe just supplies its own discovery step.

Recipes do not make the recurring shape any less recurring — the underlying drudgery is still there, just dispatched efficiently. The recipe pattern is for "this is going to happen again", not "we have solved the underlying problem."
