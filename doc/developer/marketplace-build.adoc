How the marketplace source under `marketplace/bundles/` is transformed into deployable plugin artifacts. This document is orientation — the canonical specs live next to the code they govern.

== Pipeline

image::../resources/diagrams/marketplace-build-pipeline.svg[Marketplace build pipeline — marketplace/bundles/ source is transformed by marketplace/targets/generate.py into target/claude/ per-bundle output with role-eligible agents expanded into per-level variants; the meta-project-local /sync-plugin-cache rsyncs that target directory into ~/.claude/plugins/cache/ which is what Claude Code actually loads, align=center]

The middle hop (`target/claude/`) is produced by `marketplace/targets/generate.py --target claude` or by the link:../../.claude/skills/finalize-step-deploy-target/SKILL.md[`project:finalize-step-deploy-target`] finalize step in the meta-project. The `/sync-plugin-cache` slash command rsyncs that directory into the plugin cache; consumer projects of Plan Marshall use the cache directly via `/plugin marketplace update` and never see the SOURCE or TARGET stages.

== Variant Emission

The Claude-target reader under `marketplace/targets/claude/` walks each bundle. The dispatcher entry point is `emitter.py`; the variant-emission logic (`LEVEL_TABLE`, `emit_variants_for_agent`, the role-eligibility check) lives in the sibling `variant_emitter.py`. For an agent declaring `implements: plan-marshall:extension-api/standards/ext-point-dynamic-level-executor`, the emitter produces up to eight files:

1. The **canonical** no-suffix file (e.g. `execution-context.md`) with `implements:` and `levels:` stripped. Claude Code dispatches this when the resolver returns `inherit`. The dispatched subagent inherits the parent session's model.
2. Seven **level-suffixed variants** (`execution-context-{level-1|level-2|level-3|level-4|level-5|level-6|level-7}.md`), each with `model:` and `effort:` baked into the frontmatter from the level → primitive mapping in link:../../marketplace/bundles/plan-marshall/skills/plan-marshall/standards/effort-levels.md[`effort-levels.md`]. The `-level-6` and `-level-7` variants are alias-capability-gated — the build emitter's guard skips a variant when the resolved alias does not accept the level's effort (`xhigh` / `max`), leaving fewer files in that case.

`plugin.json` registers every emitted entry so Claude Code's agent registry sees the canonical and each level variant.

The marketplace ships two role-eligible canonical agents: `plan-marshall:execution-context` and `plan-marshall:execution-context-reader`. Every plan-marshall `Task:` invocation dispatches a canonical or one of its level variants. The workflow body the agent runs is supplied at dispatch time via the `workflow` field in the prompt body, per link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/ext-point-execution-context-workflow.md[`ext-point-execution-context-workflow.md`].

== Adding a Workflow Doc

A new workflow doc lives under `{bundle}:{skill}/workflow/{file}.md` (single-phase consumer) or `plan-marshall/skills/plan-marshall/workflow/{file}.md` (multi-phase / cross-cutting). The doc declares `implements: plan-marshall:extension-api/standards/ext-point-execution-context-workflow` in frontmatter and provides at minimum an Inputs table, Workflow steps, and an Output contract (`status` + `display_detail`).

No new agent file is required — the new workflow is dispatched through the existing seven `execution-context` variants by calling sites that pass the new workflow's `{bundle}:{skill}/workflow/{file}.md` notation in the `workflow` prompt-body field.

== Dispatch-Cost Considerations

Each `Task:` dispatch carries a fixed overhead (skill-load preamble, Worktree Header echo, return-TOON marshalling). The granularity heuristics in link:../../marketplace/bundles/plan-marshall/skills/extension-api/standards/dispatch-granularity.md[`extension-api:dispatch-granularity`] govern when a step belongs in a dispatch envelope vs. inline in the orchestrator's context:

* **Script-over-dispatch** — deterministic work (regex matches, structural checks, build invocations) lives in a script; the dispatch is reserved for LLM judgement.
* **Bundle-shared-context** — a multi-step LLM workflow runs inside ONE dispatch envelope rather than N sequential dispatches.
* **Per-iteration only when parallel-or-different-models** — spinning up N parallel dispatches is justified when each subagent runs independently. The sole such case in the marketplace is the `enrich-module` workflow dispatched under `--phase phase-6-finalize` from `architecture-refresh` Tier-1.

== Adapter System

The marketplace generator also supports non-Claude targets via adapters under `marketplace/targets/`:

[source,bash]
----
python3 marketplace/targets/generate.py --target opencode --output target/opencode
----

The OpenCode adapter produces output conforming to the OpenCode specification; only the Claude target is validated end-to-end. Adapter implementations register in `TARGET_REGISTRY` (see `marketplace/targets/generate.py`).

== Deterministic Versioning and Staleness Signaling

The generator derives a deterministic, monotonic version from source alone and emits it — together with content fingerprints — into a per-target `dist-manifest.json`. The installed executor and `marshal.json` embed the version and fingerprints they were provisioned at, so a later session can detect when its derived state has fallen behind the published marketplace.

=== The `0.1.N` version scheme

The single base version `0.1` lives only in `marketplace/.claude-plugin/marketplace.json` (`metadata.version`). link:../../marketplace/targets/generate.py[`generate.py`] computes the full version as `{base}.{N}`, where `N` is the first-parent commit count (`git rev-list --count --first-parent HEAD`). The result is deterministic — the same commit yields the same version in CI and locally — and monotonic, since `N` only grows along first-parent history.

Version comparison is a full int-tuple compare with no zero-padding, so `0.1.9` sorts below `0.1.10` and a future `0.2` base bump orders correctly above every `0.1.N`. An empty or `unknown` sentinel parses to the empty tuple — the lowest possible version — so a fresh or unstamped surface is never treated as newer than a real published version.

The source bundle `plugin.json` files keep the static base version; only the generated target tree carries the live per-landing `0.1.N`, which the generator stamps into every target-tree bundle `plugin.json`. An explicit `--version` flag overrides the computed value verbatim — CI uses it to stamp an identical version across both target legs.

=== The `dist-manifest.json` fields

The generator emits one `dist-manifest.json` at each target output root, carrying six fields:

[cols="1,3"]
|===
| Field | Meaning

| `version`
| The deterministic `0.1.N` version this tree was generated at.

| `source_sha`
| The source commit sha (`git rev-parse HEAD`) the tree was generated from.

| `executor_scripts_fingerprint`
| Machine-portable fingerprint of the executor's script set — relativized notation-to-path mappings, insensitive to script-body edits, so it moves only when a script is added, removed, moved, or renamed.

| `executor_changed_at_version`
| The version at which `executor_scripts_fingerprint` last changed.

| `config_seed_fingerprint`
| Canonical-JSON hash of the default config seed (`get_default_config()`).

| `config_changed_at_version`
| The version at which `config_seed_fingerprint` last changed.
|===

The two `changed_at` fields are the staleness signal: a consumer whose embedded version is older than a `changed_at` value knows the corresponding derived state is stale.

=== CI-side `changed_at` derivation

The `changed_at` values are derived entirely CI-side by the link:../../.github/workflows/claude-distribute.yml[`claude-distribute.yml`] distribute workflow — there is no finalize step and no commit back to `main`. On each publish the workflow fetches the previous `dist-manifest.json` from the current dist-branch tip and passes it to the generator via `--previous-manifest`. The generator diffs each fingerprint against that previous manifest:

* an unchanged fingerprint carries the previous `changed_at` forward;
* a changed fingerprint advances `changed_at` to the current version;
* a first publish (no previous manifest) bootstraps every `changed_at` to the current version.

The workflow checks out full history (`fetch-depth: 0`, needed for the first-parent count) and serializes its runs per target via a `concurrency` group, so concurrent pushes never race on the orphan-branch publish or the previous-manifest fetch.

=== Executor and config staleness signaling

The installed executor embeds `MARSHALL_VERSION`, and `marshal.json` stamps `system.provisioned_version` / `system.config_seed_fingerprint` at init (refreshed by `manage-config sync-defaults`). The deterministic `generate_executor preflight` verb (see link:../../marketplace/bundles/plan-marshall/skills/tools-script-executor/SKILL.md[`tools-script-executor` SKILL.md] § Canonical invocations) compares these embedded values against the installed `dist-manifest.json` under two asymmetric ownership rules:

* **The executor is safe derived state** (ADR-002, the move-based cwd-pinned model). When its embedded version is older than `executor_changed_at_version`, the verb regenerates it in place and reports `executor_action: regenerated`; otherwise `executor_action: fresh`.
* **`marshal.json` holds user decisions and is never auto-mutated.** When `system.provisioned_version` is older than `config_changed_at_version`, the verb reports `marshal_status: stale` advisory-only, routing the user to `/marshall-steward` for a config reconcile rather than rewriting their config.

A fresh install with no manifest resolves both `changed_at` values to the empty sentinel, so nothing is stale and the verb is a no-op reporting `fresh`. The verb is surfaced at two sites: the `/plan-marshall` cross-session re-entry preflight and the `/marshall-steward` health menu (via link:../../marketplace/bundles/plan-marshall/skills/marshall-steward/SKILL.md[`marshall-steward` SKILL.md] `determine_mode check-staleness`).

== Open-Standard Alignment

Plan Marshall's component format predates and overlaps with the `SKILL.md` open standard published at https://agentskills.io[agentskills.io]. This section is an informative gap analysis — it records where the two agree, where Plan Marshall extends the spec, and how the adapter system above relates to the standard. It makes no behavioural claims and adds no build steps.

=== Where the formats conform

The open standard defines a skill as a directory containing a `SKILL.md` file whose YAML frontmatter carries a `name` and a `description`. Every Plan Marshall skill satisfies this base shape: each lives at `marketplace/bundles/<bundle>/skills/<skill>/SKILL.md` and declares `name` plus `description` in the form the spec expects (see link:../../marketplace/bundles/pm-plugin-development/skills/plugin-architecture/references/frontmatter-standards.md[`frontmatter-standards.md`] § Skill Frontmatter). A reader that understands only the open standard can therefore discover and name every Plan Marshall skill from its frontmatter alone.

=== Where Plan Marshall extends the spec

Plan Marshall adds frontmatter fields the open standard does not define. These are additive — a spec-only reader ignores them, and they never change the meaning of `name` or `description`:

* `user-invocable` (required) — whether the skill appears in the slash menu or is loaded only internally.
* `mode` (required) — the execution archetype (`knowledge` / `workflow` / `script-executor` / `manifest`) that tells an agent how to consume the skill.
* `implements` (optional) — declares that the skill satisfies a Plan Marshall extension-point contract.

The full optional set (`argument-hint`, `compatibility`, `disable-model-invocation`, `license`, `metadata`) is documented in `frontmatter-standards.md`. None of these fields is part of the agentskills.io spec; they are Plan Marshall's own workflow metadata.

=== How the adapter system relates to the standard

The `marketplace/targets/` adapter system (see _Adapter System_ above) is Plan Marshall's own multi-assistant export mechanism — it transforms the source bundles into per-target output such as `target/claude/` and `target/opencode/`. It is **not** an agentskills.io conformance layer: it neither validates against the published spec nor emits a spec-conformant package. The adapter predates the standard and serves a different purpose (cross-assistant portability of Plan Marshall's own format), so the two are independent. A future agentskills.io target, if one were added, would register in `TARGET_REGISTRY` like any other adapter.

== Plugin cache sync (meta-project only)

When modifying plugin source files (skills, agents, commands) inside the Plan Marshall meta-project, changes do not take effect until the plugin cache is updated. After editing files in `marketplace/bundles/`, run:

[source]
----
/sync-plugin-cache
----

This synchronises all bundles from `target/claude/` to `~/.claude/plugins/cache/plan-marshall/` using rsync with `--delete` to ensure exact mirroring. The slash command and its finalize-step counterpart are project-local under `.claude/skills/` — consumer projects do not get this surface, only the meta-project.

== Registered marketplace path

The Claude Code marketplace registration MUST point at `target/claude/`, not at the source `marketplace/` directory. The source `marketplace/bundles/<bundle>/.claude-plugin/plugin.json` declares only the canonical agent files; the build target expands each role-eligible agent into per-level variants (`{name}-level-1.md` through `{name}-level-7.md` — seven suffix variants, with `-level-6` and `-level-7` alias-capability-gated) under `target/claude/<bundle>/agents/` and emits a variant-aware `plugin.json` plus a top-level `target/claude/.claude-plugin/marketplace.json`. Registering the source path makes Claude Code's plugin loader install only the canonicals, so every dispatch site that resolves to `execution-context-{level}` fails with `Agent type not found`.

One-time migration on a developer machine:

1. Ensure `target/claude/` is current: `python3 marketplace/targets/generate.py --target claude --output target/claude`.
2. Re-point the marketplace: edit `~/.claude/plugins/known_marketplaces.json` so the `plan-marshall` entry's `source.path` and `installLocation` point at `/path/to/plan-marshall/target/claude` (NOT `.../marketplace`). Alternatively, in-app: `/plugin marketplace remove plan-marshall` then `/plugin marketplace add /path/to/plan-marshall/target/claude`.
3. Reinstall the plugins so install metadata picks up the variant-aware `plugin.json`: `claude plugin uninstall plan-marshall@plan-marshall && claude plugin install plan-marshall@plan-marshall` (repeat per bundle as needed).
4. Restart Claude Code or run `/reload-plugins`.
5. Verify the variants are registered: ask a fresh session to list `plan-marshall:` entries from its available-agents header. Expect canonical + seven level variants.
