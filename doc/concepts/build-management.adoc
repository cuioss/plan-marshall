= Build Management
:nofooter:
:toc: left
:toclevels: 2

xref:../../README.md[Plan Marshall] » xref:README.adoc[Concepts]

A 10-minute Maven build streamed into a stock LLM session burns tokens, fragments attention, and fills the context window with `[INFO] Building module-7…` line by line — none of it useful, all of it billable. Plan Marshall makes the build a single synchronous tool call: the model dispatches one operation, suspends on it, and resumes when the build finishes with a structured TOON containing status, parsed errors, and a pointer to the full log on disk. No tokens spent waiting, no log scrolled into context, no attention spent babysitting a process the LLM cannot influence anyway.

== Build-tool independence at the prompt layer

From the LLM's point of view, "run the verify build" is a single call with a single shape. The workflow does not know whether the module is Maven, Gradle, npm, or pyprojectx; it asks for the *canonical command* (`verify`, `quality-gate`, `module-tests`, `coverage`, …) by module name, and the resolver returns a fully-wrapped executable that the LLM runs verbatim. The build-system invocation is determined per module at architecture-discovery time by `/marshall-steward` and persisted in `.plan/project-architecture/{module}/enriched.json` — a deterministic lookup, not a probabilistic match.

This means a skill author writing a verify step writes one workflow that works against every build system in the project. Adding a new build tool (Go, Rust, …) is a matter of writing a new `build-{tool}` skill that implements `ext-point-build` and a companion `plan-marshall-plugin` extension that maps canonical commands to the new tool's invocation. The rest of the system — dispatch sites, verify-step lists, retrospective fragments — continues to work unchanged because it only ever sees the canonical names.

== Why no tokens during the wait

image::../resources/diagrams/build-dispatch-sequence.svg[Build dispatch sequence — the LLM resolves the canonical command, dispatches the build skill, suspends on the tool call while the build subprocess runs, then receives a structured TOON return; no tokens are billed to the orchestrator during the wait, align=center]

Build skills capture all build output to a log file before returning anything to the LLM. The return is the *summary* — exit status, duration, parsed errors, the path to the full log. The LLM consults the log only when the structured errors aren't enough to act on. Three consequences compound:

* **A 10-minute build costs ~100 tokens.** Just the TOON envelope. The duration of the wait is decoupled from the token bill.
* **The build output never enters working memory** unless the LLM explicitly reads the log. The context window stays focused on the task that triggered the build, not the build's intermediate noise.
* **Most plans run multiple builds** (incremental compile, quality-gate, module-tests, verify). At ~100 tokens each, the build budget is a rounding error. The same plan without this abstraction would burn tens of thousands of tokens on build-output streaming alone.

The synchronous shape is the load-bearing property here. The LLM is *suspended* on the tool call, not polling, not babysitting, not generating filler tokens while waiting. From the model's perspective the build is one operation that takes however long it takes and returns one structured answer.

[CAUTION]
====
**Zero tokens during the wait does not mean zero cost.**

Long-running builds still consume wall-clock time the developer pays for in latency. The `manage-metrics` accumulator records per-phase build duration, so the cost is measurable; the canonical-commands resolver records observed durations and adapts its timeout ceiling per command, so subsequent runs of the same build set the right budget instead of hard-coding a one-size-fits-all value. The abstraction makes the build cheap *in tokens*; it does not make it instantaneous.

One second-order token effect: when a build's wait exceeds the prompt cache's ephemeral TTL (~5 minutes), the cached context prefix expires during the suspension, so the *resume* turn re-establishes the cache (a `cache_creation` cost on the next turn rather than the cheap `cache_read`). This is a per-turn caching artifact, not the build's own cost, and it is bounded by the small isolated-envelope context (xref:token-management.adoc[Token Management] § Per-dispatch context isolation) — minor next to the latency the build already costs.
====
